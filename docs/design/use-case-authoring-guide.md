# 用例编写指南(Use Case Authoring Guide)

> 本指南规定各子域 `use-cases.md` 的编号规则、编写规范,以及用例与菜单(action-auth.xml)的关联机制。
> 用例是"场景维度"文档,与"机制维度"文档(state-machine/posting 等)互补:机制文档回答"某机制怎么工作",用例文档回答"这个域有哪些可验证的使用场景"。

## 1. 用例编号规则

每个用例必须有**全局唯一编号**,格式:

```
UC-{DOMAIN}-{NN}
```

- `UC` — 固定前缀(Use Case)
- `{DOMAIN}` — 域简称(3 字母大写),与工程简称一致:
  - `MD` master-data / `PUR` purchase / `SAL` sales / `INV` inventory
  - `FIN` finance / `AST` assets / `PRJ` projects / `MFG` manufacturing
  - `MAIN` maintenance / `QA` quality / `SYS` system(系统管理)/ `L10N` 本地化
- `{NN}` — 域内顺序号(01-99),两位数字,从 01 起

**示例**:`UC-PUR-01`(采购域第 1 个用例)、`UC-FIN-12`(财务域第 12 个)。

**编号约束**:
- 编号一旦分配并写入 `use-cases.md`,**不可复用、不可重排**(避免引用断裂)。删除用例时编号作废,不补位。
- 一个用例编号对应一个 `## UC-xxx-NN {标题}` 二级标题。
- 编号在 `use-cases.md` 标题、action-auth.xml 的 `app:useCases`、测试用例注释三处保持一致引用。

## 2. 用例文档结构

每个用例(`## UC-xxx-NN`)包含四部分:

```markdown
## UC-PUR-01 标准采购全流程(主路径)

**场景**:一句话描述业务情境。

**前置**:触发条件/依赖(主数据就绪、状态前置等)。

**行为链路**:步骤序列,引用机制文档不重复(如"见 state-machine.md §1")。

**可验证断言**:用自然语言与伪代码交织表达,聚焦"查什么/等于什么/触发什么"。
```

### 编写规范

1. **断言是核心**:每条断言必须可验证(能转测试期望)。避免"系统应该正确处理"这类不可验证的空话。
2. **引用不重复**:机制细节用"见 {机制文档}.md §{章节}"引用,不在用例复述(遵循 AGENTS.md 规则 11,避免双真相源)。
3. **语言与伪代码交织**:用业务术语("由请购单生成订单")而非类名调用(`ErpPurOrder.from()`),AI 自行映射。伪代码只用于表达关系/约束/触发条件。
4. **跨域断言允许**:端到端用例可验证跨域结果(如采购用例验证 finance 凭证、inventory 库存)。
5. **聚焦域特有机制**:不写通用 CRUD(创建/查询/删除),写该域特有的业务机制(三单匹配/过账/批次追溯/齐套校验等)。
6. **粒度**:每域 5-15 个核心路径用例,覆盖主路径 + 关键分支 + 易错边缘场景。

## 3. 用例与菜单的关联(app:useCases)

菜单(action-auth.xml 的 `<resource>`)通过 `app:useCases` 属性标记它支撑哪些用例。

### 机制

`app:useCases` 是 XDSL 扩展属性(`app:` 命名空间),写在 `<resource>` 上,值为用例编号。一个菜单可关联多个用例(逗号分隔),一个用例也可被多个菜单关联。

```xml
<auth ... xmlns:app="app">
  <site id="main">
    <resource id="ErpPurOrder-main" displayName="采购订单" ...
              app:useCases="UC-PUR-01,UC-PUR-03,UC-PUR-08">
      ...
    </resource>
  </site>
</auth>
```

### 命名空间声明

使用 `app:useCases` 的 action-auth.xml 根节点必须声明 `xmlns:app="app"`(类似 `xmlns:i18n-en="i18n-en"`)。

### 关联原则

- **单据/功能页面菜单**:关联覆盖该单据的主路径与关键分支用例(如"采购订单"菜单关联 UC-PUR-01 全流程、UC-PUR-03 分批入库、UC-PUR-08 请购转单)。
- **报表/看板菜单**:关联该报表/看板验证的用例(如"试算平衡表"关联 UC-FIN-01 过账,各域看板关联 UC-xxx 看板用例,见 dashboards.md)。
- **主数据 CRUD 维护页**(物料/往来单位/仓库/科目/币种等维护页):**无需关联独立用例**。主数据是"被引用"的基础,其业务场景是横切的(如扫码开单发生在订单录入),用例挂在业务单据菜单上,不挂在主数据维护页。CRUD 维护本身是通用的,无独立业务语义。
- **纯配置/分类页**(项目类型/活动类型/维护团队/MRP 独立需求等):**无需关联用例**。配置类无独立业务场景。
- **占位菜单(待实现功能)**:可暂不关联,或关联到设计文档已定义的用例编号。
- 不强制每个菜单都关联(如纯导航的分组菜单、主数据维护页、配置页不需要),但**单据/功能页面、报表、看板应关联**。

> **工具的"无用例关联"警告**对主数据 CRUD 维护页和纯配置/分类页是预期内的(这些菜单本就无独立业务用例),不应视为缺口。真正的缺口是"有业务语义但无用例"的菜单。

## 4. JS 工具:用例-菜单对照与完备性检查

工具位置:`tools/use-case-map.js`(Node.js,无外部依赖)。

### 作用

1. **菜单→用例对照**:解析所有 action-auth.xml 的 `app:useCases`,列举每个菜单关联的用例编号与标题。
2. **完备性检查**:识别"无用例关联的功能菜单"(可能遗漏场景设计)和"无菜单引用的用例"(孤儿用例)。
3. **用例统计/概览**:按域统计用例数、菜单覆盖率,输出总览表。

### 用法

```bash
node tools/use-case-map.js              # 完整对照 + 完备性报告 + 概览
node tools/use-case-map.js --menu       # 仅菜单→用例对照
node tools/use-case-map.js --coverage   # 仅完备性检查
node tools/use-case-map.js --overview   # 仅域级概览
```

### 输出示例

```
== 菜单→用例对照 ==
[erp/pur] 采购订单 (ErpPurOrder-main) → UC-PUR-01, UC-PUR-03, UC-PUR-08
[erp/pur] 采购入库单 (ErpPurReceive-main) → UC-PUR-01, UC-PUR-03

== 完备性检查 ==
⚠ 无用例关联的功能菜单(可能遗漏场景设计):
  [erp/fin] 总账余额 (ErpFinGlBalance-main)
⚠ 无菜单引用的孤儿用例:
  UC-PUR-06 数量差异(短收)

== 域级概览 ==
域            用例数  关联菜单数  覆盖率
purchase      8       6          75%
finance       12      8          67%
...
```

### 工具的完备性检查价值

- **发现遗漏场景**:功能菜单无任何用例关联 → 该功能可能缺少场景设计,提示补充 use-case。
- **发现孤儿用例**:用例无菜单引用 → 可能功能未建菜单,或用例已过时。
- **覆盖率度量**:量化"已设计的用例有多少被菜单功能实际承载",辅助判断设计完备度。

## 5. 用例与测试的衔接

- 每条用例的"可验证断言"应能直接转测试期望(查实体/查凭证/比较金额/验证状态迁移)。
- 测试类可在注释中引用用例编号(如 `// 覆盖 UC-PUR-01: 标准采购全流程`),建立测试↔用例双向追溯。
- 用例文档末尾的"用例与测试的衔接"章节说明该域用例如何映射到测试类型(单元/集成/端到端)。

## 6. 维护规则

- 新增功能时:先写用例(分配编号)→ 建/关联菜单(app:useCases)→ 实现并写测试(引用编号)。
- 用例编号不可复用;废弃用例标记 DEPRECATED 但保留编号。
- 用例与机制文档同步:机制变更时,检查引用该机制的用例断言是否需更新。
- 定期运行 JS 工具检查完备性(无用例菜单/孤儿用例)。
