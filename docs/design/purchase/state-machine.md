# 采购单据状态机

> **设计要点依据**：本状态机按 `docs/skills/state-machine-business-review-prompt.md` 的 10 个审查维度组织。审查本状态机时使用该提示词。

## 适用对象

本状态机适用于采购域的**业务单据**：采购订单、采购入库单、采购发票、付款单、采购退货单。采购单据采用**三轴状态分离**（业务/审核/付款），不是单一状态机。

## 三轴状态分离

采购单据的"状态"由三个独立维度表达，避免单状态机的组合爆炸：

| 状态轴 | 字段语义 | 适用单据 | 表达什么 |
|--------|----------|----------|----------|
| 单据状态 | docStatus | 全部 | 单据生命周期（草稿/已生效/已作废） |
| 审核状态 | approveStatus | 全部 | 审核流程（未提交/已提交/已审核/已驳回） |
| 付款状态 | paidStatus | 采购发票 | 收付款进度（未付/部分/已付清） |

持久化状态码字典以 `model/app-erp-purchase.orm.xml` 为准。

> **为何三轴分离而非单状态机**：若用单状态机表达"已审核+部分付款""已审核+已付清""已作废+未付"等组合，状态数会笛卡尔积爆炸。三轴分离让每个维度独立演化，业务语义清晰。这是"条件作为字段而非状态"反模式的正确应用——付款进度是派生字段，不是独立工作流状态。

本文以"审核状态机"为主线展开（最复杂），单据状态与付款状态作为关联维度。

## 1. 状态定义（审核轴）

每个状态表达"等待什么"：

| 状态 | 业务含义（等待什么） | 可修改单据 |
|------|----------------------|------------|
| 未提交（UNSUBMITTED） | 等待提交审核 | 是 |
| 已提交（SUBMITTED） | 等待审核人决策 | 否 |
| 已审核（APPROVED） | 终态：已审核通过，已触发后续业务 | 否（需红冲/反审核） |
| 已驳回（REJECTED） | 等待修改后重新提交 | 是 |

## 2. 迁移完整性（审批轴）

> 本节只讨论 **approveStatus（审批轴）** 迁移。docStatus（业务生命周期轴）独立演化，详见 `../domain-design-guidelines.md` §14.2。新建单据的完整态：`docStatus=DRAFT, approveStatus=UNSUBMITTED`。

```
未提交 (UNSUBMITTED)
  ├─ 提交 → 已提交 (SUBMITTED)
  │            ├─ 审核通过 → 已审核 (APPROVED)
  │            │              ├─ [触发后续业务：写库存/生成凭证]
  │            │              │              └─ posted=true 后物理锁定，纠错需红冲/反审核
  │            │              └─ 反审核 → 已驳回（REJECTED，需先冲销已生成结果）
  │            ├─ 撤销提交 → 未提交（UNSUBMITTED，仅审核人未处理时允许）
  │            └─ 驳回 → 已驳回 (REJECTED)
  │                          └─ 修改后重新提交 → 已提交
  └─ 作废 → 已作废（docStatus=CANCELLED）

> **反审核目标态说明**：反审核目标态是 `REJECTED`（可重新提交），**不是** `UNSUBMITTED`（初始态）。语义理由见 `../domain-design-guidelines.md` §16.4——反审核的单据已发生过业务（已生成下游单据、已过账），不应回退为"未提交"；`REJECTED` 保留"曾审核过"的历史语义，便于审计追溯。

> **撤销提交约束**：仅提交人可操作；审核人一旦开始审核（nop-wf 已激活），提交人不可再撤回。
```

每条迁移的触发、前置、结果：

| 迁移 | 触发人 | 前置条件 | 结果 |
|------|--------|----------|------|
| UNSUBMITTED → SUBMITTED | 采购员 | 单据填写完整、行数据有效 | 进入待审核 |
| SUBMITTED → APPROVED | 审核人/管理员 | 已提交状态；入库单需可用量充足（库存校验）；发票需三单匹配通过 | 触发后续业务（见下文） |
| SUBMITTED → REJECTED | 审核人/管理员 | 已提交状态 | 退回修改 |
| SUBMITTED → UNSUBMITTED | 提交人 | 审核人尚未处理（nop-wf 未激活） | 撤回提交，恢复可编辑 |
| REJECTED → SUBMITTED | 采购员 | 已驳回状态、已修改 | 重新提交 |
| APPROVED → REJECTED（反审核） | 审核人/管理员 | 已审核、且已冲销所有已生成结果（库存冲销移动单、财务红字凭证） | 回到已驳回（保留曾审核语义） |
| 任意非终态 → 作废 | 采购员/管理员 | 单据未到终态 | docStatus → CANCELLED |

**SUBMITTED → APPROVED 触发的后续业务**（按单据类型）：

| 单据类型 | 审核通过后触发 |
|----------|----------------|
| 采购入库单 | 调用 `IErpInvStockMoveBiz` 生成入库移动单（incoming） |
| 采购发票 | 财务域生成应付凭证（AP_INVOICE） |
| 付款单 | 财务域生成付款凭证（PAYMENT）+ 核销发票 |
| 采购退货单 | 调用库存域生成出库移动单（outgoing）；若已开票生成红字发票 |
| 采购订单 | 仅状态推进，不直接触发库存/凭证（订单是意向，下游单据才触发） |

## 3. 终态与恢复

- **终态**：`已审核（APPROVED）`（审核轴终态）、`已作废（docStatus=CANCELLED）`。
- **已审核的纠错路径**：
  - **反审核**：需先冲销所有已生成结果（库存冲销、财务红字凭证），才能 APPROVED → REJECTED（非回 UNSUBMITTED）。这是强约束，防止已入账数据被篡改。反审核目标态是 `REJECTED` 而非初始态 `UNSUBMITTED`，详见 `../domain-design-guidelines.md` §11.4。
  - **红冲单据**：不修改原已审核单据，而是生成红字退货单/红字发票冲销。这是更安全的纠错方式（保留审计轨迹）。
- **已作废不可恢复**：作废是终态，需重新创建单据。
- **已驳回可恢复**：修改后重新提交。

## 4. 异常路径

| 异常场景 | 处理 |
|----------|------|
| 入库时可用量不足（负库存场景） | 采购入库是增加库存，通常不触发可用量不足；但若启用"入库校验"（如序列号重复），拒绝审核 |
| 三单匹配失败（发票审核时） | 发票数量/金额超容差 → 拒绝审核并提示差异；详见 `three-way-match.md` |
| 财务过账失败 | 业务单据已审核 + `posted=false`；异步重试 + 兜底扫描；详见 `finance/posting.md` |
| 并发审核同一单据 | 乐观锁，后审核者失败重试 |
| 重复审核（幂等） | 已审核单据再次审核为空操作 |
| 付款核销时发票已作废 | 拒绝核销，提示发票状态异常 |
| 供应商停用后开单 | 通过 `IErpMdPartnerBiz` 校验，停用供应商拒绝新单引用 |
| 多币种汇率缺失 | 凭证过账时报错（汇率由主数据域提供），不静默用默认值 |

## 5. 可达性

- **从 UNSUBMITTED 可达**：SUBMITTED、APPROVED、REJECTED、作废 全部可达。
- **REJECTED → SUBMITTED → APPROVED** 路径保证驳回后可重新推进。
- **APPROVED → REJECTED（反审核）** 是唯一的"回退到可修改态"路径，且需冲销前置；目标态是 REJECTED（保留曾审核语义），不是初始态 UNSUBMITTED。
- **无不可达状态、无死锁**：APPROVED 与作废是终态；反审核是显式路径非循环。
- **合法循环的退出条件**：UNSUBMITTED→SUBMITTED→REJECTED→SUBMITTED（驳回修改重提）是合法循环，退出条件是"审核通过 → APPROVED 终态"。

## 6. 角色与权限

每个迁移绑定执行角色：

| 迁移 | 执行角色 |
|------|----------|
| 提交（UNSUBMITTED→SUBMITTED） | 采购员（单据创建人） |
| 审核通过（SUBMITTED→APPROVED） | 审核人/管理员（按单据类型配置审批流） |
| 驳回（SUBMITTED→REJECTED） | 审核人/管理员 |
| 反审核（APPROVED→REJECTED） | 管理员（高权限，因需冲销已入账数据；目标态 REJECTED 非 UNSUBMITTED，见 §11.4） |
| 作废 | 采购员（草稿/未提交阶段）/ 管理员（已审核后作废需高权限） |

危险操作控制：
- **反审核已审核单据**：需管理员权限 + 冲销前置校验。
- **作废已审核单据**：需管理员权限，且需冲销已生成结果。
- 采购员与审核人建议不可为同一人（职责分离，可配置）。

## 7. 外部依赖

采购单据本身不直接依赖外部系统，但触发的后续业务涉及：

| 外部场景 | 内部处理 |
|----------|----------|
| 供应商发票来自外部（如电子发票） | 外部发票数据转为内部 `采购发票（UNSUBMITTED）`，不直接使用外部状态 |
| 财务凭证生成（异步） | 通过 post-commit 事件触发，失败重试不阻塞业务单据 |
| 库存写入（跨工程调用） | 通过 `IErpInvStockMoveBiz` 同步调用，事务传播保证一致 |

外部触发渠道：
- 采购员手工创建（主要渠道）。
- 采购订单联动生成入库单（部分收货场景）。
- 外部系统集成（如电商采购对接，转为内部单据）。

## 8. TODO / 任务策略

| 状态 | 是否产生 TODO | TODO 类型 | 说明 |
|------|---------------|-----------|------|
| UNSUBMITTED | 是 | assigned（采购员） | 草稿待采购员完善并提交 |
| SUBMITTED | 是 | pool/assigned（审核人） | 待审核人决策 |
| APPROVED | 否 | — | 终态，归档；后续业务由下游单据/财务域驱动 |
| REJECTED | 是 | assigned（采购员） | 退回待修改 |

避免"已提交但无人审核"的单据沉没：审核 TODO 按单据类型分配到对应审核角色池。

## 9. 场景演练

### 场景 A：采购入库 happy path

1. 采购员创建采购订单 → UNSUBMITTED → SUBMITTED → APPROVED（订单审核）。
2. 收货时创建采购入库单 → UNSUBMITTED → SUBMITTED → APPROVED → 触发库存入库移动单 → 财务生成暂估应付凭证。
3. 收到发票 → 创建采购发票（回链入库行，三单匹配）→ APPROVED → 财务生成应付凭证。
4. 付款 → 创建付款单（核销发票）→ APPROVED → 财务生成付款凭证，发票 paidStatus → 已付清。

### 场景 B：发票三单匹配失败（异常）

1. 采购发票 SUBMITTED，审核时校验三单匹配。
2. 发票数量 > 入库数量（超容差）→ 拒绝审核，保持 SUBMITTED。
3. 采购员修改发票数量或调整容差配置后重新提交。

### 场景 C：已审核入库单反审核

1. 入库单已 APPROVED，库存已增加、暂估凭证已生成。
2. 管理员反审核 → 校验"是否已冲销"：库存冲销移动单未生成、暂估凭证未红冲 → 拒绝反审核。
3. 先生成冲销移动单 + 红字凭证 → 反审核成功 → 入库单回到 REJECTED（保留曾审核语义，可修改后重新提交，而非回退到初始态 UNSUBMITTED）。

### 场景 D：部分付款

1. 采购发票 APPROVED，金额 10000。
2. 第一次付款 6000 → 核销 → 发票 paidStatus = 部分。
3. 第二次付款 4000 → 核销 → 发票 paidStatus = 已付清。

## 付款状态机（采购发票，派生状态）

付款状态是**派生状态**，由付款单核销明细自动驱动，不是独立工作流：

```
未付 (UNPAID)
  ├─ 付款核销（累计 < 发票金额）→ 部分 (PARTIAL)
  │                                 ├─ 继续付款（累计 ≥ 发票金额）→ 已付清 (PAID)
  │                                 └─ 红冲核销（累计回退）→ 未付或保持部分
  └─ 付款核销（累计 ≥ 发票金额）→ 已付清 (PAID)
```

- 付款状态由系统根据"累计已核销金额 / 发票金额"自动计算，不由用户直接设置。
- 红冲付款（付款单作废/冲销）回退累计核销额，付款状态相应回退。

## 审查提示

审查本状态机时，使用 `docs/skills/state-machine-business-review-prompt.md`，重点检查：
- 三轴状态分离是否真正独立（付款状态是否被误当作工作流状态）。
- 反审核的冲销前置是否覆盖所有已生成结果（库存、凭证、核销）；反审核目标态是否为 REJECTED（不是 UNSUBMITTED，见 `../domain-design-guidelines.md` §11.4）。
- 角色职责分离（采购员 vs 审核人 vs 管理员）是否落实。
- 三单匹配失败路径是否在发票审核时拦截。
