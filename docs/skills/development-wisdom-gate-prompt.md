# Development Wisdom Gate Prompt

在非平凡实现工作的各产出节点（设计定稿前、计划提交审查前、实现声称完成前），将此提示作为 **AI 自检门控** 运行，用通用开发通识对当前产出做一轮系统性挑战。

## 与现有审计技能的关系

此提示**不替代** `open-ended-audit-prompt.md`、`multi-dimensional-audit-prompt.md`、`plan-audit-prompt.md` 等对象级审计技能。

| 提示 | 身份 | 时机 | 范围 |
|------|------|------|------|
| 本提示 | 开发者的自检意识 | 开发中，产出物声称完成前 | 跨维度的假设/一致性/深度 |
| open-ended-audit | 独立的攻击性审计者 | 工作声称完成后 | 隐藏的未知风险 |
| multi-dimensional-audit | 多维度审查者 | 高风险工作的独立审查 | 6+ 显式维度的系统化审计 |
| plan-audit | 计划审查者 | Plan active 前 | 单份 Plan 的完整性/可行性 |

本提示解决的是：**AI 在开发过程中天然缺少的自我怀疑能力**——它倾向于接受自己的第一个答案、停在表面完成度、不主动排查跨层矛盾。人工在这项目中做的"通用的检查"本质上就是这道门控。把它嵌入 AI 的开发流程本身，不是再加一道事后审计。

## 适用条件

满足**任一**条件时触发：

- 你刚完成了某份设计文档、计划文档或实现代码，准备声明完成
- 你在做一个非平凡的技术决策（选择框架机制、设计数据模型、确定接口契约）
- 你在多处修改后准备聚合提交
- 你发现自己在对一个不确定的事情说"应该是这样的"

## 门控维度

### 0. 假设面出（Assumption Surfacing）

列出你当前产出依赖的**全部关键假设**，逐条标注证据强度：

- `proven` — 有平台源码、官方文档、或实验数据支持
- `inferred` — 从类似模式推导，未直接验证
- `guessed` — 你认为"应该是这样"，没有直接证据
- `unstated` — 你刚刚意识到你默认了一个条件但没说

**典型检查项：**
- 你是否假设了一个平台机制的存在，但没有查平台源码或文档确认？
- 你是否假设用户的需求就是字面意思，而没有考虑更深的意图？
- 你是否假设"这个做法上一个项目这样做 ok，所以这个项目也适用"？
- 你是否默认了一个配置/约束/值的存在，而没有确认？

**反例**（来自 nop-app-erp 06-22）：AI 假设 nop 平台跨模块引用只能走纯外键+I*Biz，因为"9 个内置模块零跨引用"。实际 `entity.xdef` 明确支持 `notGenCode="true"` 机制，且平台代码生成器自身就在用。AI 漏看了关键文档就下了结论。

### 1. 深度充电（Completeness Depth Charge）

不要接受第一层答案。对每个"已完成"的声明，追问：

- "第一层我做完了。第二层呢？"—— 当你完成了主功能，检查：边界条件、异常路径、非法输入的守卫、并发/幂等、清理/回滚
- "我覆盖了显式的需求。隐式的呢？"—— 用户没说但业务常识要求的：如删除前检查引用、停用时检查活跃单据、状态迁移的非法守卫
- "主路径跑通了。变体呢？"—— 列表为空时、值为 null 时、极端值（超大/超长/负数）、多语言/多币种场景

**典型检查项：**
- 你做的这个改动，是否只覆盖了用户明确说的那一种情况？
- 有没有"用户没说但合理预期应该处理"的情况？
- 如果有 N 个相似的修改点，你是否全部处理了，还是只处理了前几个就停了？

**反例**（来自 nop-app-erp 06-22）：AI 按计划"取消核心实体的注释"，激活了 58 个 to-one，认为完成了。用户一句"所有模块引用都改正确了吗" → 重新审计发现还有 **179 个仍被注释**，projects 域甚至 0 个生效。

### 2. 跨层一致性扫描（Cross-Layer Consistency Scan）

你的改动涉及的每一层都应做一致性检查：

- **意图层 → 文档层**：设计文档描述的意图和限制是否在实现中被忠实体现？
- **文档层 → 代码层**：文档中声明的行为、状态机、边界条件是否和代码实现一致？
- **代码层 → 配置层**：配置项的存在/默认值/格式是否正确？
- **代码层 → 测试层**：测试是否覆盖了你声称支持的所有路径和断言？
- **当前项目 → 平台/框架**：你的用法是否符合平台文档/源码/示例的最佳实践？
- **A 文件 → B 文件**：你修改的文件是否与其他文件中描述的规则/命名/模式一致？

**典型检查项：**
- 文档说用 A 方案，你代码实现了 B 方案——差异有记录吗？
- 你在一个域实现了一个模式（如状态机），是否与全局设计指南中定义的模式一致？
- 你引入的新配置项，是否在所有需要的配置文件中都有注册？

### 3. 意图忠实度（Purpose Fidelity）

检查你的产出是否真正解决了它要解决的问题：

- **症状 vs 根因**：你是修了表面的问题（症状），还是修了根本原因？
- **字面需求 vs 深层需求**：用户说的是 A，但深层需求可能是 B——你的方案同时满足了两者吗？
- **局部最优 vs 全局一致**：你的方案在局部看很漂亮，但在系统整体架构中是否引入了不一致？

**典型检查项：**
- 用户/需求说"加一个 X 按钮"——你做了。但用户的深层需求可能是"需要能取消这张单据"，也许应该是一个完整的取消流程而不是一个按钮。
- 你修复了一个 bug——但根因可能是某个设计决策本身有问题，你只打了补丁。
- 你的实现引入了一个新概念/新命名/新模式——它与现有系统保持了一致，还是引入了新方言？

### 4. 生态约束感知（Ecosystem Constraint Awareness）

你的产出在上下游生态中是否成立：

- **上游约束**：你的修改依赖的外部条件是否都已满足？数据库表存在吗？服务启动了吗？配置生效了吗？
- **下游影响**：你的修改会不会破坏依赖你的部分？API 签名改了吗？行为语义变了吗？
- **平行影响**：与你的修改区域平行的其他部分是否需要同步修改？（如同类实体、同类页面、同类业务逻辑）
- **回归风险**：你的修改是否可能破坏已有测试？已有的已知良好基线？

**典型检查项：**
- 你改了一个共享工具类/基础设施 → 所有调用方是否都兼容？
- 你新增了一个必填字段 → 既有数据的缺省值怎么处理？存量 seed 数据需要更新吗？
- 你修改了一个 API/schema → 客户端是否需要同步更新？
- 你的改动是否对系统已有的测试基线构成回归风险？

### 5. 第一性原理验证（First-Principles Validation）

将你的方案还原为最基本的事实，自问：

- **必要性**：这个问题真的需要解决吗？还是我在解决一个不存在的问题？
- **简单性**：有没有更简单的方法？我是不是过度设计了？
- **可逆性**：如果这个决策错了，回滚成本高吗？
- **可理解性**：6 个月后的另一个 AI（或人类）看我的代码/文档，能理解我当时为什么这么做吗？

**典型检查项：**
- 你引入了一个新的抽象层/设计模式——它真的必要吗？能不能用更简单的方式解决？
- 你的实现有多个步骤——每个步骤的存在理由是什么？
- 你的方案假设了未来的某种变化——这个假设成立的把握有多大？

## 产出格式

按上述 6 个维度逐一输出各维度裁决，格式如下：

```
## 维度 0 — 假设面出
- [假设名]: {evidence_strength} — 一句话描述
- ...

## 维度 1 — 深度充电
- 第一层（显式覆盖）: ✓ 覆盖 / ✗ 缺口
- 第二层（边界/异常/隐式）: ✓ 覆盖 / ✗ 缺口
- ...

## 维度 2 — 跨层一致性
- {层对}: ✓ 一致 / ✗ 不一致（说明）
- ...

## 维度 3 — 意图忠实度
- {检查点}: ✓ 符合 / ✗ 偏离（说明）
- ...

## 维度 4 — 生态约束
- {检查点}: ✓ 无影响 / ✗ 有影响（说明）
- ...

## 维度 5 — 第一性原理
- {检查点}: ✓ 成立 / ✗ 不成立（说明）
- ...

## 综合裁决
通过 | 有条件通过（列出条件）| 不通过（列出阻塞维度）
```

## 与项目定制化层的关系

**本提示是纯方法论的，不假设任何平台/框架/项目类型**。它捕获的是通用开发通识——在任何业务产品开发中，一个严谨的开发者都应该问自己的问题。

因此在复制使用时：
1. 本提示**不需要**项目定制化层（不同于 `nop-platform-conformance-audit-prompt.md` 等需要项目注入）
2. 但各维度的**搜索加权**可以项目化——例如 Nop 项目可在维度 0 加权"平台源码证据"，ERP 项目可在维度 4 加权"业财过账/数据删除保护区域"
3. 各维度下的**典型检查项**可以从项目具体失败模式中不断补充
