# 开放式审计提示


> **项目定制化层（nop-app-erp）**：使用本提示前必须先读 `docs/skills/README.md §项目定制化层（nop-app-erp）`，将本仓库的保护区域（`module-<domain>/model/*.orm.xml` ask-first、会计/财务/数据删除）、验证命令（`mvn clean install -DskipTests`）、命名约定（`Erp<Domain>` 实体前缀、`erp-<short>/<dict>` 字典、`erp.err.<short>` ErrorCode 前缀）和已知失败模式注入上下文。本提示的通用默认值在本仓库不充分。


当结构化审计检查清单可能遗漏隐藏问题且审查者应积极搜索未知未知数时使用此提示。

这是通用默认提示。复制模板后，根据项目的真实失败历史、保护区域、命名约定和误报容忍度进行调整。

```text
阅读 `AGENTS.md`、`docs/index.md`、活动需求和 owner docs、活动计划（如果存在）、最近日志和实时更改代码。

审计对象是当前工作涉及的**完整上下文**——需求、设计、计划、代码、验证证据与流程合规的交集，**不是单个计划或单个工件**。上面列出的每一项输入都是上下文来源，不是对审计对象的限定；不要因为"活动计划（如果存在）"存在，就把计划当成唯一对象。

运行开放式审计：搜索标准检查清单之外、跨工件且跨维度的隐藏问题。如果工作暗示更深层次的风险，不要局限于任何单一类别（如仅模型正确性、仅测试覆盖、仅文档）。

反窄化自检：如果你发现自己只在核实**单个计划/单个工件**的内部声明是否正确（例如某计划声称的列名/字段/计数是否与源一致），你做的是**窄结构化审计**——应改用 `plan-audit-prompt` / `code-quality-audit-prompt` / `orm-model-audit-prompt` 等对象特定提示。开放式审计要求把视野拉到项目级：跨计划的虚假关闭、owner-doc 整体差距、路由/技能的系统性误用、跨批次才可见的重复失败模式、保护区域渗漏、文档漂移。

寻找隐藏问题，例如：
- 从未写下的假设
- owner-doc 差距
- 虚假关闭或薄弱证明
- 不匹配的路由或不必要的技能使用
- 仅偶然通过狭窄验证的脆弱代码路径
- 应该提升为可复用检查的重复失败模式
- 跨批次/跨计划才可见的系统性偏差（在任何一个计划内部看都永远正确）

像对抗性审查者一样行动，寻找默认流程可能遗漏的内容。

如果仓库的复制项目有已知的高成本缺陷、保护域或重复错误模式，把它们作为**搜索加权**（在这些区域多花时间），但**不要让它们收窄维度**：项目的"已知失败模式"清单常常偏向某一类（如全是 ORM/Java 微模式），若任由它主导，会把多维搜索压成单维核实。

按严重性排序，首先返回发现。
如果发现阻塞问题，说 `needs revision` 并明确指出需要修订的**对象**（是某个计划、某份 owner doc、某个流程，还是整批已关闭的工作），以及确切的隐藏风险或缺失的后续工件。
如果没有阻塞问题，说 `passes open-ended audit` 并按维度列出仍值得警惕的剩余未知数。
```
