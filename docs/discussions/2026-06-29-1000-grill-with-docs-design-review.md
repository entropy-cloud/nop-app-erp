### Q1: flow-overview.md §3 状态映射表——expresses which axis?

**发现**：`flow-overview.md` §3.1 的状态表用 `UNSUBMITTED / SUBMITTED / APPROVED / CANCELLED` 描述"单据状态"，但 `domain-design-guidelines.md` §11.1 明确使用双轴（docStatus + approveStatus）。UNSUBMITTED 是 approveStatus 的取值，CANCELLED 是 docStatus 的取值——表格可能混用了两轴。

**选项**：
- A（推荐✅当前选中）：表达 approveStatus（审批轴）。表格标题改为"审批状态映射（approveStatus）"，脚注说明 CANCELLED 属于 docStatus 独立轴，补充 docStatus 映射表。
  - 理由：表格的大多数取值是 approveStatus 的标准值（UNSUBMITTED / SUBMITTED / APPROVED / REJECTED），属于同一轴。
  - 利弊：明确后消除二义性，但正文需补充更多内容（docStatus 映射表）。
- B：表达的是"用户可见的综合状态"——混合两轴。
  - 理由：用户看到的单一状态可能聚合了两轴信息。
  - 弊端：模糊三轴分离设计的清晰性。
- C：纯文字简述不严谨，无需修改。
  - 理由：owner doc 不是正式规范。
  - 弊端：作为流程总览应准确反映设计意图。

**决议**：选 A。明确为 approveStatus 映射，补充 docStatus 映射表，脚注说明 CANCELLED 归属。

### Q2: 会计期间初始态 CLOSED——语义矛盾

**发现**：`domain-design-guidelines.md` §11.2 会计期间状态机写为 `CLOSED / OPEN / CLOSING / CLOSED_FINAL`，CLOSED 作为初始态——一个刚创建的会计期间不应是"已结账"状态。

**选项**：
- A（推荐✅当前选中）：改为 `NOT_OPENED → OPEN → CLOSING → CLOSED`。CLOSED 只作终态，新增 NOT_OPENED 作为预建但未开启期间的初始态。
  - 理由：CLOSED 的语义是"不可再修改"，作为初始态违反了"结账期间不允许新增/修改凭证"的约束。NOT_OPENED 语义清晰。
  - 利弊：新增一个状态（NOT_OPENED），但状态机语义更合理。
- B：保留 CLOSED 但文档特殊说明其"初始非终态"含义。
  - 理由：不改状态机，仅加注解。
  - 弊端：CLOSED 作为初始态与同一枚举中 CLOSED_FINAL 的语义冲突。
- C：保持原样，排序不代表迁移顺序。
  - 理由：枚举只是取值集合。
  - 弊端：明显误导读者。

**决议**：选 A。改为 `NOT_OPENED → OPEN → CLOSING → CLOSED`。

### Q4: 过账事件契约与幂等

**发现**：`finance/posting.md` 定义了异步过账机制（审核 → 发布过账事件 → 财务域生成凭证），但：
1. 没有定义 `PostingEvent` 的消息体结构
2. 兜底扫描和事件可能同时处理同一单据

**选项**：
- A（推荐✅当前选中）：在 posting.md 中补充 PostingEvent 字段定义（billType, billHeadCode, tenantId, acctSchemaId）。过账操作前置检查 `posted=true` 时直接跳过（幂等保证）。
  - 理由：事件契约是跨域集成的基础，必须明确。幂等检查防止重复过账。
  - 利弊：文档增加，但避免了一个常见的过账 Bug。
- B：只补充契约定义，幂等用 posted 字段天然保证。
  - 理由：posted 字段本身就是过账成功的标志，检查它即可。
  - 弊端：没有明确定义事件字段会在实现时引入歧义。
- C：不需要——兜底扫描和事件不会同时命中同一批次。
  - 理由：实际情况中两者时间差很大。
  - 弊端：在负载场景或重启后仍可能并发触发。

**决议**：选 A。补充 PostingEvent 契约 + 前置幂等检查。

### Q7: posted=true 单据的物理锁定策略

**发现**：已过账单据的数据完整性保护策略不明确。posted=true 后是否允许直接 UPDATE 行数据？

**选项**：
- B（推荐✅当前选中）：物理锁定——posted=true 的单据行锁定为只读，纠错只能走红冲/反向单。不允许任何直接 UPDATE。
  - 理由：已过账数据影响财务报表，任何直接修改都会破坏审计轨迹。这是 ERP 数据完整性的核心要求。
  - 利弊：安全性最高，但纠错路径更长（需要生成反向单）。
- A：管理员可强改，但强制记录变更前后审计日志。
  - 理由：灵活性高。
  - 弊端：强改后被审计发现的风险大。
- C：不做特殊锁定，依赖业务规范。
  - 理由：实现最简单。
  - 弊端：数据完整性完全依赖人治，不安全。

**决议**：选 B。物理锁定，只支持冲销纠错。

### Q8: 跨域同步调用（采购→库存）的容错

**发现**：采购入库审核同步调用 inventory 生成移动单（REQUIRED）。如果 inventory 模块不可用，整个采购入库事务回滚。

**选项**：
- C（推荐✅当前选中）：保持同步 REQUIRED。采购入库更新库存是强一致性要求，不可降级。库存不可用时调用方重试。
  - 理由：采购入库和库存写入必须在同一事务中，否则会出现"入库单已审核但库存未增加"的数据不一致。
  - 利弊：强一致性保证数据正确，但降低了可用性（库存模块故障时采购入库功能不可用）。
- A：改为异步事件 + 补偿回滚。
  - 理由：提高可用性。
  - 弊端：增加了最终一致性的复杂度，补偿回滚逻辑复杂。
- B：加熔断器，失败后走兜底队列。
  - 理由：折中方案。
  - 弊端：在熔断期间入库单已审核但库存未更新，数据不一致。

**决议**：选 C。保持同步 REQUIRED。

**发现**：每个域的 `model/app-erp-<domain>.orm.xml` 独立定义了自己的字典。跨域通用状态值（DRAFT/CANCELLED/APPROVED/REJECTED）的 code 和 description 是否一致未经审计。域专属值（CONFIRMED/COUNTING/INSPECTING）留在各域。

### Q10: 单据编号作用域

**发现**：单据编号（凭证号、订单号、移动单号）的作用域未定义——orgId 内唯一还是全局唯一？

**选项**：
- A（推荐✅当前选中）：orgId 内唯一。同一公司内单据编号连续不重复，不同公司可重复。
  - 理由：符合中国企业多公司独立核算的常见做法（每个公司独立出报表、独立税务申报）。
  - 利弊：实现简单（序列号按 orgId 分组），但不支持跨公司直接引用。
- B：全局唯一。
  - 理由：集团合并时可直接追溯。
  - 弊端：多公司时序列号空间膨胀，各公司编号无规律。
- C：不强制，各域自定。
  - 理由：最大灵活性。
  - 弊端：跨域/跨公司系统集成时可能冲突。

**决议**：选 A。orgId 内唯一。

### Q11: 三档数据删除策略

**发现**：AGENTS.md 将 data deletion 列为 protected area（ask-first），但设计文档没有系统性定义删除策略。

**选项**：
- A（推荐✅当前选中）：定义三档策略——① 草稿单据（DRAFT/UNSUBMITTED）允许物理 DELETE；② 已审核未过账单据只允许作废（docStatus=CANCELLED）；③ 已过账/已完成的单据完全禁止 DELETE，纠错只能走红冲/反向单。
  - 理由：草稿单据没有业务影响，允许物理删除减少数据膨胀；已审核单据已有审计价值，不允许物理删除。
  - 利弊：三档策略的边界清晰，但实现时需要在 BizModel 中统一拦截。
- B：全量软删除（deleted flag）。
  - 理由：最安全，所有数据都是逻辑删除。
  - 弊端：所有查询都需过滤 deleted=true，性能影响且实现复杂。
- C：不统一规定。
  - 理由：最大灵活性。
  - 弊端：各域实现不一致，审计线索被破坏的风险。

**决议**：选 A。三档策略。

### Q13: approveStatus 与 nop-wf 审批流的集成

**发现**：approveStatus 的值为 UNSUBMITTED / SUBMITTED / APPROVED / REJECTED，但 nop-wf 有自己的状态（ACTIVATED / SUSPENDED / COMPLETED 等）。两者是否需要新增 PENDING 态作为桥接？

**选项**：
- A：approveStatus 新增 PENDING 态：UNSUBMITTED → SUBMITTED（提交到 wf） → PENDING（nop-wf 流转中） → APPROVED/REJECTED。
  - 理由：明确区隔"已提交等待进 wf"和"wf 处理中"。
  - 利弊：多一个状态，但语义更精细。
- B：不用——SUBMITTED 已包含"等待审核"语义。
  - 理由：从业务视角看，"已提交"和"处理中"是一回事。
  - 利弊：状态少，实现简单。
- C（推荐✅当前选中）：nop-wf 状态独立管理，approveStatus 不跟踪 wf 内部状态。wf 完成的终态（APPROVED/REJECTED）回调更新 approveStatus。
  - 理由：保持业务状态机洁净，让 nop-wf 处理工作流特有的状态（如加签、转审、会签），approveStatus 只管业务终态。
  - 利弊：解耦效果最好，但需要设计 nop-wf 完成回调接口。

**决议**：选 C。nop-wf 状态独立管理，业务状态机只接受终态回调。

### Q14: 汇率时间点——业务日期 vs 过账日期

**发现**：业务单据使用业务日期汇率，但过账可能在几天后。如果汇率已变，本位币金额按哪个汇率计算？

**选项**：
- A（推荐✅当前选中）：用业务日期汇率固定本位币金额。凭证行同时记录源币和本位币，本位币按业务日期锁定。汇率差异在期末汇兑损益调整中统一处理。
  - 理由：业务日期是交易发生的会计归属日期，汇率锁定在该日才符合"交易发生日确认"的会计核算原则。
  - 利弊：符合会计准则（ASC 830 / IAS 21），但不支持过账时即期汇率入账的需求。
- B：用过账日期汇率重新计算。
  - 理由：反映过账时的真实价值。
  - 弊端：不符合交易发生日确认原则，且同一业务单据在不同时间过账会得到不同的本位币金额。
- C：双汇率记录——业务日期汇率入账，差异单独生成汇兑损益凭证。
  - 理由：最精确。
  - 弊端：实现复杂，额外凭证增加系统负担。

**决议**：选 A。业务日期汇率固定，期末汇兑损益调整。

### Q16: 应收应付核销分摊规则

### Q31: 两步调拨与在途库存

**发现**：`flow-overview.md` §2.7 描述了两步调拨（发出→在途→接收）和在途库存，但 `inventory/README.md` 和 `cross-domain.md` 没有提到。

### Q32: 出库时批次/序列号的自动分配策略

**发现**：销售出库或生产领料时，系统需要从一个或多个批次中选择出库的批次。

- B：全 FIFO 不配置。

- C：全手动。
  - 理由：用户全权决定。
  - 弊端：增加用户操作负担，容易选错批次。

**决议**：选 A。FIFO 默认 + FEFO 可配（物料级 issueStrategy）。

### Q33: BOM 版本变更对现有工单的影响策略

**发现**：ORM 已有 `ErpMfgBomVersion` 支持 BOM 多版本。BOM 变更后，已创建的工单应使用旧版本还是新版本？

**选项**：
- A（推荐✅当前选中）：工单创建时锁定当时的 BOM 版本（快照）。BOM 变更不影响已创建的工单。管理员可手动将特定工单升级到新版本。
  - 理由：生产执行过程中 BOM 变更不应影响已开始的工单（否则已领料的子件可能不符合新版 BOM）。快照方式保证工单执行期间引用的一致性。
  - 利弊：需要 BOM 版本快照机制，但保证了数据一致性。
- B：BOM 变更后所有未执行工单自动升级。
  - 理由：总是使用最新 BOM。
  - 弊端：已部分执行的工单可能出现 BOM 版本不匹配。
- C：始终用最新版。
  - 理由：不维护版本。
  - 弊端：工单引用变化中的 BOM 可能造成子件不匹配。

**决议**：选 A。工单锁定创建时的 BOM 快照。

### Q34: 主数据审核流程

**发现**：物料/往来单位等主数据的新建和修改是否需要审核？设计文档中没有定义。

- C：全量审批。
  - 理由：最严格。
  - 弊端：用户更新备注也需要领导审核，不切实际。

### Q36: 负库存的会计处理

**发现**：`inventory/README.md` 提到负库存配置（`erp-inv.allow-negative-stock`），但没有说明负库存的财务影响。

**选项**：
- A（推荐✅当前选中）：负库存是临时容错机制——出库按移动加权平均成本出库（即使余额为负）；补货入库后成本自然平滑；不触发凭证生成（凭证在业务单据审核时生成，与库存余额正负无关）；不会自动调整。
  - 理由：负库存只应短暂存在（先出后入），不产生独立的会计事件。
  - 利弊：实现简单，但需强调负库存不是正常业务流程。
- B：出库成本按上次正库存时的成本计算。
  - 理由：避免负成本。
  - 弊端：在持续负库存时成本可能严重偏离实际。
- C：不允许负库存。
  - 理由：最安全。
  - 弊端：先收货后入单、跨期到货等常见场景无法处理。

**决议**：选 A。负库存是临时容错机制，财务上无特殊处理。

### Q37: 外币期末重估

**发现**：`finance/posting.md` 提到期末做汇兑损益调整，但规则不完整。

**选项**：
- A（推荐✅当前选中）：仅对货币性科目（现金/银行存款/应收/应付）重估——非货币性科目（固定资产/存货/股本）按历史汇率不动。用期末最后一天汇率。差异进当期汇兑损益。

- B：不做期末重估。
  - 理由：结算时确认即可。
  - 弊端：期间的汇率风险在报表中不可见。
- C：全量重估。
  - 理由：最全面。
  - 弊端：对历史成本计价的固定资产和存货重复估值不符合会计准则。

**决议**：选 A。货币性科目期末重估。

### 审查方法切换

### Q41: 配置管理体系

**背景**：全局和公司级的业务开关（负库存政策、成本方法、审批模式、过账模式等数十个开关）需要统一的存储和管理机制。

**选项**：
- A（推荐✅当前选中）：全局配置使用 nop-config 内置机制（app-beans.xml + @r-cfg）。业务层面的配置（与主数据/业务实体相关）单独建表（每个配置域按作用域设计，如仓库级配置关联仓库表）。
  - 理由：nop-config 适合技术性全局开关（如服务超时、日志级别），业务配置需要与业务实体关联（如"每个仓库的负库存政策"需在仓库表中配置）。
  - 利弊：分层管理最合理——技术配置走 nop-config，业务配置走实体关联表。
- B：全部用 nop-config 统一管理。
  - 理由：统一。
  - 弊端：与仓库/物料/公司等实体的关联需要额外映射逻辑。
- C：全部用通用配置表（ErpSysConfig key-value）。
  - 理由：一个表管所有配置。
  - 弊端：大量字符串解析，没有类型安全，运维困难。

**决议**：选 A。nop-config 技术配置 + 业务配置建表。

### Q42: 成本核算方法扩展

**背景**：Q29 只选了 FIFO 作为默认实现。产品化必须支持多种计价方法，按物料分类可配置。

**选项**：
- A（推荐✅当前选中）：在 `finance/costing-methods.md` 中展开全部四种计价方法——FIFO（逐层消耗 + 跨层拆分）、移动加权平均、标准成本（差异分摊）、个别计价法（批次/序列号追溯）。`ErpMdMaterial.costingMethod` 控制选择。
  - 理由：ERP 产品必须支持多种计价方法——FIFO 适合制造企业、移动平均适合贸易企业、标准成本适合成本管控型企业、个别计价法适合高单价物料。
  - 利弊：设计文档大幅增加，但这是 ERP 产品成本化的核心能力。
- B：第一版只做 FIFO + 移动平均。
  - 理由：够用。
  - 弊端：标准成本和个别计价法场景无法覆盖。
- C：延后实现时再做设计。

**决议**：选 A。展开全部四种计价方法。

### Q43: 审批框架可配置

**背景**：Q13 假定了 nop-wf 作为审批路径。产品化必须支持不同单据类型选择不同审批模式。

**选项**：
- A（推荐✅当前选中）：新增 `docs/architecture/approval-framework.md`——定义审批模式配置化框架。每个单据类型（采购订单/付款单/合同等）通过 `approval-mode` 配置：`NONE`（不需审批）/ `DIRECT`（直接选择审核人，无工作流）/ `WORKFLOW`（nop-wf 工作流，默认）。

- B：只用 nop-wf。
  - 理由：统一。
  - 弊端：所有单据都需要配置工作流，小企业可能觉得"杀鸡用牛刀"。
- C：DIRECT + WORKFLOW，不做 NONE。

**决议**：选 A。三种模式按单据类型可配置。

### Q45: 多公司层级

### Q48: posted 锁定等级可配置

**背景**：Q11 选了三级删除策略。产品化需要按实体可配置删除模式。

- A（推荐✅当前选中）：每个实体通过 `erp-{domain}.delete-policy.{entityName}` 配置：`PHYSICAL_DELETE` / `SOFT_DELETE`(isDeleted 标志) / `FORBID`。默认值按三级策略（草稿物理删除 / 已审核软删除 / 已过账禁止），但部署时可覆盖。
  - 理由：部分企业需要全量软删除（如审计要求严格的企业不希望任何数据被物理删除），部分企业希望主动清理草稿数据以减少存储。
  - 利弊：增加配置管理成本，但适应不同企业的合规要求。
- B：全局固定三级策略。
- C：全局软删除。

- C：ASYNC with SYNC fallback。

### Q64: 外部 API 集成模式——webhook 还是 polling

**发现**：ERP 需要对外提供 API 或事件通知（如通知外部 WMS 发货、通知电商平台库存变更），但设计文档没有定义集成模式。

**选项**：
- A（推荐✅）：webhook-only。外部系统注册回调 URL，ERP 在关键事件（库存变更、订单状态变更）发生时 POST 通知。不暴露内部 API 给外部。
  - 理由：webhook 是事件驱动的最简集成方式，ERP 保持边界安全（不直接暴露数据库或业务 API）。
  - 利弊：外部系统需要可接收回调；不支持外部系统主动查询。
- B：REST API + webhook。
  - 理由：外部系统可通过 API 主动查询数据。
  - 弊端：增加 API 安全与管理负担。
- C：不定义，实施时按需实现。
  - 理由：最大灵活性。
  - 弊端：各集成点风格不一致。

**决议**：选 A。webhook-only。

### Q76: 批次拆分与成本层交互

**发现**：批次追踪（`inventory/trace-chain.md`）定义了批次出入库。FIFO 成本层（`finance/costing-methods.md` 待扩展）定义了成本层计算。但两者交互没有文档：如果一个批次被部分消耗，剩余成本层的单位成本如何变化？

**选项**：
- A（推荐✅）：在 costing-methods.md（Q42 计划扩展）中明确：① 批次追踪下，成本层按"批次 + 入库批次"分割——同一批次的第一次入库和第二次入库形成独立成本层；② 部分出库先从最早成本层扣减，不改变剩余成本层的单位成本（FIFO 标准行为）；③ 批次拆分（一个批次拆为两个子批次）时，成本层按数量比例分割原成本层。
  - 理由：成本层与批次的交互是存货核算的核心精度问题，不设计清楚会导致成本层算法与批次追踪逻辑冲突。
  - 利弊：对 costing-methods.md 的扩展增加约 1 段，但消除一个常见的存货估值 Bug。
- B：批次和成本层互不影响——批次仅用于追溯，成本计算独立于批次。同一物料的所有库存按移动加权平均计算，不按批次区分成本。
  - 理由：实现简单，适合成本核算精度要求不高的企业。
  - 弊端：批次追踪和成本核算脱钩，高单价或高频单品（如医药、电子料）的批次成本不准确。
- C：延期——codegen 阶段只做实体生成，成本核算算法在 service 实现阶段再设计。
  - 理由：保持前期灵活。
  - 弊端：service 阶段发现批次与成本层交互的算法复杂度超出预期，可能需要回溯修改 ORM 设计。

**决议**：选 A。在 costing-methods.md 中明确成本层按批次分割规则 + FIFO 扣减顺序 + 拆分按比例分割。

### Q77: 金额精度与全局舍入规则

**背景**：业财一体化 ERP 的所有成本、税额、汇率、本位币金额都涉及小数精度。现状是**碎片化**的：
- `orm-model-audit-prompt.md` 规定 DECIMAL `precision="20" scale="4"`（本位币）/ `scale="8"`（汇率）——这是存储精度。
- `finance/ar-ap-reconciliation.md` 有 `erp-fin.reconcile-precision=0.01`（核销容忍误差）。
- `assets/split-merge.md` 定义了拆分舍入模式 `DOWN_UP`（最大项补差）/ `ROUND_ALL`。

但**没有一份文档统一定义**：① 行→单据→过账每一层的舍入顺序与精度；② 币种小数位（JPY=0、CNY=2、BHD=3）与存储 scale=4 的关系；③ 舍入模式（HALF_UP / HALF_EVEN 银行家舍入 / DOWN）；④ 行金额之和与单据头金额不等时的容差与调平科目（"待处理差价/尾差"科目）。

**选项**：
- A（推荐✅，**已按用户意见修正**）：在 `domain-design-guidelines.md` 新增「金额精度与舍入约定」节，**精度本身作为业务可配置项**：① **精度与舍入模式按业务可配置**——以公司/账套为单位配置本位币小数位（默认 2，特殊行业如石油可 3-4）、汇率位（默认 8）、舍入模式（默认 HALF_UP，可选 HALF_EVEN 银行家舍入 / DOWN），容差可配（默认 0.01）；存储列宽沿用 DECIMAL(20,8) 以容纳最大配置值；② 展示精度按币种小数位（`ErpMdCurrency.decimalDigits`）；③ **计算顺序为结构性硬约定（不可配）**：行内计算 → 行舍入 → 行汇总到头 → 头二次舍入；④ 行汇总与头金额的尾差自动记入预配置的"尾差/待处理差价"科目（按公司配置），单据层强校验差额 ≤ 容差。
  - 理由（含修正）：精度不是一刀切——不同行业（零售/石油/金融）对位数与舍入法的诉求不同，业务可配置避免硬编码。但"先行后头再舍入"的**顺序**和"尾差进专设科目"是消除"差 1 分钱"的结构性保证，必须统一不可配。
  - 利弊：精度配置化增加一个配置点（并入 Q41「可配置点清单」），但适应多行业；结构性约定不变。
- B：各域自定，仅在 ORM 存储精度层统一。
  - 弊端：10 个域会出现 10 种舍入习惯，跨域汇总（如项目成本汇总到财务）必然出现分位差异。
- C：固定一套全局精度（scale=4 + HALF_UP）不可配。
  - 弊端：无法适应石油/金融等特殊精度行业。

**决议（用户已确认并修正）**：选 A（修正版）。**金额精度（小数位/舍入模式/容差）按业务可配置**（公司/账套级，默认本位币 2 位 + HALF_UP）；**计算顺序与尾差科目为结构性硬约定**不可配。配置项并入「可配置点清单」（Q41）。

### Q78: 业务流水的长期归档策略

**背景**：Q11/Q49 覆盖了**删除**策略（草稿可物理删除、已过账禁止删除），Q65 覆盖了**上线迁移**。但都没有覆盖**长期运行后**的处置：一个 ERP 运行 5 年后，凭证、库存移动单、订单行将达到千万-亿级行。`finance/multiple-accounting-schemas.md` 定义了"账套归档"（整账套只读 + 导出备份），但那是**会计账套**层面的关闭，不是业务流水表的分区/冷存。

设计文档没有定义：① 是否按会计年度对凭证/移动单做表分区；② 历史年度数据是否可分离到冷库/归档库以保障活跃期查询性能；③ 归档后如何保证审计可查（中国法规：会计凭证保管 30 年）。

**选项**：
- A（推荐✅）：在 `docs/architecture/` 新增 `data-archival-strategy.md`，规定：① 凭证/移动单/订单按 `businessDate` 年度做**分区表**（partition by year），活跃年度留主表、历史年度可分离；② 归档不删数据——历史年度标记为"已归档"后查询走只读副本或归档库，但不改变主表查询接口（对业务透明）；③ 归档周期由 nop-job 调度，仅在年度结账完成后触发；④ 法定保管期内（中国 30 年）一律物理保留，仅做存储分层不做删除。
  - 理由：分区表是亿级行 ERP 的标准工程实践，避免"运行越久越慢"。30 年保管是《会计档案管理办法》硬约束。
  - 利弊：引入分区与归档库的运维复杂度，但这是产品长期可运行的前提。
- B：不分区，依赖数据库硬件扩容。
  - 弊端：5 年后查询性能下降不可逆，索引重建成本极高。
- C：仅靠 Q11 的删除策略清理草稿即可。
  - 弊端：已审核/已过账单据（占比绝大多数）无法删除，表膨胀无解。

**决议（用户已确认）**：选 A。新增 data-archival-strategy.md，年度分区 + 分层存储 + 30 年法定保留（中国《会计档案管理办法》）。

### Q79: 时区与时间字段语义

**背景**：Q45/Q46/Q47/Q51 把产品定位为**跨国/多法域** ERP（transfer pricing、国际税法、财年配置、汇率策略）。但全仓库 grep "时区/UTC/timezone" 几乎无业务层结论——只有 `businessDate` vs `createdAt` 的区分（`domain-design-guidelines.md` §10.1）。

跨国场景下时区是真实问题：① 跨公司单据（美国公司 23:00 与中国公司次日 12:00）的 `businessDate` 归属哪个年度/期间；② 调度作业（折旧、期末过账）按哪个时区触发"月末"；③ `createdAt` 在 DB 存 UTC 还是本地时，报表展示如何转换。

**选项**：
- A（推荐✅，**已按用户意见修正**）：在 `domain-design-guidelines.md` §10.1 扩展一小节规定：① **系统时间戳（`createdAt`/`updatedAt`/`postedAt`）与 `businessDate` 一律按公司时区存储**——公司时区由 `ErpMdOrganization.timezone` 定义，每个公司（orgId）的数据以其公司时区落库；② 因数据已按 orgId 分段，每个公司的账务自然处于该公司时区，报表与期间归属直接读无需转换；③ nop-job 调度按"公司所在时区的月末"触发；④ 跨公司合并报表按 `businessDate`（各自公司时区的自然日）聚合，不做时刻级对齐——会计合并以期间为单位而非瞬时时刻；⑤ 用户跨时区登录时，展示层可按用户偏好时区显示时间戳，但须明确标注"显示时区"，**公司账务基准始终是公司时区**，不做反向改写。
  - 理由（含修正）：因 orgId 已是数据隔离边界，按公司时区统一存储时间戳与业务日，使两者在该公司账务中始终一致，报表/期间无需任何转换。**UTC 存储方案被否决**——它要求所有读路径做时区换算，徒增复杂度，且与"账务归属于单一公司时区"的会计惯例冲突。
  - 利弊：单部署跨多个物理时区录入同一公司账时，"公司时区"仍是该公司账务的唯一基准（符合会计惯例）；用户显示时区仅作展示不改账。
- B：全系统统一用服务器本地时，不做时区。
  - 弊端：跨国部署时单据时间与本地业务日期错乱，月末跨日切错期间。
- C：UTC 存时间戳 + businessDate 独立（原推荐方案）。
  - 弊端：所有读路径需换算，与会计按公司时区出账的惯例相悖（已被用户否决）。

**决议（用户已确认并修正）**：选 A（修正版）。**时间戳与 businessDate 一律按公司时区存储**（orgId 分段，每公司一个 `ErpMdOrganization.timezone`）；**UTC 方案否决**。报表按 businessDate 聚合不做时刻对齐；用户显示时区仅作展示不改账。

### Q80: 审批委派/代理审批

**背景**：`roles-and-permissions.md` 定义了审核人角色，Q13/Q43 定义了审批模式（NONE/DIRECT/WORKFLOW）和 nop-wf 集成。但全仓库 grep "委派/代理审批/代办/代审/delegation/proxy" **零业务结论**。

实际场景：审核人请假/出差时，单据会阻塞。nop-wf 框架层面通常支持委派，但设计文档未定义：① 谁有权委派（仅本人/管理员代委派）；② 委派期间审计归属（委派人的责任不转移）；③ 委派到期自动收回；④ 高危单据（如大额付款）是否禁止委派。

**选项**：
- A（推荐✅）：在 `docs/architecture/approval-framework.md`（Q43 计划新增）中补充「委派规则」节：① 任意审核人可主动委派给指定代理人，需设置起止时间；② 委派期间代理人审批的审计日志同时记录"代理审批人"和"原审核人"，**责任不转移**；③ 委派到期自动收回，未处理单据回到原审核人；④ 高危单据（反结账/大额付款/资产处置，由配置定义"不可委派单据类型"）禁止委派。
  - 理由：委派是审批流的基础缺口，无委派则审核人请假即业务停摆。责任不转移是内控硬要求。
  - 利弊：需在审批框架文档中增加一节，但避免业务阻塞。
- B：不设计，依靠 nop-wf 原生能力即可，文档不展开。
  - 弊端：实现者可能忽略审计归属和禁委派规则，留下内控漏洞。
- C：仅支持管理员代委派。
  - 弊端：增加管理员负担，且仍缺审计/禁委派规则。

**决议（用户已确认）**：选 A。补充委派规则 + 责任不转移 + 不可委派清单。

### Q81: 汇率主数据来源与缺失处理

**背景**：多币种四件套（`domain-design-guidelines.md` §10.3）要求每笔外币单据携带 `exchangeRate`。Q14/Q51 定义了"用哪个日期的汇率"。但 grep "汇率来源/汇率源/每日汇率" 仅在 `master-data/ui-patterns.md` 出现"汇率表格（支持批量录入）"——即**纯手工录入**。

设计文档未定义：① 汇率由谁维护、何时录入（每日/每月）；② 单据录入时若当日汇率缺失如何处理（阻止/告警/取最近一日）；③ 是否支持外部汇率源自动取数（央行/商业银行 API）；④ 历史汇率是否可修改（已过账单据引用的汇率不可改）。

**选项**：
- A（推荐✅）：在 `master-data/` 下补充汇率管理小节（或扩展现有文档）：① 汇率按"币种对 + 生效日期"维护，每日一条，由财务员录入或外部源（可配）自动同步；② 单据取汇率规则——按 `businessDate`（或 Q51 配置的日期策略）精确匹配，缺失时取**最近一日有效汇率**并标记 `rateType=FALLBACK`，同时发预警通知财务员补录；③ 已被过账单据引用的汇率记录**锁定不可修改**，纠错只能改后续单据；④ 汇率源（手工/外部 API）通过 `erp-md.exchange-rate-source` 配置。
  - 理由：汇率缺失会导致单据无法录入或本位币金额错误。FALLBACK + 预警是"不阻塞业务 + 暴露问题"的平衡。
  - 利弊：增加汇率管理设计，但消除"汇率缺失阻塞全部外币业务"的风险。
- B：纯手工录入，缺失即阻止。
  - 弊端：节假日/忘录会阻塞所有外币单据。
- C：外部 API 自动取数，无需人工。
  - 弊端：API 故障时无人工兜底，且企业内部约定汇率（如集团内部结算汇率）无法支持。

**决议（用户已确认）**：选 A。汇率日表 + FALLBACK 兜底 + 预警 + 已引用锁定 + 来源可配。

### Q82: 存货明细账与总账的跨账对账

**背景**：`domain-design-guidelines.md` §7.3"数据对账"列了四项：库存余额 vs 流水汇总（库存域内部）、应收/应付余额 vs 发票汇总（财务内部）、**总账 vs 明细凭证**（财务内部）。但**没有**跨财务-库存的"存货明细账价值 vs 总账存货科目余额"对账。

在业财一体化 ERP 中，存货估值（`ErpInvCostLayer`/`ErpFinGlBalance`）与总账存货科目之间天然存在跨账一致性校验需求——若两者不等，说明过账链路有漏过/重过。这是月末结账前的标准控制点（SAP 的 MM-FI 对账）。

**选项**：
- A（推荐✅）：在 `finance/period-close.md`（或新增 `finance/subledger-gl-reconciliation.md`）中补充"辅助账-总账对账"：① 月结前自动校验"库存域各类物料期末存货价值之和 = 总账对应存货科目余额"；② 不平则生成对账差异报告，差异不为零禁止结账；③ 同样的校验扩展到 AR/AP open-item 余额与总账应收/应付科目、固定资产净值与总账固定资产科目。
  - 理由：跨账对账是发现过账漏/重的最后一道闸门，不做则错误会累积到年报才暴露。
  - 利弊：增加月末一个校验步骤，但这是会计内控的标准要求。
- B：仅做现有四项内部对账。
  - 弊端：库存域和财务域各自平衡，但两者之间不等也不会被发现。
- C：信任过账引擎，不做跨账校验。
  - 弊端：引擎 Bug 或并发漏过账无法被察觉。

**决议（用户已确认）**：选 A。月末做辅助账（存货/AR/AP/资产）↔ 总账跨账对账，差异不为零禁止结账。

### Q83: 单据编号断号/跳号处理

**背景**：Q10 定义了 orgId 内唯一，Q44 升级为可配置编号模板（`{prefix}{yyyyMMdd}{seq:6}` 等）。但全仓库 grep "断号/跳号/编号并发/缺号" **零结论**。

序列号生成在并发与事务回滚下会产生**断号**（事务回滚后序列不回收）。设计文档未定义：① 是否允许断号（中国会计实务中部分企业要求凭证号连续不可断）；② 凭证号是否与业务单据号策略不同（凭证号法定要求更严）；③ 断号是否提供补号机制。

**选项**：
- A（推荐✅）：区分两类——① 业务单据号（订单/出入库/收付款）：允许断号，序列不回收，编号仅作唯一标识不作连续凭证；② 凭证号：**不允许断号**，采用"凭证号在过账成功后才分配"（DRAFT 凭证无凭证号），事务回滚不消耗号段；③ 是否启用连续性校验由 `erp-fin.continuous-voucher-number`（默认 STRICT）配置；④ 提供断号查询报表供审计，但不自动补号。
  - 理由：中国《会计法》要求凭证号连续可追溯，业务单据无此要求。过账后分配号是凭证号连续性的标准工程解法。
  - 利弊：凭证号分配时机从创建推迟到过账，实现略复杂但合规性最高。
- B：全部允许断号，序列不回收。
  - 弊端：凭证号断号可能不满足法定审计要求。
- C：全部强制连续 + 补号机制。
  - 弊端：补号逻辑复杂且易引入并发问题，得不偿失。

**决议（用户已确认）**：选 A。业务号允许断号；凭证号过账后分配保连续，不补号。

## 第五轮：Nop 平台机制复用分析

### 分析目的

第四轮 Q77-Q83 的决议给出了"要做什么"，本节回答"用什么实现"——逐项对照 Nop Platform（`../nop-entropy/docs-for-ai/`）已有机制，判断每项决议是**直接复用平台能力**、**部分复用 + 应用层补充**，还是**平台无对应能力、纯应用层自建**。

复用程度分三档：
- 🟢 **直接复用**：平台已有机制直接满足决议，仅配置/插数据即可。
- 🟡 **部分复用**：平台提供基础能力，但决议的关键语义需应用层扩展或组合。
- 🔴 **纯应用层自建**：平台无对应能力，需在 BizModel / 自定义实体中实现。

> 一条贯穿原则（来自 `00-start-here/ai-defaults.md`）：**Model → Delta → Java**。能配置解决的不写 Java，平台已有的不重造。

### 总览矩阵

| 决议 | 平台机制 | 复用 | 应用层需自建/扩展 |
|------|----------|------|-------------------|

### Q83：凭证号连续性 — 🟢 直接复用（强，含两个关键约束）

**平台已有**（`03-modules/nop-sys.md` + `03-runbooks/generate-business-code.md`）：
- nop-sys **CodeRule（模式 `D{@year}{@month}{@seq:5}`）+ Sequence（`NopSysSequence`）**，正是为此设计。
- **`cacheSize` 决定连续性**（`generate-business-code.md` "Sequence 并发模型"节）：
  - `cacheSize > 0`：进程缓存号段，**吞吐高、崩溃会跳号** → **业务单据号**（Q83 决议"业务号允许断号"）。
  - **`cacheSize = 0`：每次回库，慢但不断号——文档明示"财务凭证等需连续的场景用"** → **凭证号**（Q83 决议"凭证号保连续"）。
- ORM 列 `tagSet="code"` → autoExpr 自动集成（`doSave` 时自动回填编号）。

**两个关键约束（应用层必须遵守）**：
1. **取号在 `REQUIRES_NEW` 独立事务，不被外层回滚**——刻意设计以避免重号。这意味着：若凭证号在过账事务**早期**分配且事务随后失败，号码会被"浪费"形成断号。因此 Q83 决议的"**凭证号过账后分配**"是必须的——在过账逻辑的**临近提交阶段**才调用 `codeRuleGenerator.generate()`，确保取号前业务已基本成功。
2. **`resetType` 字段当前实现不会按日/月/年自动归零**（`generate-business-code.md` 明确警告）——按年/月重置需**应用层用多个 seqName**（如 `voucher_seq_2026`、`voucher_seq_202601`）+ 模式 `{@year}{@seq:6}` 实现，**不要假设 `resetType=YEARLY` 会自动重置**。

**结论**：凭证号连续性**零成本复用** nop-sys Sequence（`cacheSize=0`）；应用层只管"分配时机（临近提交）+ 按期间多 seqName"。这条决议与平台能力的契合度最高。反模式警示：禁止自建 `SELECT MAX(no)`、禁止塞数据库触发器、禁止依赖 `resetType` 自动重置。

## Q78 深度分析：分区 DDL 与 nop-entropy `ddl.xlib` 机制

> 本节回应"分区操作是否应仿照 `ddl.xlib` 提供数据库兼容机制"的问题。结论：**应该，且这是唯一符合 Nop 平台范式的路径**。原 Round 5 把 Q78 标为 🔴（纯手写）是错的，特此展开修正。

### 1. nop-entropy 的 DDL 跨库机制是怎么实现的

经源码核查（`nop-persistence/nop-orm`），DDL 生成是一个**基座 + 方言覆写**的两层结构，与 `control.xlib`（控件）、`xlib` Delta 定制同一套思想：

**(a) 基座 `ddl.xlib`**（`_vfs/nop/orm/xlib/ddl/ddl.xlib`）
- 定义全部 DDL 标签的**默认实现**：`CreateTable`、`DropTable`、`AddColumn`、`ModifyColumn`、`AddIndex`、`AddUniqueKey`、`AddTenantIdForTable` 等。
- `CreateTable` 标签（行 52-74）通过组合子标签拼建表语句：列定义（`ColSqlType`/`DefaultValue`/`ColNullDef`/`ColCommentDef`）→ 唯一键（`TableUniqueConstraints`）→ 主键 → 收尾钩子 `<thisLib:TableCollation/>`。

**(b) 方言文件 `ddl_{dialect}.xlib`**（mysql/postgresql/oracle/mssql/mariadb/h2/h2gis/duckdb 共 8 份）
- 以 `x:extends="ddl.xlib"` 继承全部标签，仅**覆写**该库有差异的标签。
- 覆写手段：同名标签重声明整体覆盖；或 `<source x:override="prepend">` 增量前置。
- 例：`ddl_mysql.xlib` 覆写 `TableCollation`（行 54-56）输出 `CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_as_cs`；覆写 `ColNullDef`（MySQL 允许 `NULL` 显式写出）。

**(c) 派发器 `DdlSqlCreator`**（`io.nop.orm.ddl.DdlSqlCreator`）
- `getDmlLibPath(dialectName)`（行 50-61）：按 `/nop/orm/xlib/ddl/ddl_{dialectName}.xlib` 查找；找不到则剥 `.` 后缀回退（如 `mysql.8` → `mysql`）；仍找不到返回原路径（交由 VFS/Delta 解析）。
- `createTable(table)` → `XLang.getTagAction(dmlLibPath,"CreateTable").generateText(...)`。

**关键先例——`TableCollation` 钩子（ddl.xlib 行 193-196）**：基座实现是**空 source**，纯扩展点。`CreateTable` 在闭合括号后调用它（行 73 `<c:unit>)</c:unit><thisLib:TableCollation/>`），各方言按需注入尾部 SQL。**分区子句在各库中位置与此一致（都在 `)` 之后），因此 `TablePartition` 完全可照搬此模式。**

核查结论：grep `partition` 在 `ddl/` 目录**零命中**——平台当前确实没有分区标签，但**扩展点与机制已就绪**，缺的只是一个新标签。

### 2. 跨库分区语法差异（决定标签抽象的形状）

| 库 | 分区模型 | 建表语法 | 新增分区 |
|----|----------|----------|----------|
| MySQL/MariaDB | 单表内分区 | `... ) PARTITION BY RANGE (YEAR(business_date)) (PARTITION p2024 VALUES LESS THAN (2025), ...)` | `ALTER TABLE ... ADD PARTITION (...)` |
| Oracle | 单表内分区 | `... ) PARTITION BY RANGE (business_date) (PARTITION p2024 VALUES LESS THAN (...))` | `ALTER TABLE ... ADD PARTITION ...` |
| PostgreSQL | 声明式分区（父表 + 子分区） | 父表 `... ) PARTITION BY RANGE (business_date)`；子表独立 `CREATE TABLE ... PARTITION OF 父 FOR VALUES ...` | `CREATE TABLE 子 PARTITION OF ...` |
| SQL Server | 分区函数 + 分区方案 | 需先建 `CREATE PARTITION FUNCTION` + `CREATE PARTITION SCHEME`，建表 `... ON scheme(col)` | 调函数/scheme，非 ADD PARTITION |
| H2 / DuckDB | 无原生分区 | 不支持 | — |

**形状判断**：MySQL/Oracle/MariaDB 与 `TableCollation` 同形（一条尾部子句），可一个标签直接输出。PostgreSQL 需输出"父表 PARTITION BY + 子分区 DDL"多条语句，SQL Server 需"函数+方案+ON"——这两类差异比 collation 大，但 **DDL 标签本来就支持输出多条语句**（`CreateTables` 即如此），`AddPartition`/`CreatePartition` 标签可分别处理。因此抽象层成立，只是各方言覆写体量不同。

### 3. 建议的落地方案（仿 ddl.xlib）

**模型层**（表达"哪些表分区、按什么列/粒度"）：在 orm.xml 实体上用 `tagSet="partition"` 标记需分区的表，分区细节（列、类型、年度范围）通过**实体 tag 配置**或**独立的分区描述**承载——不污染 IEntityModel 核心。理由：分区是**时变**的（每年 ADD PARTITION），不像列定义是模型期固定的，把"建表时的初始分区 + 运行期新增"分开更合理。

**DDL 标签层**（建议两个新标签）：

| 新标签 | 位置 | 基座行为 | 各方言覆写 |
|--------|------|----------|------------|
| `TablePartition` | `ddl.xlib` 的 `CreateTable` 中，紧随 `<thisLib:TableCollation/>` 之后调用 | 基座**空实现**（无分区） | mysql/oracle/mariadb 输出 `PARTITION BY RANGE (...) (...)`；postgresql 输出父表 `PARTITION BY RANGE` + 子分区（多条）；mssql 输出 `ON scheme(col)`（依赖预建函数）；h2/duckdb 空 |
| `AddPartition` | 独立标签，供年度归档作业（nop-job）调用 | 基座空 | mysql/oracle `ALTER TABLE ADD PARTITION`；postgresql `CREATE TABLE 子 PARTITION OF`；mssql 切换 scheme 范围；h2 空 |

**实现路径二选一**（按 NOP 范式优先级）：

1. **平台增强（推荐）**——在 nop-entropy 的 `ddl.xlib` 加 `TablePartition`/`AddPartition` 空钩子 + 各方言文件覆写。分区是所有 Nop 业务应用的通用需求，属于平台能力。改动需记入 `nop-entropy/ai-dev/logs/`（见 AGENTS.md）。
2. **应用层 Delta 覆写**——若不动平台，在 nop-app-erp 的 `/delta/nop/orm/xlib/ddl/` 下覆写 `ddl_mysql.xlib` 等，重新声明 `CreateTable`（或新增分区子标签）。代价：要在应用侧重复维护各方言，且无法复用给其他 Nop 应用。

> 与 `ai-defaults.md` 的"Model → Delta → Java"一致：优先走模型/平台层，写新 Java 代码是最后手段。手写裸分区 SQL 放 nop-db-migration 脚本属于**反模式**——它绕过了方言抽象，换库（如 H2 测试 → MySQL 生产）时分区 DDL 不会自动适配。

### 4. 对 Q78 决议与文档落位的修正影响

- **Q78 决议不变**：仍是"年度分区 + 分层存储 + 30 年保留"。

### 5. 自检：这是否与"应用项目不修改平台"冲突？

不冲突。nop-app-erp 当前处于 bootstrap 阶段（AGENTS.md），ORM 模型尚未代码生成。分区能力要么作为**平台贡献**（推荐，惠及所有 Nop 应用），要么作为**应用 Delta**（可接受，本应用自洽）。两种都不违反"不手改生成物"红线——ddl.xlib 是**非下划线源文件**，本就允许扩展。唯一要避免的是"在手改 nop-entropy 平台源文件"与"应用 Delta"之间含糊——选定一条并记录。

### 6. 方案精化：分区作为"增量迁移"而非"建表时耦合"（推荐，修正第 3 节）

> 用户洞察：**先建普通非分区表，后续用 ALTER 改成分区表**，把分区做成增量 SQL，用一个 xlib 封装分区全生命周期。此方案优于第 3 节"在 `CreateTable` 里加 `TablePartition`"——分区与建表解耦，代码生成始终产出通用非分区 DDL，分区成为**运行期 opt-in**。

**为什么更好**：
- 建表 DDL 保持通用——H2/DuckDB 测试、小客户部署一律非分区，零额外开销。
- 分区按部署/按时间启用——小客户永不分区，大客户表增长到阈值才转。
- 一个 xlib 封装分区**全生命周期**，与 `ddl.xlib` 同构（建表标签族 vs 分区标签族并存）。

#### 6.1 各库"非分区 → 分区"的 ALTER 可行性（关键核查）

| 库 | 能否 ALTER 既有表为分区表 | 机制 | 落地成本 |
|----|--------------------------|------|----------|
| MySQL / MariaDB | ✅ 直接 | `ALTER TABLE ... PARTITION BY RANGE COLUMNS(business_date) (PARTITION p2024 VALUES LESS THAN (...))` | 中（表重建，5.6+ 支持；分区列须在所有唯一键中） |
| Oracle | ✅ 在线 | `ALTER TABLE ... SET PARTITIONING ... ONLINE`（12.2+） | 中（在线转换，业务不中断） |
| **PostgreSQL** | ❌ **不支持直接 ALTER** | 必须新建分区父表 → `INSERT ... SELECT` 迁数据 → RENAME 换名（SWAP） | 高（全表复制 + 换名，非一条 ALTER） |
| SQL Server | ⚠️ 间接 | 需先建 `PARTITION FUNCTION`+`SCHEME`，再 `CREATE CLUSTERED INDEX ... ON scheme(col)` 重建 | 高（函数+方案+索引重建） |
| H2 / DuckDB | ❌ 无原生分区 | 测试环境直接跳过（标签基座空实现） | 零 |

**结论**：MySQL/Oracle/MariaDB 是"一条 ALTER 搞定"；**PostgreSQL 与 SQL Server 本质是"重建+迁移"**，但**这正是 xlib 抽象的价值**——把这些库特定的复杂步骤封装进一个标签，对调用方暴露统一接口 `ConvertToPartitioned(table)`，内部按方言决定走 ALTER 还是 SWAP。

#### 6.2 必须前置解决的硬约束：分区列须进入所有唯一键

MySQL/Oracle 的分区硬性要求：**分区列必须是每个唯一键（含主键 PK）的组成部分**。本项目单据头表当前 PK 是代理 `id`（BIGINT），`businessDate` 是普通列。若按 `businessDate` 分区，`ALTER ... PARTITION BY` 会**直接失败**，除非先改键结构。两条出路：

- **方案 A：建表时 PK 即为复合键 `(id, businessDate)`**——从模型源头消除障碍。~~推荐~~ **已否决**：会在 bootstrap 阶段把物理层约束前溢到 ORM 模型，污染模型纯净度。
- **方案 B（采纳）：分区是物理层关注点，PK 约束由框架层在 `partition.xlib` 内部解决**——ORM 模型 PK 保持 `(id)` 不变；`ConvertToPartitioned` 标签在方言覆写中**自行封装所需的键结构重建**（如 MySQL 先 `DROP/ADD PRIMARY KEY` 再 PARTITION BY，或采用不依赖 PK 的分区表达式）。应用层与 ORM 模型完全不感知。

> **决策（用户已确认）**：待分区表的主键**暂时不修改**，PK 保持 `(id)`。分区属于**物理层**，"分区列须进 PK"的约束**后续在框架层（`partition.xlib` 的 `ConvertToPartitioned` 方言实现）想办法解决**，不前溢到 ORM 模型。这保证 `<domain>/model/*.orm.xml` 保持业务语义纯净，物理实现细节全部下沉到框架 xlib。

#### 6.2.1 关键解耦：ORM 逻辑主键 ≠ 物理主键（源码核查）

用户洞察："orm 层看到的 id 还是 id，物理层的主键并不需要是 orm 的主键。"——经 nop-entropy 源码核查，**此解耦成立**：

- **ORM 层始终用逻辑主键**：`GenSqlHelper`（`sql/GenSqlHelper.java:121,142,663`）拼 `WHERE` 用 `entityModel.getPkColumns()`；`OrmSessionImpl.load`（`session/OrmSessionImpl.java:1230`）、一级/二级缓存键、乐观锁 `UPDATE ... WHERE id=? AND version=?`、关联装载——**全部以 ORM 逻辑主键（`idProp`=id）为准**，不触碰物理 PK 约束。
- **物理 PK 是 DDL 层产物**：`ddl.xlib` 的 `CreateTable`（行 70-72）按 `table.idProp` 生成 `PRIMARY KEY(...)`——**默认 ORM PK 与物理 PK 耦合**，但可由 DDL 层覆写解耦。
- **因此"只送 id 查询"无任何问题**：`id` 是全局唯一代理序列，`WHERE id=?` 永远唯一命中；即便物理 PK 为 `(id, businessDate)`，DB 仍按复合索引最左前缀 `id` 高效定位，返回 1 行。

**解耦的成立条件（不可违背）**：`id` 必须保持**全局唯一**（代理序列保证）。一旦 `id` 不唯一，`WHERE id=?` 将跨分区返回多行——但这违背代理键契约，本就不允许。

**框架层落位**：`partition.xlib` 的 `ConvertToPartitioned`（方言覆写）负责把物理 PK 改为 `(id, businessDate)`（MySQL 走 `DROP/ADD PRIMARY KEY` + `PARTITION BY`；PG 走父子表 SWAP），**ORM 模型与 ORM 运行时全程无感**。这正是"物理层在框架层解决"的精确含义。

#### 6.2.2 一个需记录的物理约束：MySQL 分区表不支持外键

`ConvertToPartitioned` 落地时须注意：**MySQL/MariaDB 分区表不支持物理外键约束**（FK 指向分区表或从分区表引出均不被引擎强制）。影响评估：
- **ORM 关联不受影响**——nop-entropy 关联是 ORM 层逻辑 join（`IEntityJoinConditionModel`），不依赖物理 FK，分区后照常工作。
- **仅丢失 DB 级引用完整性**——孤儿数据需靠应用层（Q70 的 `@RefLink` + 消弧事件）兜底，而非物理 FK。
- **结论**：此约束不阻塞分区方案，但 `data-archival-strategy.md` 须显式说明"分区表无物理 FK，引用完整性由 Q70 应用层机制保证"。

#### 6.3 单 xlib 设计：`partition.xlib`（与 `ddl.xlib` 并列）

新增**一个**独立标签库 `/nop/orm/xlib/partition/partition.xlib`（基座）+ `partition_{dialect}.xlib`（方言覆写），封装分区全生命周期，与 `ddl.xlib` 同构、同派发机制（复用 `DdlSqlCreator` 的 `dialectName` 路径约定）：

| 标签 | 调用时机 | 基座行为 | 方言覆写要点 |
|------|----------|----------|--------------|
| `ConvertToPartitioned` | 一次性迁移（部署启用分区时） | 空（H2/DuckDB 等无分区库直接 no-op） | mysql/mariadb/oracle：单条 `ALTER ... PARTITION BY`；postgresql：生成"建父表+迁数据+RENAME 换名"多语句脚本；mssql：建 FUNCTION+SCHEME+重建聚集索引 |
| `AddPartition` | 年度新增分区（nop-job 每年初触发） | 空 | mysql/mariadb/oracle：`ALTER TABLE ADD PARTITION`；postgresql：`CREATE TABLE 子 PARTITION OF 父 FOR VALUES`；mssql：`ALTER PARTITION SCHEME`+`ALTER PARTITION FUNCTION` split range |
| `DropPartition` / `ArchivePartition` | 历史年度归档（Q78 归档作业） | 空 | mysql/mariadb：`ALTER TABLE DROP PARTITION`（瞬时删整分区，归档最爱的 O(1) 操作）；oracle：`ALTER TABLE ... DROP PARTITION`；postgresql：detach 子分区；mssql：`ALTER TABLE SWITCH`+merge |
| `ListPartitions` | 巡检/对账 | 基座返回空 | 各库查 `information_schema` / `DBA_TAB_PARTITIONS` / `sys.partitions` |

**调用入口**：一个 `PartitionDdlCreator`（仿 `DdlSqlCreator`），按 `dialectName` 加载 `partition_{dialect}.xlib`，提供 `convertToPartitioned(table, config)` / `addPartition(table, range)` / `dropPartition(table, name)` 方法。归档作业（Q78 BizModel）与年度新增作业（nop-job）只调这些方法，**完全不接触裸 SQL**。

#### 6.4 修正第 3 节的结论

- **撤回**第 3 节"在 `ddl.xlib` 的 `CreateTable` 里加 `TablePartition` 标签"的建议——那会把分区耦合进建表，违背"建表通用、分区增量"的更好范式。
- **保留**并强化为：分区独立成 `partition.xlib`（建表后增量），`AddPartition`/`DropPartition` 走同一 xlib。
- **无 ORM 前置依赖**：待分区表 PK 保持 `(id)` 不变（用户已确认）。"分区列须进 PK"等物理约束全部由 `partition.xlib` 的 `ConvertToPartitioned` 方言实现内部消化，不污染 ORM 模型。
- **平台 vs 应用**同第 5 节：`partition.xlib` 适合作为**平台贡献**（通用能力），否则应用 Delta 自管。

**对 Q78 文档落位的再次修正**：`docs/architecture/data-archival-strategy.md` 须写明——① 初始建表通用非分区，ORM 模型不承载分区信息；② 分区是运行期物理层增量，由 `partition.xlib` 封装 `ConvertToPartitioned`/`AddPartition`/`DropPartition` 全生命周期，PK 等物理约束在框架层解决；③ 归档用 `DropPartition`（MySQL/Oracle 的 O(1) 整区删除是分区架构的最大红利，远优于逐行 DELETE）。
