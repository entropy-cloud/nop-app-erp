  - 所有事实基于 `~/sources/erp/` 下源码实测，关键论断附绝对路径证据
- 产出核心架构结论：`docs/architecture/domain-module-split-analysis.md`
  - **结论**：nop-app-erp 进销存+财务一体化应按业务域拆成多个领域 Maven 模块（每域独立 `model/app-erp-<domain>.orm.xml`），而非单 `app-erp.orm.xml`
  - 平台证据：`OrmModelLoader.loadOrmModel()` 运行时遍历所有模块的 `/{moduleId}/orm/app.orm.xml` 自动合并（`nop-orm-model/.../loader/OrmModelLoader.java:43-66`）；`nop-quarkus-demo` 聚合 8+ 领域模块先例
  - 纠正了初始误判（曾倾向单 model 参考 nop-app-mall）：mall 是单一电商域，不适用业财一体跨域场景
  - "50 表阈值"无官方数字，但平台内置模块普遍 ≤21 实体，mall 54 是已验证最大单文件；erp 预估 80-120 表远超合理规模
  - 硬约束：跨领域实体不做 ORM 层 `refEntityName` 强引用（平台零反例），走 `I*Biz` + Processor
- Key decision: 业财打通采用 Metasfresh 的 `IAcctDocProvider` 跨模块聚合机制（类型安全注册 + EventBus 异步过账），凭证三件套采用赤龙范式（Voucher/VoucherLine/VoucherBillR + 模板）。
- 该架构结论为 analysis-grade，落地需经 plan audit + 人工确认；`model/*.orm.xml` 是 ask-first 保护区域，本次未修改。
