### 关键决策

- **坏账核销/恢复凭证经 CloseVoucherWriter 直接持久化（非 Provider）**：坏账四步分录来自余额/辅助账聚合（非来源单据），且核销/恢复会触发既有 ArApItem 变异——若走 `IErpFinVoucherBiz.post` 的 Provider 模型会触发 ArApItem 生成管道（坏账不应生成新辅助账）。复用 `CloseVoucherWriter`（损益结转/汇兑重估同范式），凭证结构一致可被 `voucherBiz.reverse` 红冲。
- **核销不进 P&L**：BAD_DEBT_WRITE_OFF 仅动 BS（借 Allowance/贷 AR），损失在计提时（BAD_DEBT_RESERVE 借信用减值损失）已确认；恢复（BAD_DEBT_RECOVERY）反转核销；释放（BAD_DEBT_RELEASE）是唯一贷记信用减值损失的场景（减少损失=增加利润）。测试断言核销凭证无 6701 科目。
- **Allowance 账面来源**：cumulative Σ(credit−debit) over 所有已过账非红冲 VoucherLine（allowance 科目），ErpFinGlBalance 当前未由过账引擎维护（参 ProfitLossClosingService），以 VoucherLine 为权威。
- **审批门控**：write-off/recovery 强制审批（config-gated `bad-debt-write-off-require-approval` 默认 true）；计提/释放是期间估计批量动作无审批（设计 Decision，期末门控+规则命中日志缓解）。

### 关键决策

- **核销落库机制**：复用 0300-3 `create`+`post` 路径（不重写核销原语），引擎只负责生成候选行。rejected：引擎直接操作 ArApItem 跳过核销单（丢失审计载体 + 绕过约束校验）。
- **分摊策略**：三策略经 config + 方法参数覆盖。BY_AMOUNT 非唯一金额退化为 UNMATCHED（报告列出等待人工），不强行 1:N 匹配。
- **定时执行载体**：nop-job-local 直调 BizModel（单次量未达数万阈值，过度工程 rejected）。nop-batch 分 chunk 迁移留 Deferred。
- **双面一致性口径**：经 ArApItem.sourceBillCode = 域级发票 code 关联，比较 finance settled vs domain paidAmount/receivedAmount。仅报告 + 告警，不自动修复。
