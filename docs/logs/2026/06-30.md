### 2026-06-30（mission 'crud' 健康检查：修复 8 个新域 codegen 阻塞）

- 触发：`missions/crud.json` 健康检查（`mvn clean install -DskipTests` + `mvn test`），8 个新业务域（crm/cs/hr/aps/contract/drp/logistics/b2b）首次纳入全量构建即失败。
- 根因均为 ORM 源模型（`module-*/model/*.orm.xml`）违规，导致 codegen 产物无法编译/生成：
  - **module-crm**：3 处 `NopAuthUser` to-one 缺 `notGenCode="true"` → 生成的 setter 调 `NopAuthUser.getId()`（该实体主键为 `userId`）。修复：补 `notGenCode="true" biz:moduleId="nop/auth"`，对齐 module-cs 既有模式（`module-cs/model/app-erp-cs.orm.xml:193`）。
  - **module-cs**：4 处 `stdDataType="bigdecimal"` 非合法 std-data-type → codegen 解析失败。修复：改为 `decimal`（`module-cs/model:409,437,588,589`）。
  - **module-hr / module-logistics**：6 处 `stdSqlType="TIME"` → Oracle 方言不支持（MySQL/PG 支持），DDL 生成对三方言全量失败。修复：改为 `stdSqlType="VARCHAR" precision="8"` + `stdDataType="string"`（存 "HH:mm:ss"）。
  - **module-drp**：`ErpInvDrpDockAppointment` 声明 `org` 关系（`leftProp="orgId"`）但缺 `orgId` 列。修复：补列 `propId="17"`（`module-drp/model:253`）。
- 验证状态：**全绿**。`mvn clean install -DskipTests` = BUILD SUCCESS（全模块含 `app-erp-all`）；`mvn test` = BUILD SUCCESS。
- 注意：上述属 codegen 阻塞性模型修正（非法枚举值/缺失列/与平台模式不一致），非业务语义变更。8 新域 codegen 骨架现已就绪，可进入 BizModel 深化。
