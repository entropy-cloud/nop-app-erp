### 2026-07-15（复核 BizModel @SingleSession 清理计划 blocker 根因，实证推翻「生产依赖」误诊 — plan 2026-07-14-2256-1）

- 执行 `docs/plans/2026-07-14-2256-1-bizmodel-singlesession-cleanup.md` 时发现 Phase 1 全 `[ ]` + `Plan Status: blocked`，附 Execution Blocker Record（2026-07-14）声称「移除 @SingleSession 致 finance 15+ 测试类失败，证伪计划核心前提，@SingleSession 是生产必需」。该 blocker 将仓库回滚至基线。
- **平台源码核实推翻 blocker 根因**：blocker 只看 `SingleSessionMethodInterceptor`（AOP，受 @SingleSession 触发），遗漏生产路径真正建 Session 的位置——GraphQL 引擎 `executionInvoker`：
  - `biz-defaults.beans.xml:6-9` `nopGraphQLEngine.executionInvoker = nopSingleSessionFunctionInvoker`；`SingleSessionFunctionInvoker.java:35-39` **无条件**对**每个** GraphQL 操作（query+mutation）调 `runInSessionAsync`，**不**以 @SingleSession 为门控。
  - `OrmTemplateImpl.java:204-211` `runInSession` 命中线程已绑 Session 时**复用**；`OrmEntityDao.saveEntity`→`OrmTemplateImpl.save`→`runInSession`→复用引擎 Session。
  - 推论：经引擎调用的 @BizMutation 已获单一 ORM Session，@SingleSession 在 BizModel 上**冗余但无害**（其 AOP runInSession 命中复用分支，零行为差异）。blocker「@BizMutation 不提供 Session、dao 各自 runInNewSession」论断不成立。
- **平台文档佐证**：`testing.md:49` 明文「方法多步 ORM 报 no-current-session 时，**不要给 BizModel 加 @SingleSession**——根因是测试直调缺 session 环境，改用 IGraphQLEngine 即可」。blocker 正好踩中此误判。
- **实证复现**：对 finance 单域执行注解移除（14 BizModel / 42 处）后 `mvn test -pl module-finance/erp-fin-service -am`：**197 run, 19 Failures + 38 Errors = 57 失败**。逐条归类 errorCode **全部**归因于「测试直调 BizModel 绕过引擎」：`update-entity-no-current-session`（多数）、`entity-not-in-session`、级联 `period-close.illegal-transition`、级联断言（SETTLED→OPEN）。**零失败指向生产依赖**。实测受影响 24 测试类 / 57 方法；9 模块估算 60-100+。事后 `git checkout` 回滚，仓库恢复 166 violations / 49 files 基线。
