### 关键决策

- **SYNC 同事务派发 + 失败隔离**（裁决 3）：默认 SYNC 保证监听者回退与红字凭证原子提交（强一致），但派发循环 try/catch 隔离单监听者失败——避免监听者抛错回滚红字凭证（违反法律效力）。失败落 5.1 异常工作台 PENDING 队列供人工处置。这与 `posting.md §冲销机制方向二` "凭证一旦过账具有法律效力不回滚" + "失败进告警队列人工处理" 一致。
- **复用各域既有 reverseApprove 语义**（裁决 4）：监听者不重复造回退逻辑，直接镜像各 `*Processor.doReverseApprove` 的状态迁移（approveStatus APPROVED→REJECTED + posted=false）。库存类单据无 approveStatus 轴，仅 posted 翻转。
- **保护区域零触及**：本期无 ORM 实体新增、无列新增、无字典扩展。SPI + DTO + Registry + 监听者全部为 service 层 Java bean（非 BizModel/IBiz），通过 IoC `ioc:collect-beans` 自动聚合。

### 实现偏离补注

- **`IErpFinVoucherBiz.reverse()` 无 `@Transactional(REQUIRES_NEW)`**（与计划基线描述略有出入）：Facade `ErpFinVoucherBizModel.reverse()` 仅 `@BizMutation`（不像 `post()` 叠加 `REQUIRES_NEW`），跟随调用方事务（REQUIRED）。该边界对 SYNC 同事务通知 + `txn().afterCommit` 均可用（Phase 1 Explore 核实）。裁决 3 的失败隔离（try/catch 包裹派发）独立于事务边界，无论 SYNC/ASYNC 均生效。
- **监听者跨实体经 IDaoProvider 而非注入 I*Biz**：各域监听者回退自身域实体用 `IDaoProvider`（同模块），与 `ErpPurInvoiceProcessor` 既有 reverseApprove 模式一致（其内部用 `invoiceDao().updateEntity`）。监听者不反向调用 finance，保持 DAG 解耦。
- **Non-Goal 遵守**：资产/费用/票据/薪酬/物流域回退监听器（SPI 已预留，按需 Follow-up）；冲销审批工作流、冲销 LOCKED 状态（开源共识不内建，Deferred）；通用领域事件总线（平台无，本期仅建 finance 范围 reversal 监听）。

### 关键技术决策

- **`@Optional` 注解**：可选 GraphQL 参数（cancelReason/resolution/beforeMinutes/npsScore/cesScore/comment）须标 `@io.nop.api.core.annotations.core.Optional`，否则 GraphQL 引擎抛 `field-null-arg`

- **避免循环依赖**：`ErpCsSurveyBizModel` 校验工单存在用 `daoProvider().daoFor(ErpCsTicket.class)` 只读校验，不注入 `IErpCsTicketBiz`（TicketBizModel 已注入 IErpCsSurveyBiz）
