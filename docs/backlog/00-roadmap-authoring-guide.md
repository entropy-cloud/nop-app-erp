# Roadmap Authoring Guide

## Terminology Note

Roadmap 有两层：

- **里程碑 (milestone)** 是粗粒度的能力分组——一个组织容器。里程碑**没有状态**。
- **工作项 (work item)** 是里程碑内的原子可标记单元。**只有工作项携带状态**（`todo` / `ready` / `done`）。

词汇层级：**roadmap → milestone → work item**。不要用 "phase / 阶段" 指代 roadmap 单元。

## Purpose

本指南定义 `docs/backlog/implementation-roadmap.md` 是什么、如何编写以及何时更新。路线图是可选的。仅当项目足够大，平面待办表格不再能显示全局进度时才使用它。

## What a Roadmap Is

路线图是里程碑索引和工作项状态表面。其核心用途：

1. 阅读路线图后，AI 或维护者知道哪些工作项尚未开始（`todo`）、已就绪待实施（`ready`）或已完成（`done`），无需重新遍历每个文档和代码库。
2. 记录每个工作项的依赖关系、owner doc 和可复用框架/平台功能。
3. 是选择下一个工作项的入口点。

## Containment: Milestones And Work Items

- **里程碑**把相关的工作项分组（如「核心业务循环」「18 域 CRUD」）。一个里程碑可包含一个或多个工作项。里程碑只是一个分组/标签——它**永远不携带状态字段**。其进度通过扫描所属工作项来读取。
- **工作项**是原子可标记单元——一个内聚、可独立交付的切片。它携带 `todo` / `ready` / `done`，是出现在状态块中的单元。一个无法由单次交付完成的工作项过大，必须拆分。

## Roadmap Role: Human–AI Alignment + AI Work Queue

路线图服务于两个具有不同访问模式的受众：

- **人工**将其用作指导和观察表面：他们决定存在哪些里程碑与工作项及其优先级顺序。人工读取 Work Item Status 以查看 AI 驱动的开发已达到何处。人工**不**审查单个实施工作。
- **AI**将其用作工作队列：它读取 Work Item Status，按既定顺序获取第一个 `todo` 工作项，自动实施它，然后通过将工作项移动到 `done` 来写回。AI **不**重新仲裁优先级、跳过或发明新工作项——如果路线图需要结构更改（新增/删除/重排序里程碑或工作项），AI 会标记它们供人工审查。

实施质量由结束审计强制执行，而非人工审查。路线图是人工在不阅读每份实施工作的情况下指导和观察进度的方式。

## Closed Loop

路线图与执行形成闭环开发循环：

1. AI 读取 Work Item Status 并获取第一个 `todo` 工作项（按既定顺序）。
2. AI 实施该工作项（人工不审查）。
3. 在结束审计通过时，AI 写回：工作项移动到 `done`，并且任何每组件/真相源状态同步。
4. AI 返回步骤 1 处理下一个 `todo` 工作项。

若实施完成但没有任何工作项被更新，说明该工作项大于一次交付能完成的范围，必须拆分。"当前进行中的工作"从活跃工作项读取，而不是 `project-context.md` 中的字段。

## What a Roadmap Is NOT

- 不是实施规格。没有实现步骤、复选框或关闭标准。
- 不是设计文档。它引用 owner docs；不重述业务规则。
- 不是待办列表。路线图是编排层；待办项引用路线图工作项。

## Status Tracking

**状态只存在于工作项上。** 里程碑永远没有状态字段；不要为里程碑添加状态。

| 状态 | 含义 | 操作 |
| --- | --- | --- |
| `todo` | 未开始 | 下一个工作的候选 |
| `ready` | 已通过草案审查，排队待实施 | 等待实施 |
| `done` | 已完成并通过结束审计 | 更新 owner docs 和日志 |

状态转换：

- 独立草案审查通过后：`todo` -> `ready`
- 独立结束审计通过后：`ready` -> `done`。在结束审计通过前**不要**标记 `done`。

## Structure

路线图通常按顺序包含：

1. 标题 — 最后更新日期、源文档
2. 目的 — 此文件是什么（固定文本，引用本指南）
3. Work Item Status — 唯一的动态状态块（工作项按里程碑分组）
4. 框架/平台复用 — 栈已提供的功能，团队无需重新构建
5. 当前基线 — 现有内容和主要差距的简短摘要
6. Milestones — 里程碑索引；每个里程碑列出其工作项（Work Item / 状态 / Owner Doc / 依赖 / 复用）
7. Work Item Details — 每个工作项的简短交付范围（无复选框）
8. 依赖图 — Mermaid 流程图
9. 横切关注点 — 跨工作项关注点
10. 规则 — 编写和更新规则

省略不适用的部分。

## Writing Rules

1. 保持粗粒度。Work Item Details 是简短列表，不是实现步骤。
2. 显式标注框架/平台复用，避免重建现有功能。
3. 保持状态准确。过时的状态比没有状态更糟。
4. 保持表格和图表之间的依赖一致；冲突时表格获胜。
5. 不要重复 owner-doc 内容。Work Item Details 仅列出交付范围。
6. 里程碑没有状态。只在所属工作项上跟踪状态；不要给里程碑标题加状态。

## Update Triggers

| 事件 | 更新 | 前置条件 |
| --- | --- | --- |
| 草案审查通过 | 工作项 `todo` -> `ready` | 草案审查已独立通过 |
| 结束审计通过 | 工作项 `ready` -> `done` | 必须等待结束审计通过 |
| 结束揭示新的复用机会 | 更新复用部分和该工作项 | 结束审计完成 |
| 新的或调整的 owner doc | 检查对 Work Item Details 的影响 | — |

## Anti-Patterns

- 将路线图编写为详细的实施规格
- 在路线图中重述 owner-doc 业务规则
- 让状态过时
- 在结束审计通过前标记 `done`
- 不标注现有框架/平台功能，导致重复重建
- 工作项大于一次交付能完成的范围，因此完成的实施不在路线图中更新任何内容，循环停滞
- 给里程碑加状态，或在别处维护第二个每项状态列（如带状态的覆盖表），创建与 Work Item Status 不同步的动态块
- AI 重新仲裁优先级或发明工作项，而不是按人工设定的顺序执行
- 在 `project-context.md` 中跟踪"活跃工作/当前阻塞/AI 自主权"作为字段——这些变化频繁且容易过时；改为从活跃工作项读取进行中的工作
- 用 "phase / 阶段" 指代 roadmap 单元
