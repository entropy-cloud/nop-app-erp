## 2. 模块化机制（核心参考价值）

### 模块发现与注册（`trytond/modules/__init__.py`）
- **模块独立发布**：业务模块以 Python egg 形式发布到 PyPI，通过 `pkg_resources.iter_entry_points('trytond.modules')` 发现（`modules/__init__.py:38-44` `update_egg_modules()`）
- **模块加载**：`import_module(name)`（`:55-67`）按 entry point 动态导入，模块可在独立可安装路径下
- **每个模块自带 `tryton.cfg`**：声明 `depends`/`xml`/`co_CACHE` 等，类似 Odoo 的 `__manifest__.py`

### Pool 机制（全局模型注册表，`trytond/pool.py`）
```python
class Pool(object):
    classes = {
        'model': defaultdict(OrderedDict),
        'wizard': defaultdict(OrderedDict),
        'report': defaultdict(OrderedDict),
    }
```
- 所有 model/wizard/report 注册到全局 Pool，按 `__register__(module_name)` 钩子注册（`pool.py:PoolBase.__register__`）
- **`PoolMeta` 元类**（`pool.py:PoolMeta`）：允许子模块用 `__setup__`/`__post_setup__`/`__register__` 钩子扩展父模块的 model——这是 Tryton 的"模型扩展"机制（类似 Odoo 的 `_inherit`，但更受控）

### 模型基类体系（`trytond/model/`）
丰富的 mixin 组合，业务 model 按需继承：

| 基类/Mixin | 文件 | 职责 |
|---|---|---|
| `ModelView` | `modelview.py` | 视图绑定 |
| `ModelSQL` | `modelsql.py` | SQL 持久化 |
| `ModelStorage` | `modelstorage.py` | 存储生命周期 |
| `Workflow` | `workflow.py` | **状态机转换声明式定义** |
| `MatchMixin` | `match.py` | 记录匹配 |
| `MultiValueMixin` | `multivalue.py` | 多值字段 |
| `ActiveMixin` | `active.py` | `active` 标志 |
| `TreeMixin` | `tree.py` | 树形结构 |
| `UnionMixin` | `union.py` | 联合视图 |

### 状态机（`trytond/model/workflow.py`，关键设计）
```python
class Workflow(object):
    _transition_state = 'state'
    cls._transitions = set()  # {(from_state, to_state), ...}

    @staticmethod
    def transition(state):
        def check_transition(func):
            @wraps(func)
            def wrapper(cls, records, *args, **kwargs):
                # 只对当前状态在 _transitions 集合内的记录执行
                ...
```
- **状态转换用装饰器声明**：`@Workflow.transition('done')` 标注方法，运行时自动校验当前状态是否允许转换到目标状态
- **转换规则集中声明**：`_transitions = {('draft','confirmed'), ('confirmed','done'), ...}`
- **声明式 + 编译期可检查**，比散落的 if-else 状态判断更安全

## 3. 业务实体清单

仓库内**无业务实体**（业务模块在 PyPI）。从框架机制可推断标准 Tryton 业务模块（`account`/`stock`/`production`/`purchase`/`sale`/`company`/`product`/`party`）的 model 结构：
- 每个业务 model 继承 `ModelSQL + ModelView + Workflow`（+ 需要的 mixin）
- 字段在 model 类用类属性声明（类似 Odoo）
- 视图/数据/权限在模块目录的 XML 文件

## 4. 业财一体机制

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Tryton 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **业务模块独立发布** | PyPI egg + `entry_points('trytond.modules')` 发现 | nop-app-erp 的领域模块（如 `app-erp-finance`）可独立版本化，主 app 聚合引入 |
| 2 | **声明式状态机（转换规则集中）** | `workflow.py:_transitions = set()` + `@transition(state)` 装饰器 | nop 用 DSL/字典声明单据状态机（`draft→submitted→approved`），转换规则集中校验 |
| 2a | **状态机横向对比** | 13 项目流程实现对比 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——Tryton 的声明式设计是 13 项目中最优雅的 |
| 3 | **Model mixin 组合而非深继承** | `model/` 下 11 个 mixin（Workflow/Match/Tree/Union/MultiValue...） | nop 实体按需组合能力（树形/多值/激活/工作流），避免深继承层级 |
| 4 | **`__setup__`/`__post_setup__`/`__register__` 模型生命周期钩子** | `pool.py:PoolBase` | nop 实体/模型加载暴露受控的初始化钩子，支持跨模块扩展 |
| 5 | **PoolMeta 受控扩展**（类似 Delta） | `pool.py:PoolMeta` 元类 | nop 已有更强的 Delta 机制，理念相通：扩展不修改原模型源 |
| 6 | **基础数据/动作/视图/序列号/翻译 框架内建** | `trytond/ir/`（action/sequence/translation/rule/queue/cron） | nop 复用 `nop-sys`（序列号/字典）+ `nop-rule` + `nop-job`，不需自建 |

## 6. 不建议借鉴的点

- **GTK/Sao 桌面/老式 Web 前端**已过时，nop 用 AMIS。
- **Python ORM**（modelsql.py）与 nop 的 Java ORM 不可直接照搬。
- **纯框架定位**（无开箱业务）对 nop-app-erp（要交付产品）不适用；但"模块独立发布"理念值得借鉴。
- **entry_points 动态发现**在 Java/Maven 生态不常用，nop 用 Maven 依赖 + VFS moduleId 发现。
