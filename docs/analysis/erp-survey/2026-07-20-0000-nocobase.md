# NocoBase 调研报告

> 国内团队开源的**可扩展低代码/无代码平台**（约 21k GitHub Stars），基于 Node.js + TypeScript + React + Sequelize。**对 nop-app-erp 的平台架构策略有对照参考价值**——NocoBase 与 Nop Platform 都是模型驱动/插件化的低代码平台，但路径截然不同。本报告聚焦架构对比，而非作为 ERP 业务设计参考（NocoBase 不包含任何预置 ERP 业务域）。

## 1. 基本信息

| 项 | 值 |
|---|---|
| 技术栈 | Node.js 18+ · TypeScript · Koa · Sequelize 6 · React 18 · Ant Design 5 · Formily 2 |

## 3. 核心架构——无代码引擎

### 3.1 Collection（集合/表）与 Field（字段）

**数据模型在运行时通过数据库中的元数据定义**，而不是通过代码生成。

**Collection 定义**（`core/database/src/collection.ts`）：
```typescript
interface CollectionOptions {
  name: string;                 // 表名
  fields?: FieldOptions[];      // 字段定义
  inherits?: string[];          // 继承（父集合）
  sortable?: CollectionSortable;
  autoGenId?: boolean;
  template?: string;            // 模板名称
  viewName?: string;            // 数据库视图
}
```

**Field 类型**（31 种，`core/database/src/fields/`）：
- 标量类型：string, text, boolean, number, integer, float, percent, date, datetime, json, blob, password, email, phone, url
- 选择类型：radio, select, multiple-select, checkboxes
- 关联类型：belongs-to, has-many, has-one, belongs-to-many
- 特殊类型：formula, sequence, sort, uuid, nanoid, snowflake-id, uid, virtual

### 3.2 Repository 模式

Repository（`core/database/src/repository.ts`）封装数据访问层：
- `create`, `find`, `findOne`, `findAndCount`, `update`, `destroy`
- 关联操作：`add`, `set`, `remove`, `toggle`
- 过滤系统：`$eq`, `$ne`, `$in`, `$and`, `$or`, 日期操作符等

### 3.3 UI Schema 系统

UI 与数据解耦——页面和区块定义为 JSON Schema（存储于 `uiSchemas` 集合）：

```text
用户操作 → REST API → Resourcer 中间件 → ACL 检查 → Action Handler → Repository → 数据库
```

UI Schema 是关键设计点——**区块（Block）可插拔**：表格区块、表单区块、看板区块、日历区块、甘特图区块等，通过 JSON 配置组合。

## 5. ACL 权限系统

### 5.1 核心引擎

ACL 是 Koa 中间件（`core/acl/src/acl.ts`）：

```typescript
class ACL {
  roles: Map<string, ACLRole>;
  availableStrategy: Map<string, ACLAvailableStrategy>;
  availableActions: Map<string, ACLAvailableAction>;
  allowManager: AllowManager;

  can({role, resource, action}) → CanResult | null
  middleware() → Koa 中间件
}
```

**权限评估流程**：
1. 用户请求携带 `role` 名称
2. `acl.can({role, resource, action})` 评估
3. role = `root` → 完全放行
4. 检查角色-资源-动作权限 → 返回参数或 null
5. 检查角色策略 → 返回策略参数或 null
6. 合并固定参数（如 `createdById = $currentUser`）
7. 返回 null → 403

### 5.2 数据域过滤

权限可以注入 SQL 过滤条件：
```json
{ "createdById": "$currentUser" }
```

模板变量支持：`$user`, `$nRole`, `$date`

## 7. 与 Nop Platform 的对比分析

### 7.1 架构对照

| 维度 | NocoBase | Nop Platform |
|------|----------|-------------|
| **驱动范式** | UI 驱动无代码（浏览器内 WYSIWYG） | 模型驱动低代码（XML → 代码生成） |
| **模型真相源** | 数据库 `collections` 表（运行时 JSON） | 磁盘上的 XML 文件（`*.orm.xml`，编译时） |
| **模式演进** | 动态：通过 API 在运行时增删字段 | 管道：Model → codegen → DB migration |
| **插件架构** | JavaScript/TypeScript 扩展 Koa 应用 | Java/JAR 通过 Nop IoC 容器扩展 |
| **技术栈** | Node.js + TypeScript + Koa + React + Sequelize | Java + Spring + Quarkus + Nop IoC |
| **ORM 层** | Sequelize（动态模型映射） | Nop ORM（模型驱动，编译时代理生成） |
| **业务逻辑** | 工作流节点 + 自定义指令插件 | BizModel Java 类 + xbiz XML（DSL） |
| **代码生成** | 无（运行时自省） | `nop-cli gen` 从 XML 生成完整模块链 |
| **Delta 定制** | Schema 节点替换 / Plugin 覆盖 | **内置 delta 机制**（x:extends, delta/ 目录） |
| **多模块拆分** | 无域边界，所有 Collection 同一数据库 | **严格域模块分离**（每个域独立 Maven 模块 + orm.xml） |
| **ERP 就绪度** | 平台 + 手动配置（无预置业务域） | **19 域预建 ORM 模型**（进销存/财务/制造/质量等） |

### 7.2 核心差异点分析

**1. 模型真相源**
- NocoBase：数据库是模式真相源，Collections 表存储元数据。API 随时可改结构。
- Nop：XML 文件是真相源。数据库从模型推导，改模型需 codegen + migration。

**2. 代码生成 vs 运行时解释**
- NocoBase：无代码生成。Sequelize 模型在运行时从集合元数据动态构建。
- Nop：`nop-cli gen` 生成完整 Java 源码（Entity/DAO/BizModel/Service/Page）。编译期类型安全。

**3. Delta 定制**
- NocoBase：Schema 补丁 / Plugin 行为覆盖，无正式分层机制。
- Nop：**正式 delta 机制**（x:extends, delta/ 目录），支持多层级覆写模型/配置/页面。

**4. 业务域预置**
- NocoBase：**零 ERP 业务域**。需要从头配置 Collection + Workflow + UI。
- Nop：**19 个业务域完整预置**（模块化 orm.xml + 生成的 CRUD + BizModel）。

### 7.3 NocoBase 的优势场景

- **快速原型**：非开发者可通过浏览器页面创建表和页面
- **插件生态**：80+ 内置插件覆盖常见需求（认证/文件/导出/通知/可视化等）
- **WYSIWYG 页面设计器**：所见即所得的区块配置
- **工作流可视化**：可视化工作流设计器，支持条件/循环/并行/人工节点
- **数据源代理**：可连接外部数据库和 REST API 作为数据源

### 7.4 ERP 场景的局限（与 nop-app-erp 相比）

1. **无预置业务域**——库存/采购/销售/财务/制造等需从零搭建
2. **无事务性业务逻辑抽象**——缺少 BizModel 的 `@BizMutation` 事务包装
3. **无复式记账原语**——没有科目表、凭证、会计期间等内建支持
4. **大规模复杂查询性能**——Node.js + Sequelize 在复杂 ERP 场景有挑战
5. **无编译期类型安全**——模式运行时动态，无编译期校验
6. **正式 delta 定制弱**——无法像 Nop 一样对任意层级做精确覆盖

## 8. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | NocoBase 证据 | 对 nop 的落地建议 |
