## 2. 模块拆分方式

父 POM 列出 ~140 个 Maven 模块，命名规范 `de.metas.<domain>[.<sub-feature>]`，业务边界极清晰：

- **会计核心**：`de.metas.acct.base`、`de.metas.acct.webui`
- **业务核心**：`de.metas.business`（聚合 order/invoice/payment/inventory/bpartner/costing/allocation）
- **垂直行业**：`de.metas.vertical.pharma.*`、`de.metas.vertical.healthcare.alberta`、`vertical-healthcare_ch`
- **支付网关（SPI 模式）**：`de.metas.payment.{esr,sepa,paypal,revolut,sumup}`
- **物流网关**：`de.metas.shipper.gateway.{spi,go,dhl,dpd,nshift,derkurier}`
- **制造**：`de.metas.manufacturing`、`de.metas.adempiere.libero.liberoHR`
- **UI**：`*.webui`、`*.mobileui`、`*.rest-api`

### 三层源码分层（关键工程实践）
每个模块源码按 `src/main/` 下三目录隔离（POM profile 自动挂载，`pom.xml:575-677`）：
- `java/` — 新业务逻辑（`de.metas.*`）
- `java-gen/` — 生成的 `X_*`/`I_*` 表模型（禁止手改）
- `java-legacy/` — 继承自 Adempiere 的 `org.compiere.*`

> 证据：`de.metas.acct.base/src/main/java-legacy/org/compiere/acct/Doc.java` 仍在 legacy 包。新代码不依赖 legacy 包，可逐步替换。**这点与 nop 的"生成物不可手改 + 保留层 + Delta"理念完全一致。**

## 4. 业财一体机制（与 iDempiere 共享 ADempiere FactAcct 内核）

### 共同内核
1. **每个业务单据表带 `Posted CHAR(1)` 字段**作为"是否已过账"标志（`X_C_Order:98`、`X_C_Invoice:82`）。
2. **`Doc` 抽象基类**（每种单据一个子类 `Doc_<Table>`），定义 `loadDocumentDetails()` + `createFacts(AcctSchema)` 两个钩子。
3. **`Fact`** = 一张单据在一个 `AcctSchema` 下产生的一组分录容器；**`FactLine extends X_Fact_Acct`**（直接是 `Fact_Acct` 表的一行）。证据：`org.idempiere.acct/.../doc/FactLine.java:62`。
4. **四种 PostingType**（`Fact.java:78-84`）：`Actual`/`Budget`/`Commitment`/`Reservation`——同一凭证可分预算/承诺/实际多套。

### Metasfresh 的现代化演进（关键参考价值）

**(a) 过账入口 = Spring `@Service IPostingService`**（`PostingService.java`）：
- `schedule()` → `sendAfterCommit`（`:64-74`）→ `postNow`（`:78-112`）→ `acctDocRegistry.get(acctSchemas, record).post(...)`

**(b) 异步 = OSGi EventBus（不是 DB 轮询）**（`DocumentPostingBusService.java`）：
- `sendAfterCommit`（`:104`）用 `trxManager.accumulateAndProcessAfterCommit` 在事务提交后投递到 `IEventBus`（`:86`），handler 订阅（`:89-93`）
- **比 iDempiere 30 秒轮询更即时**，延迟更低、不污染主事务

**(c) Doc 注册 = Spring 构造注入聚合（替代反射命名约定）**（`AcctDocRegistry.java:25-33`）：
- 注入 `List<IAcctDocProvider>`，按 `docTableName` 建 `ImmutableMap`（`:67-79`）
- `StandardAcctDocProvider.java:56-79` 用显式 `ImmutableMap.<String,AcctDocFactory>builder().put(I_C_Invoice.Table_Name, Doc_Invoice::new)` 方法引用登记 14 张标准单据
- **编译期类型安全，无反射字符串**（iDempiere 的 `DefaultDocumentFactory` 靠 `Class.forName("Doc_" + table)` 是脆弱设计）

**(d) 多 provider 跨模块自动注册（杀手锏）**：
- `ManufacturingAcctDocProvider`（`de.metas.manufacturing`）、`BankingAcctDocProvider`（`de.metas.banking`）、`PayrollAcctDocProvider`（liberoHR）各自是 `@Component`
- Spring 自动收集 → **新增单据类型 = 新建模块 + 一个 Bean，零改动核心**

**(e) 细粒度账户规则**（`InvoiceAcct.java:41-56`）：
- 持有按"specific→generic"排序的 `InvoiceAcctRule` 列表
- 按 `(acctSchemaId, accountConceptualName, invoiceAndLineId)` 三元组匹配返回 `ElementValueId`
- **同一科目可按产品/伙伴/发票行级别覆盖科目映射**

**(f) 科目解析器分层**（`de.metas.acct.base/.../accounts/`）：
- 13 类业务对象各一个独立账户解析仓储：`BPartnerCustomerAccounts`/`VendorAccounts`/`GroupAccounts`、`ProductAccounts`/`ProductCategoryAccounts`、`TaxAccounts`、`ChargeAccounts`、`ProjectAccounts`、`WarehouseAccounts`、`CostElementAccounts`、`CashAccountType`

### 多租户/多公司/多币种
- **AD_Client/AD_Org 机制保留**：`AD_Client_ID` 是租户、`AD_Org_ID` 是公司/组织，所有表带这两列。Metasfresh 包装为 `ClientId`/`OrgId`。
- **多币种**：每张单据带 `C_Currency_ID`；`FactLine` 同时存 `AmtSourceDr/Cr`（源币种）和 `AmtAcctDr/Cr`（本位币），转换由 `MConversionRate`/`CurrencyConversionContext` 完成。
- **多会计科目表**：`C_AcctSchema` 按 `AD_Client_ID` + `AD_Org_ID` 配多套，同一单据可在多套科目表下各生成一组 Fact_Acct。

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Metasfresh 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **「业务表 + Posted 标志」最小耦合业财一体契约** | `X_C_Order:98`、`X_C_Invoice:82`、`Doc.java:95-141` 表→DocBaseType→AD_Table_ID 映射 | nop 给所有可过账单据统一 `posted` 字段 + `(entityName, recordId)` 反查，不在业务表写外键到凭证表 |
| 2 | **Doc/DocLine/Fact/FactLine 四件套（凭证生成模板方法）** | `Doc.java:148` + `Doc_Invoice.createFacts:427-508` | nop 用 DSL/模板方法定义 `IDoc` 接口，每种单据一个 `<bizDoc>` 元模型，借贷规则用规则表/表达式而非硬编码 Java |
| 3 | **用 ImmutableMap<TableName, Factory> 注册单据类型（编译期类型安全）** | `StandardAcctDocProvider.java:56-79` | nop IoC 容器用 `Map<String, IDocFactory>` 注入，key 用 `table_name`/`entityName`，避免反射命名约定 |
| 4 | **多 AcctDocProvider 跨模块自动聚合** | `AcctDocRegistry.java:25-33` 注入 `List<IAcctDocProvider>` | nop 用 `@Inject` 集合注入，"凭证生成规则"做成可插拔模块，垂直行业（制药/医疗）单独打包 |
| 5 | **过账异步化用 EventBus 而非 DB 轮询** | `DocumentPostingBusService.java:104-110`（事务提交后投事件） | nop 用消息总线/事件队列，过账失败可重试且不阻塞单据提交；保留 `Posted` 字段做兜底扫描 |
| 5a | **EventBus 是 DocumentEngine 的工程演进** | 13 项目流程实现对比 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——Metasfresh 的 EventBus 是 iDempiere DocumentEngine 的现代化演进 |
| 5b | **现代化工作流包 + REST API 移动端工作流** | `de.metas.workflow.{WFNode,WFProcess,WFActivity}` + `de.metas.workflow.rest-api` | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——Metasfresh 将旧版 `org.compiere.wf.*` 重写为 `de.metas.workflow.*`，新增 REST API 工作流层支持移动端审批/拣货/制造 |
| 6 | **科目解析分层：按业务对象维度（13 类 AccountsRepository）** | `de.metas.acct.base/.../accounts/`（BPartner/Product/Tax/Charge/Project/Warehouse/CostElement） | nop 科目映射做成多维决策表（伙伴组/产品类别/项目/仓库 → 科目），用 nop 规则引擎或元数据驱动，而非 if-else |
| 7 | **多套并行凭证（多 AcctSchema + 4 种 PostingType）** | `PostingService.java:95` + `Fact.java:78-84` | nop 凭证表带 `acctSchemaId` + `postingType` 维度，支持同一业务在"管理账/税务账/预算账"下分别出凭证 |
| 8 | **三层源码分层隔离遗留** | `src/main/{java,java-gen,java-legacy}` + POM profile | nop 用代码生成器生成表模型，手写业务与生成物分目录，禁止手改——理念完全一致 |
| 9 | **同一事务提交后异步过账（事务边界清晰）** | `trxManager.accumulateAndProcessAfterCommit` | nop 主事务只落业务单据 + Posted=N，过账在 post-commit 异步执行，失败重试不影响主单据 |

## 6. 不建议借鉴的点

- **140 个 Maven 模块**对单一团队是过度拆分；nop-app-erp 应按业务域粗粒度拆（5-7 个领域模块）而非按 sub-feature 细拆。
- **Spring Boot 2.4** 已较旧（nop 用 Quarkus）；DI 模式可借鉴但具体栈不可照搬。
- **AD_Client/AD_Org 双层租户模型**是 ADempiere 历史遗产；nop 用 `nop-auth` 的多租户机制即可，不需复刻 AD_* 表。
