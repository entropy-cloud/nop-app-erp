# ERPNext 调研报告

> 全球第二的开源 ERP，主打"100% 开源无付费墙"。模块粒度比 Odoo 粗，但业财一体（GL 自动过账）机制对 nop-app-erp 极具参考价值。

## 1. 基本信息

| 项 | 值 |
|---|---|
| 技术栈 | Python · Frappe 框架 · MariaDB · JS（前端） |
| License | GPL-3.0（**100% 开源，无企业版功能墙**） |
| 定位 | 中型企业全能 ERP，无锁定 |
| 实测顶层业务域 | **~15 个**（`erpnext/erpnext/` 顶级目录） |
| 源码规模 | 约 148 MB · 5310 文件 |

## 2. 模块拆分方式（粗模块 + doctype 驱动）

ERPNext 顶层只有约 15 个业务域目录，每个域内用 **doctype + 子表 + purpose** 表达复杂度：

| 业务域 | 目录 | 核心 doctype |
|---|---|---|
| 采购 | `buying/doctype/` | `Supplier`、`Request for Quotation`、`Supplier Quotation`、`Purchase Order`(+`_item`)、`Supplier Scorecard` |
| 销售 | `selling/doctype/` | `Customer`、`Quotation`、`Sales Order`(+`_item`)、`Product Bundle`、`Sales Team`、`Installation Note` |
| 库存 | `stock/doctype/` | `Item`(+变体/属性/价格/条码)、`Warehouse`/`Warehouse Type`、`Stock Entry`(+`Stock Entry Type`/`_detail`)、`Stock Ledger Entry`、`Purchase Receipt`、`Delivery Note`、`Material Request`、`Batch`、`Serial No`、`Bin`、`Landed Cost Voucher`、`Stock Reconciliation`、`Putaway Rule`、`Quality Inspection` |
| 制造 | `manufacturing/doctype/` | `BOM`(+`BOM Item`/`BOM Operation`/`BOM Explosion Item`/`BOM Creator`)、`Work Order`(+`Work Order Item`/`Work Order Operation`)、`Job Card`(+`_Operation`/`_Time Log`/`_Scheduled Time`)、`Operation`、`Routing`、`Workstation`(+`Workstation Type`)、`Production Plan`、`Downtime Entry` |
| 财务 | `accounts/doctype/` | `GL Entry`、`Journal Entry`、`Sales Invoice`(+`_item`/`_payment`)、`Purchase Invoice`、`Payment Entry`、`Account`、`Cost Center`、`Fiscal Year`、`Accounting Period`、`Pricing Rule`、`Payment Term`/`Template`、`POS Invoice` |
| 其他 | `projects`/`crm`/`assets`/`maintenance`/`quality_management`/`subcontracting`/`support`/`shopping_cart`/`edi`/`regional` | — |

> **与 Odoo 的关键差异**：Odoo 顶层 200+ addon（细模块+模型继承）；ERPNext 顶层 ~15 模块（粗模块 + doctype + purpose）。后者更适合单一团队维护。

## 3. 业财一体机制（核心参考价值）

ERPNext 是**强一致性的业财一体**——业务单据 `on_submit` 即触发 GL 过账，`on_cancel` 即冲销，无需像 Odoo 那样"显式开票"。

### 统一入口
- `erpnext/accounts/general_ledger.py:34` `make_gl_entries(gl_map, cancel=..., merge_entries=..., update_outstanding=...)` → `save_entries()`（`:65`）+ `create_payment_ledger_entry()`（`:58`）
- 冲销：`:607` `make_reverse_gl_entries()`

### 业务单触发点（含库存单也过账）
- 采购入库：`stock/doctype/purchase_receipt/purchase_receipt.py:379` `on_submit → make_gl_entries()` / `:452` `on_cancel → make_gl_entries_on_cancel()`
- 销售出库：`stock/doctype/delivery_note/delivery_note.py:492` `on_submit` / `:513` `on_cancel`
- 销售发票：`accounts/doctype/sales_invoice/sales_invoice.py:458` `on_submit → self.make_gl_entries()`（方法 `:1024`），GL 构造委托 `services/gl_composer.py SalesInvoiceGLComposer`（`:1065`）

### GL Entry 结构（`gl_entry.json:9-57`）
- `posting_date`、`account`(Link→Account)、`party_type/party`
- `debit/credit` × **三币种**：`…_in_account_currency`/`_in_transaction_currency`/`_in_reporting_currency`（`:35-43`）
- `voucher_type`+`voucher_no`+`voucher_detail_no`（反查源单）
- `against_voucher_type`+`against_voucher`（核销关联）
- `cost_center`、`project`（核算维度）
- `is_advance`/`is_opening`

> **双账本并行**：`Payment Ledger Entry`（付款台账，`:58`）+ GL Entry，用于应收应付核销。

## 4. 库存单据设计（Stock Entry 一单多 purpose）

`Stock Entry`（`stock/doctype/stock_entry/stock_entry.json:149`）一张单覆盖 **13 种 purpose**：

```
Material Issue / Material Receipt / Material Transfer /
Material Transfer for Manufacture / Material Consumption for Manufacture /
Manufacture / Repack / Send to Subcontractor / Disassemble /
Subcontracting Delivery|Return / Return Raw Material to Customer / Receive from Customer
```

由 `Stock Entry Type`（doctype）做配置化预设。

**配套表**：
- **`Stock Ledger Entry`**（`stock_ledger_entry.json:10-46`）：不可变库存流水，`actual_qty`+`qty_after_transaction`+`incoming_rate`/`outgoing_rate`+`valuation_rate`+`stock_value`+`stock_value_difference`+`stock_queue`（FIFO 队列），靠 `voucher_type`+`voucher_no` 反查源单。
- **`Bin`**：`(item, warehouse)` 维度的汇总余额表（类比 Odoo 的 `stock.quant` 但更汇总）。

## 5. BOM / Work Order / Job Card（制造单据状态机）

### BOM（`manufacturing/doctype/bom/bom.json`）
- `item`(Link)、`quantity`（产出量 :150）、`uom`、`is_active`/`is_default`（支持多版本并存 :166/:177）、`with_operations`（:188）、`inspection_required`、`allow_alternative_item`
- 子表：`BOM Item`（`item_code`/`qty`/`source_warehouse`/`bom_no`/`operation` 绑定消耗工序）、`BOM Operation`（`operation`/`workstation`/`time_in_mins`/`fixed_time`/`hour_rate`/`operating_cost`/`batch_size`/`wip_warehouse`/`fg_warehouse`）、`BOM Explosion Item`（展开后缓存）

### Work Order（`work_order.json:124`）
状态机 **9 态**：
```
Draft → Submitted → Not Started → In Process →
Stock Reserved / Stock Partially Reserved →
Completed / Stopped / Closed / Cancelled
```
- 关键字段：`production_item`+`bom_no`、`qty`、`wip_warehouse`/`fg_warehouse`、`planned_start_date`/`planned_end_date`/`expected_delivery_date`、`skip_transfer`、`use_multi_level_bom`、`transfer_material_against` ∈ {`Work Order`,`Job Card`}
- `on_submit` 时 `create_job_card_from_wo()`（`work_order.py:612/988`）+ `make_stock_entry()` 产出领料/完工 Stock Entry。

### Job Card（`job_card.json:271`）
状态机 **8 态**：`Open → Work In Progress → Partially Transferred / Material Transferred → Submitted → Completed`（含 `On Hold`/`Cancelled`）
- 字段：`operation`、`total_time_in_mins`、`workstation`、`for_quantity`/`completed_qty`、`started_time`
- 子表：`Job Card Operation`、`Job Card Time Log`（实际工时）、`Job Card Scheduled Time`、`Job Card Item`

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | ERPNext 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **业务单据统一 `on_submit/on_cancel` 钩子触发 GL** | `general_ledger.py:34,607`；`purchase_receipt.py:379,452`；`delivery_note.py:492,513` | nop 建立统一 `on_post/on_reverse` 钩子，所有业务单据过账走同一入口，业务取消即冲销 GL |
| 1a | **钩子驱动 vs 流程引擎选型** | 13 项目流程实现对比 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——ERPNext 的 on_submit 钩子是"横切关注点"而非流程引擎，是 ERP 业财一体的核心模式 |
| 2 | **GL 行用"凭证指针"反查源单** | `gl_entry.json:24-31`（`voucher_type+voucher_no+voucher_detail_no` + `against_voucher_type+against_voucher`） | nop 凭证行不依赖外键强耦合，用 `(源单类型, 源单号, 源单行)` 三元组反查业务行 |
| 3 | **业务单直接驱动 GL（强一致业财一体）** | `purchase_receipt.py:379`、`delivery_note.py:492`、`sales_invoice.py:458` | nop 采用 ERPNext 模式而非 Odoo 的"显式开票"——业务单据 submit 即过账 |
| 4 | **多币种+多维度核算内建于 GL 行** | `gl_entry.json:35-43`（debit/credit × 三币种）+ `cost_center`/`project` | nop 凭证行原生支持多币种与可扩展会计维度，避免后期重构 |
| 5 | **`Stock Ledger Entry` 不可变流水 + FIFO 队列** | `stock_ledger_entry.json:10-46`（`stock_queue`/`valuation_rate`/`stock_value`） | nop 库存流水应不可变，含 `valuation_rate`/`stock_value`/FIFO 队列，业务单只生成移动单→写流水→更新余额 |
| 6 | **`Stock Entry Type` 配置化单据模板** | `stock_entry/stock_entry.json:149`（13 purpose） | nop 借鉴做"单据类型模板"，预设不同 purpose 的默认值/科目 |
| 7 | **Work Order + Job Card 双层 + 物料预留独立状态维度** | `work_order.json:124`（9 态）+ `job_card.json:271`（8 态）+ `Stock Reserved`/`Partially Reserved` | nop 工单分"计划层 Work Order"+"执行层 Job Card（含工时记录）"，物料预留作为独立状态维度 |
| 8 | **粗模块 + doctype 元数据驱动** | ~15 顶层域 vs Odoo 200+ addon | nop-app-erp 采用粗模块（purchase/sale/stock/manufacturing/finance），业务模型独立、财务横切（on_submit 钩子注入） |

## 7. 不建议借鉴的点

- **doctype 元数据系统**是 Frappe 框架特性，nop 有自己的 xmeta/orm 模型驱动，不需复刻 Frappe 的 DocType 设计。
- **13 purpose 的 Stock Entry 单一单据**虽然简洁，但单据类型耦合度高；nop 可保留 ERPNext 的 purpose 思路，但用"作业类型参数化 + 单据模板"而非硬编码 13 种分支。
