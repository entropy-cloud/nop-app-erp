## 2. 模块拆分方式（OSGi Bundle）

每个 `org.*` 子目录是一个 OSGi plugin，由 `MANIFEST.MF` 的 `Bundle-SymbolicName` + `Require-Bundle` 声明依赖。**每个 plugin 独立 classloader，运行期可热部署**（与 Metasfresh 单 classloader 形成对比）。

| 类别 | 代表 plugin | 证据 |

| 业务+模型核心 | `org.adempiere.base`（X_*/M_* 表模型、`DocumentEngine`、`MAcctSchema`） | `org.adempiere.base/src/org/compiere/model/` |
| 会计凭证引擎 | `org.idempiere.acct`（Doc/Fact/FactLine + 报表 + GL Distribution + 预算） | `org.idempiere.acct/META-INF/MANIFEST.MF` Export-Package :18-21 |

### 业务模型集中度（关键观察）
**所有业务模型集中在 `org.adempiere.base` 单一 plugin 的 `src/org/compiere/model/` 下**——即 OSGi 拆分的是"技术关注点"（数据库/UI/报表/调度），而非"业务领域"。业务实体（X_C_Order/X_C_Invoice/X_M_InOut 等）全在 base plugin。这是 iDempiere 与 nop-entropy"业务领域各成独立模块"哲学的**关键差异**。

### 扩展点机制
每个业务 plugin 用 OSGi DS（`@Component(service=...)`）注册扩展点。证据：`org.idempiere.acct/src/org/idempiere/acct/AcctActivator.java:56-63` 用 `mappedModelFactory.scan(...)` 注册模型，`mappedProcessFactory.scan(...)` 注册流程。

## 4. 业财一体机制（ADempiere FactAcct）

### 内核（与 Metasfresh 共享）
1. **每个业务单据表带 `Posted CHAR(1)` 字段**（`X_C_Order:98`、`X_C_Invoice:82`）。
2. **`Doc` 抽象基类** + 每种单据一个 `Doc_<Table>`，定义 `loadDocumentDetails()` + `createFacts(AcctSchema)`。
3. **`Fact`** = 单据在一个 AcctSchema 下的分录容器；**`FactLine extends X_Fact_Acct`**（`FactLine.java:62`）。
4. **凭证生成 = 借贷分录编排**。以销售发票为例（`Doc_Invoice.createFacts:427-508`）：ARI 走 `Charge CR → TaxDue CR → Revenue CR → Receivables DR`；API 走对称的 `Expense DR / TaxCredit DR / TradeDiscountReceived DR / Payables CR`（`:617-858`）。账户通过 `getAccount(ACCTTYPE_*, as)` 按 AcctSchema 解析。
5. **`Doc.post(force, repost)`**（`Doc.java:482-559`）：校验 DocStatus ∈ {Completed,Closed,Voided,Reversed}（:488-491）→ SQL 行级锁 `SET Processing='Y'`（:510-518）→ `loadDocumentDetails` → `createFacts` → 写 Fact_Acct → 置 `Posted='Y'`。再过账时先 `deleteAcct()`（:552）。
6. **四种 PostingType**（`Fact.java:78-84`）：`Actual`/`Budget`/`Commitment`/`Reservation`。

### iDempiere 特有的实现路径

**过账触发**：`DocumentEngine.postIt()`（`org.adempiere.base/.../process/DocumentEngine.java:572-586`）调静态 `postImmediate(ctx, AD_Client_ID, AD_Table_ID, Record_ID, force, trxName)`（:371/405）。文档 Complete 时由状态机自动触发（:365）。

**Doc 工厂用反射 + 命名约定**（脆弱设计）：`DefaultDocumentFactory.getDocument`（`org.idempiere.acct/.../doc/DefaultDocumentFactory.java:87-101`）按 `Doc_<去下划线的表名>` 反射 `Class.forName` + 三参构造器。表→类映射注释见 :66-84。OSGi 注册为 `@Component(service=IDocFactory.class, property="gaap=*")`（:34）。

**异步路径**：`AcctProcessor extends AdempiereServer`（`org.idempiere.acct/.../server/AcctProcessor.java:52,60`），**30 秒轮询**扫描 `Posted='N' AND Processed='Y'` 的单据（构造器参数 `30`）；按 `AD_Client_ID`/`AD_Org_ID` 设置 Env 上下文（:82-88），加载 `MAcctSchema[]`（:72）后批量过账。

> **对比**：iDempiere 用 30 秒 DB 轮询；Metasfresh 演进为事务提交后 EventBus 投递（更即时）。nop 应采用 Metasfresh 式 EventBus。

**跨 plugin 扩展点**：`FactsValidator`（`org.idempiere.acct.base.model.FactsValidator`）+ `FactsValidationEngine` 允许第三方在凭证写库前校验/改写。

### 多租户/多公司/多币种（原生支持，关键参考价值）
- **AD_Client/AD_Org 机制**：`X_AD_Client`/`X_AD_Org`/`X_AD_OrgInfo`；`AD_Client_ID` 是租户、`AD_Org_ID` 是公司/组织，所有表带这两列。`Doc.post` 校验 `p_po.getAD_Client_ID() == m_as.getAD_Client_ID()`（`Doc.java:498-504`）保证凭证与单据同租户。
- **多币种**：每张单据带 `C_Currency_ID`；`FactLine` 同时存 `AmtSourceDr/Cr`（源币种）和 `AmtAcctDr/Cr`（本位币），转换由 `MConversionRate` 完成。
- **多会计科目表**：`C_AcctSchema` 按 `AD_Client_ID` + `AD_Org_ID` 配多套，同一单据可在多套科目表下各生成一组 Fact_Acct。

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | iDempiere 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **多公司/多组织原生建模** | `AD_Client_ID`(租户) + `AD_Org_ID`(公司) 双层，所有业务表+凭证表都带 | nop 用 `nop-auth` 多租户 + 业务表加 `orgId` 字段表达公司/组织维度 |
| 2 | **多币种：源币种+本位币双字段** | `FactLine.AmtSourceDr/Cr`（源）+ `AmtAcctDr/Cr`（本位币） + `MConversionRate` | nop 凭证行内建双币种字段，避免后期重构 |
| 3 | **多会计科目表并行** | `C_AcctSchema` 按 Client+Org 配多套，单据在多套下各出一组 Fact_Acct | nop 支持管理账/税务账/集团账多套科目表并行出凭证 |
| 4 | **Doc 状态机驱动过账（Complete 自动触发）** | `DocumentEngine.java:365`（Complete→postIt） | nop 单据状态机在"完成"状态自动触发过账，无需手工调用 |
| 4a | **DocumentEngine 统一状态机横向对比** | 13 项目流程实现对比 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——iDempiere 的 DocumentEngine 是 13 项目中最成熟的内建状态机 |
| 5 | **Posted 字段 + 异步扫描兜底** | `Posted='N' AND Processed='Y'` 扫描 + `AcctProcessor` | nop 保留 `posted` 字段 + 兜底扫描 job（主路径用 EventBus） |
| 6 | **FactsValidator 凭证写库前校验/改写扩展点** | `FactsValidationEngine` + `FactsValidator` | nop 凭证生成暴露校验/改写扩展点，支持租户/场景定制借贷规则 |
| 7 | **GL Distribution（科目分摊）** | `org.idempiere.acct` 含 GL Distribution | nop 支持凭证行按规则分摊到多个维度（部门/项目/产品线） |
| 8 | **四种 PostingType 并行** | `Fact.java:78-84`（Actual/Budget/Commitment/Reservation） | nop 凭证支持实际/预算/承诺/预留四种类型并行 |

## 6. 不建议借鉴的点

- **反射命名约定的 Doc 工厂**（`Class.forName("Doc_" + table)`）脆弱，重命名即失效——应学 Metasfresh 的 `ImmutableMap<TableName, Factory>`。
- **30 秒 DB 轮询**异步过账延迟高——应学 Metasfresh 的 EventBus。
- **OSGi 66 插件**复杂度高，nop 用 Maven 多模块即可。
- **所有业务模型集中在 `org.adempiere.base` 单 plugin**——这违背了 nop"业务领域各成独立模块"的原则，不应学。
- **Swing/ZK 前端**已过时，nop 用 AMIS。
