# ERP5 调研报告

> Nexedi 出品的 ERP5 开源 ERP/CRM/KM 系统，基于 Python/Zope，使用革命性的**五类统一业务模型（UBM）**——仅用 5 个抽象类（Resource/Node/Movement/Item/Path）+ 类别分类系统，可表达任意业务交易。**对 nop-app-erp 的元模型设计有独特的架构参考价值**，是与传统关系型 ERP 设计哲学最不同的参照系。

## 1. 基本信息

| 项 | 值 |
|---|---|
| 技术栈 | Python 3 · Zope 应用服务器 · ZODB 对象数据库 · ZSQLCatalog |
| License | GPL（含 AGPL 选项） |

## 3. 五类统一业务模型（UBM）

### 3.1 核心类概览

```text
Node（节点）      → 谁/哪里：组织、人员、仓库、账户、项目、机器
Resource（资源）  → 什么：产品、服务、货币、时间、IP、碳排放额度
Movement（移动）  → 交易：订单行、发票行、库存移动、会计分录
Item（物料）      → 具体实例：序列号、批次、可跟踪单元
Path（路径）      → 容量/路线：运输路线、生产路径、渠道路由
```

### 3.2 生产级实现

| 类 | 文件 | 行号 | 继承链 |
|----|------|------|--------|
| **Node** | `product/ERP5/Document/Node.py` | :36 | `XMLObject` |
| **Resource** | `product/ERP5/Document/Resource.py` | :56 | `XMLObject` + `XMLMatrix` + `VariatedMixin` |
| **Movement** | `bootstrap/erp5_core/DocumentTemplateItem/.../document.erp5.Movement.py` | :66 | `XMLObject` + `Amount` + `CompositionMixin` |
| **Item** | `bootstrap/erp5_core/DocumentTemplateItem/.../document.erp5.Item.py` | :37 | `XMLObject` + `Amount` |
| **Path** | `bootstrap/erp5_core/DocumentTemplateItem/.../document.erp5.Path.py` | :37 | `MappedValue` + `Amount` |

### 3.3 Movement 的"通用复式记账"语义

Movement 是 ERP5 UBM 的**核心**——它概括了所有业务交易的五元组：

```text
Movement {
    source: Node,            # 来源节点（谁付出）
    destination: Node,       # 目标节点（谁接收）
    resource: Resource,      # 资源类型（交易什么）
    quantity: Number,        # 数量/金额
    variation: {...}         # 维度（颜色/尺寸/时间）
}
```

只需这一结构即可表达：
- **会计**：货币资源在账户节点间的借贷流转
- **制造**：原材料资源投入生产节点，产成品资源产出
- **CRM**：客户节点与机会资源的关联移动
- **物流**：仓库节点间的库存资源移动

### 3.4 类别驱动的关系模型（取代外键）

ERP5 **不用外键**——所有关系通过**类别分类（Category System）** 建模：

```text
portal_categories/
  source/               → 关系的来源端
    organisation/       ── 组织类型来源
    person/             ── 人员类型来源
    account/            ── 账户类型来源
  destination/          → 关系的目的端
    organisation/
    location/
  resource/             → 参与的资源类型
    product/
    service/
    currency/
  function/             → 业务功能（销售/生产/会计）
  variation/            → 变体维度（颜色/尺寸/材料）
```

**示例**：一张销售订单行：
```python
order_line.categories = [
    "source/organisation/my_company",      # 来源是我公司
    "destination/organisation/customer_x",  # 目标是客户
    "resource/product/sku_001",             # 资源是 SKU-001
    "function/sales",                       # 业务功能是销售
    "variation/colour/blue",                # 变体：蓝色
]
```

### 3.5 获取（Acquisition）机制

属性获取引擎（`Base.py:917-1132`）沿关系图自动获取关联对象的属性值：

```python
order_line.getSourceTitle()      # 从源组织获取公司名称
order_line.getSourceAddress()    # 通过获取链从源组织获取地址
```

这实现了类似 ORM 延迟加载 + 关系遍历的能力，但完全在动态类别系统上实现。

## 5. 工作流引擎

### 5.1 两种工作流模型

| 类型 | 文件 | 特点 |
|------|------|------|
| **状态工作流**（DCWorkflow） | `Core/Workflow.py` | 传统状态机：states + transitions + guards + scripts |
| **交互工作流**（Interaction） | `InteractionWorkflow.py` | **无状态、事件驱动**（信号/槽模式） |

### 5.2 状态工作流

- States：命名状态（`draft` / `open` / `closed` / `cancelled`）
- Transitions：状态转换，带**触发类型**（自动/用户/方法）、前/后脚本、守卫权限
- Variables：跟踪状态的工作流变量（`validation_state`, `action`, `actor`）

### 5.3 交互工作流（独特）

交互工作流是**无状态的事件处理器**——类似于 Qt 的信号/槽：

```text
方法调用 SaleOrder.confirm()
  → WorkflowMethod 包装器拦截
  → 触发 "confirm" 转换
  → 执行检查脚本
  → 更改进度状态（draft → confirmed）
  → 触发后续交互工作流：
      • SimulationMovement_generateOrder()  — 创建模拟移动
      • AccountingTransaction_generate()    — 创建会计交易
      • StockCache_invalidate()             — 刷新库存缓存
```

### 5.4 业务模板示例（分配工作流）

**文件**：`bt5/erp5_base/WorkflowTemplateItem/portal_workflow/assignment_workflow/`

- 状态：`draft`, `open`, `closed`, `cancelled`, `deleted`, `updated`
- 转换：`open`, `close`, `cancel`, `delete`, `update`
- 脚本：`checkActiveAssignmentConsistency`, `updateUserSecurityGroup`, `setStopDate`

## 6. 动态 Portal Type 系统

Portal Types **在运行时从 ZODB 中的声明性定义动态构建**，而非静态 Java/Python 类：

1. 每个 `portal_type` 定义指定：Property Sheets、Workflows、Base Categories
2. `AccessorHolderType` 在类加载时**动态创建 getter/setter**
3. `PropertySheet`（`Core/PropertySheet.py:39-63`）从 ZODB 查找属性定义并生成访问器
4. 最终化的 `portal_types.resetDynamicDocuments()` 同步 Python 类

### 属性表（Property Sheets）

属性表是**可组合的字段集合**，通过声明性特征选择应用于 Portal Types：

```python
class Item(XMLObject, Amount):
    property_sheets = (
        PropertySheet.Base,
        PropertySheet.CategoryCore,    # 类别关系
        PropertySheet.DublinCore,      # 元数据（标题、描述）
        PropertySheet.Price,           # 定价字段
        PropertySheet.Item,            # Item 特定字段
        PropertySheet.Amount,          # 数量/单位
        PropertySheet.Reference,       # 引用号
    )
```

## 7. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | ERP5 证据 | 对 nop 的落地建议 |
|---|--------|-----------|-------------------|
| 1 | **统一业务模型（UBM）** | 5 类表达一切交易 | nop 的 orm.xml 设计可借鉴"资源流动"视角来建模跨域交易，而非仅按业务域拆表 |
| 2 | **类别驱动关系（取代外键）** | CMFCategory 分类引擎 | nop 的跨域引用考虑弱指针 + 类别标记的组合，降低模块间耦合 |
| 3 | **属性表组合（基于特征）** | PropertySheet 声明性组合 | nop 的实体字段分组可使用 XMeta 的 view 分组实现类似的逻辑组合 |
| 4 | **BT5 可版本化插件** | Business Template + dependency_list | nop 的模块化可借鉴 BT5 的依赖声明和版本号机制 |
| 5 | **交互工作流（事件驱动）** | InteractionWorkflow + WorkflowMethod 包装 | nop 的 xbiz 事件监听器（@OnChange）可以此模式扩展为声明式编排 |
| 6 | **Portal Type 动态构建** | 运行时类生成 | nop 的 delta 定制机制与 ERP5 的动态类生成有相似的灵活目标，途径不同 |
| 7 | **模拟（Simulation）引擎** | SimulationMovement 三步（plan→ordered→confirmed→stopped→delivered） | nop 的库存预测/MRP 可借鉴"计划→承诺→实际"三阶段模拟模型 |
| 8 | **获取（Acquisition）链** | 沿关系图自动获取属性 | nop 的 XMeta 可配置 `getSourceValue()` 等关联获取，减少手动查询 |

## 8. 关键差异与适用边界

| 维度 | ERP5 | nop-app-erp | 对比说明 |
|------|------|-------------|----------|
| **存储** | ZODB 对象数据库 | 关系型数据库（MySQL/PG） | 根本性差异，ERP5 的模式依赖对象数据库 |
| **关系** | 类别分类（软关系） | FK + 关联表（硬关系） | nop 更传统，适合 SQL 查询优化 |
| **元模型** | 运行时动态类生成 | 编译时代码生成 | nop 的 codegen 策略提供编译期类型安全 |
| **UI** | Zope 模板 + ERP5 Forms XML | AMIS XML JSON Schema | 都是声明式，技术栈不同 |
| **技术栈** | Python/Zope | Java/Nop Platform | 不可直接迁移，但设计思想可参考 |
