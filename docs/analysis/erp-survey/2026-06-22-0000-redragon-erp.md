# 赤龙 ERP 调研报告

> 国产开源 ERP 中**财务业务一体化闭环最完整**的项目。模型命名规范（AR/AP/Voucher），符合会计师专业认知。**对 nop-app-erp 的财务模块设计是首要参考来源。**

## 2. 模块拆分方式

Maven 多模块（`erp-parent`），按业务域切：

| 模块 | 职责 |
|---|---|
| `erp-common` | 公共（含凭证 Service `voucher/service/spring/`） |
| `erp-masterData` | 主数据（物料、科目表、组织、部门、员工） |
| `erp-inv` | 库存（入库 InvInput / 出库 InvOutput / 库存台账 InvStock） |
| `erp-finance` | 财务（AP 应付 / AR 应收 / Voucher 凭证） |
| `erp-order` | 订单（采购 PO / 销售 SO） |
| `CasServer` | CAS 单点登录 |

> **注意**：`erp-common` 和 `erp-finance` 有同名实体副本，真正运行用的是 `erp-finance` 下的版本。

## 3. 核心业务实体清单

### AP 应付（发票 + 付款）

**`ApInvoiceHead`**（表 `ap_invoice_head`，`erp-finance/.../ap/invoice/dao/model/`）：
- 业务关键字段：`apType`(应付类型)、`invoiceType`、`invoiceSourceType`(发票来源 PO/INPUT)、`invoiceSourceHeadCode`(回链订单/入库单头编码)、`payer`/`receiver`(付款方/收款方)、`amount`、`currencyCode`、`referenceNumber`(纸质发票号)、`payMode`、银行三件套 `bankCode`/`subBankCode`/`bankAccount`、`orgCode`/`staffCode`/`departmentCode`
- **三段状态机**：`status`(NEW/CONFIRM/CANCEL)、`approveStatus`(UNSUBMIT/SUBMIT/APPROVE/REJECT)、`paidStatus`(N/Y/PART)——业务状态、审批状态、收付款状态分离

**`ApInvoiceLine`**（表 `ap_invoice_line`）：`invoiceHeadCode`、`invoiceSourceLineCode`(回链订单/入库行)、`quantity`、`amount`、`taxRate`、`taxAmount`、`memo`——**税额独立成列**，行级可挂物料

**`ApPayHead`**（表 `ap_pay_head`，付款头）：`payType`、`vendorCode`、`currencyCode`、`amount`、`payDate`，同样三段状态机

**`ApPayLine`**（表 `ap_pay_line`）**核心是核销关系**：`payHeadCode` + `invoiceHeadCode` + `invoicePayAmount`(本次核销金额)——**一行付款可核销多张发票（多对多核销）**

### AR 应收（与 AP 镜像对称）

`ArInvoiceHead`/`ArInvoiceLine`、`ArReceiptHead`/`ArReceiptLine`，字段结构与 AP 完全对称（`ArReceiptHeadServiceImpl:145` 用 `autoCreateVoucher(code, ..., "RECEIPT")` 生成凭证）。

### 凭证体系（赤龙业财打通的精华）

**`FinVoucherHead`**（表 `fin_voucher_head`）：`voucherType`(凭证字 收/付/转)、`voucherNumber`(按凭证字 Redis 流水号 `FinVoucherUtil`)、`voucherDate`、`billNum`(单据数量)、`status`/`approveStatus`

**`FinVoucherLine`**（表 `fin_voucher_line`）：`subjectCode`(科目)、`drAmount`/`crAmount`(借贷方金额，**每行只填一侧，另一侧为 0**)、`memo`——**中国"借贷分录"标准写法**

**`FinVoucherBillR`**（表 `fin_voucher_bill_r`）**业财回链表**：`voucherHeadCode` + `billType`(PAY/RECEIPT/INPUT/OUTPUT/AP_INVOICE) + `billHeadCode`——**凭证与业务单据 1:N 的反向索引**

**`FinVoucherModelHead`/`FinVoucherModelLine`**（凭证模板，表 `fin_voucher_model_*`）：
- 模板头带 `businessType`(业务类型，如 INPUT/AP_INVOICE)
- **模板行的 `drAmount`/`crAmount` 字段填占位符字符串 `"AMOUNT"` 而非数字**——这是模板化凭证生成的关键技巧

### 库存与闭环

**`InvStock`**（库存台账）：`billHeadCode`/`billLineCode`/`billType`/`materialCode`/`warehouseCode`/`stockNumber`(可正可负)/`staffCode`/`departmentCode`
- **库存以"流水明细"形式存储而非汇总余额**，入库正数、出库负数，汇总即余额

**业务闭环**：
```
采购订单 PO → 入库 INPUT（写库存+生成凭证）
            → 应付发票 AP_INVOICE（核销入库行+生成凭证）
            → 付款 PAY（核销发票+生成凭证）
```
每一跳都生成独立凭证并由 `FinVoucherBillR` 回链。

### 科目表（COA）

**`MdFinanceSubject`**（`erp-masterData/.../subject/dao/model/`，表 `md_finance_subject`）：
- `subjectCode`/`subjectName`/`parentSubjectCode`(**父子树形科目**)、`segmentCode`/`segmentDesc`(**科目段值编码/描述，多段式弹性科目结构**)、`sort`、`status`
- 模仿 Oracle EBS 的"弹性会计科目"国产化简化

## 4. 业财打通机制（核心参考价值）

核心在 `FinVoucherModelHeadServiceImpl.autoCreateVoucher(billHeadCode, Double[] amount, businessType)`（`erp-common/.../voucher/service/spring/FinVoucherModelHeadServiceImpl.java:143`）：

### 审批 `APPROVE` 触发
各业务 Service 的 `updateApproveStatus()` 在 `APPROVE` 分支调用 `autoCreateVoucher`：

- `ApInvoiceHeadServiceImpl.java:135-155`（`AP_INVOICE`，传入 `[amountSum, taxAmountSum, voucherAmount]` 三档金额）
- `ApPayHeadServiceImpl.java:130-145`（`PAY`）
- `InvInputHeadServiceImpl.java:145/195`（`INPUT`，同时写 `InvStock` 库存）
- `InvOutputHeadServiceImpl.java:149/202`（`OUTPUT`，出库时 `stock.setStockNumber(-line.getOutputQuantity())` **用负数表示出库**）

### 反审批 `UNSUBMIT` 触发
通过 `FinVoucherBillRService.getVoucherHeadCodeByBillHeadCode()` 找回凭证并删除——**单据与凭证生命周期严格绑定**。

### 模板占位符填充机制
模板里的 `AMOUNT` 占位符按金额数组下标依次填入分录（应付发票可拆成"货款/税额/应付总额"三条不同科目行）。

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | 赤龙证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **凭证三件套 Voucher/VoucherLine/VoucherBillR** | `FinVoucherHead`/`Line`/`BillR` | nop 财务核心：凭证头+借贷分录行+业财回链表，构成中式复式记账完整骨架 |
| 2 | **凭证模板 + `AMOUNT` 占位符** | `FinVoucherModelLine.drAmount/crAmount = "AMOUNT"` + `autoCreateVoucher(billCode, Double[]{档1,档2,...}, bizType)` | nop 一套模板覆盖一类业务，应付发票可同时生成"库存/进项税/应付"三条分录；模板可用 nop 规则引擎/DSL 表达 |
| 3 | **凭证与业务单据双向回链表 `FinVoucherBillR`** | `(voucher_head_code, bill_type, bill_head_code)` | nop 反审批时按 billHeadCode 反查并删凭证，保证单据与凭证生命周期强一致 |
| 4 | **审批触发 autoCreateVoucher（同一事务同一节点落地库存+凭证）** | `InvInputHeadServiceImpl:145` + `autoCreateVoucher("INPUT")` | nop 出入库审批节点同时写库存与凭证，保证账实一致 |
| 5 | **AR/AP 字段设计：发票头行 + 来源单据回链** | `ApInvoiceHead.invoiceSourceType/invoiceSourceHeadCode` + `ApInvoiceLine.invoiceSourceLineCode` | nop 发票行保留 `source_bill_type/source_head_code/source_line_code`，实现"三单匹配"(PO-入库-发票) |
| 6 | **付款核销多对多关系表** | `ApPayLine(pay_head_code, invoice_head_code, invoice_pay_amount)` + `paidStatus`(N/Y/PART) | nop 一张付款单核销多张发票、一张发票被多次部分付款；头级聚合 `paidStatus` |
| 7 | **发票头四类状态分离** | `status`/`approveStatus`/`paidStatus` 三轴独立 | nop 单据状态设计：业务状态、审批状态、财务状态分离，避免状态混乱 |
| 7a | **三轴状态 + 声明式转换规则** | 13 项目流程实现对比 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——赤龙的三轴设计好但转换规则散布在 if-else 中，应升级为声明式 |
| 8 | **行级税额独立列（价税分离）** | `ApInvoiceLine.taxRate`/`taxAmount` 与 `amount` 分离 | nop 发票行内建税字段，便于增值税核算和价税分离凭证 |
| 9 | **凭证号按凭证字 Redis 流水** | `FinVoucherUtil`（收/付/转三类独立连续编号） | nop 凭证号按凭证字分类连续编号，符合中国会计"字第号"规范 |
| 10 | **借贷分录行只填一侧** | `FinVoucherLine.drAmount` 或 `crAmount`，另一侧为 0 | nop 的 `voucher_line` 照此中式复式记账写法 |
| 11 | **父子树形科目 + 段值编码（弹性 COA）** | `MdFinanceSubject.parentSubjectCode` + `segmentCode` | nop 科目表支持多级科目（一级编码+明细）和弹性段，便于汇总查询和报表 |
| 12 | **库存流水台账（正负数明细，汇总=余额）** | `InvStock.stockNumber`（入库正/出库负） | nop 库存可用流水明细为主、余额表为物化缓存（与管伊佳"重算余额"对比，流水审计性更强） |

## 6. 不建议借鉴的点

- **CAS 单点登录**（`CasServer`）是赤龙特有集成，nop 有 `nop-auth` 自带认证，不需复刻。
- **`erp-common` 与 `erp-finance` 同名实体副本**是历史遗留，nop 应避免模块间实体重复定义。
- **未发现独立"预算"模块**（闭环仅覆盖 发票↔收付款↔凭证↔库存，无预算控制）——若 nop 需要预算控制需另寻参考。
