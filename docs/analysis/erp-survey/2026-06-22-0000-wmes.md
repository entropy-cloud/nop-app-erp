# WMES / ZH-MES 调研报告

> 纯车间制造执行系统（MES）。**与 ERP 是互补关系，不是替代。** 对 nop-app-erp 的边界划定（"应做/不应做"）有指导价值；若 nop-app-erp 覆盖生产，离散制造字段可参考。

## 1. 基本信息

| 项 | 值 |

## 2. 模块拆分方式（经典 4 层 + 双 Web）

`WaterCloud.sln` 物理拆 7 个工程：

| 工程 | 职责 |
|---|---|
| `WaterCloud.Code` | 基础设施/公共代码 |
| `WaterCloud.CodeGenerator` | 代码生成器 |
| `WaterCloud.Data` | 数据访问（SqlSugar） |
| `WaterCloud.Domain` | 实体 + ViewModel（按业务域子目录） |
| `WaterCloud.Service` | 业务服务 |
| `WaterCloud.Web` | MVC 主站 |
| `WaterCloud.WebApi` | Web API |

实体位于 `WaterCloud.Domain/Entity/<Domain>/`，**每个 .cs 一个表**，SqlSugar 用 `[SugarTable("mes_plan")]` + `[SugarColumn(...)]` 标注映射（CodeFirst）。

## 3. 离散制造关键字段设计

### 生产计划 `PlanEntity.cs`（`mes_plan`）
- `PlanNum`/`ParentPlanNum`（**支持计划层级**）
- `OrderNum`（关联订单）
- `MaterialId`/`Code`/`Name`
- `Num`/`ActualNum`/`ProductionNum`/`BadNum`/`DefectiveNum`/`GoodNum`/`InSum`（计划/实际/生产/不良/次品/良品/入库数聚合）
- **聚合字段均为 `[SugarColumn(IsIgnore=true)]` 虚拟字段，靠查询拼接**（非冗余存储）
- `Completiondate`、`DeleteMark`/`EnabledMark`（软删停用）

### 排产计划 `PsplanEntity.cs`（`mes_psplan`）
- `PsNum` + 反查 `PlanNum`/`PlanId` + `ProcessRouteId`/`Name`（**绑定工艺路线**）+ 物料冗余字段

### 报检单 `QualityorderEntity.cs`（`mes_qualityorder`）
- `QNum` + 链 `PsNum`/`PsId`/`PlanNum`/`PlanId`（**一张质检单贯通排产→计划→物料**）

### 工艺路线 `ProcessrouteheadEntity.cs` + `ProcessroutebodyEntity.cs`
- 头表 + 工序明细的经典主子表

## 4. 与 ERP 的边界（关键参考价值）

WMES **车间执行层**覆盖了 ERP 通常不做的：
- 设备档案（`basic_equipment`）、产线/工位（`Line`/`Linebody`）
- 工艺路线与工序（`Processroutehead`/`body`）
- 停机原因（`Downtimereasons`）、报检/返修（`Qualityorder`/`Repairorder`）
- 库位锁（`Locksl`）、排产计划（`Psplan`）

采购/销售（`Procurementorder`/`OrderEntity`）虽在但只是薄壳——**MES 不替代 ERP 的进销存与财务**。

## 5. 看板/报表实现
- 没有专门的"看板"实体；逻辑分散在 Controller（`WaterCloud.Web/Areas/WMSManagement/Controllers/ReportController.cs` 是 WMS 报表入口；`HomeController.Index` 提供仪表盘聚合）。
- `ReportEntity.cs` 是**纯 POCO（无 `[SugarTable]`）**，专用于报表投影——**"报表视图对象与实体分离"的做法**。
- **SqlSugar CodeFirst**（实体标注即建表），无独立 migration 脚本。

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | WMES 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **MES 与 ERP 边界划定** | `BasicdataManagement`/`MESManage`（设备/产线/工艺/报检/排产） | nop-app-erp 聚焦进销存+财务，**应显式排除** MES 域（设备/产线/报检/看板），避免过度扩张；MES 作为独立系统集成 |
| 2 | **主子表分离的工艺路线** | `ProcessrouteheadEntity` + `ProcessroutebodyEntity` | 若 nop 覆盖生产，工艺路线用 orm 主子表关系复刻，工序明细做子实体 |
| 3 | **生产计划进度聚合用虚拟字段** | `PlanEntity.cs` 的 `ActualNum`/`BadNum`/`GoodNum`/`InSum` 标 `IsIgnore=true` | nop 在 orm 实体上用 `not-persist`/computed 字段或独立视图表达聚合，避免冗余存储 |
| 4 | **计划层级（父子计划）** | `PlanEntity.ParentPlanNum` | nop 生产计划可用自关联 `parentSid` 支持分解 |
| 5 | **报表对象与实体分离** | `WMSManagement/RePortEntity.cs`（无 `[SugarTable]` 纯 POCO） | nop 报表放单独 view 实体或 DTO，不进主 orm entities，保持实体纯净 |
| 6 | **单据链贯通（质检→排产→计划→物料）** | `QualityorderEntity` 链 `PsNum`/`PlanNum`/`MaterialId` | nop 单据间用业务编号链（非外键强引用）支持跨域追溯 |
| 6a | **完整可视化流程引擎** | `FlowRuntime.cs` + `FlowinstanceService.cs` + `sys_flowscheme` JSON 方案 | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——WMES 是 13 项目中唯一有完整可视化流程设计器的，支持条件分支/会签/第三方回调，但与业务状态机解耦 |

## 7. 不建议借鉴的点

- **.NET 6 + SqlSugar + LayUI 技术栈**与 nop（Java/Quarkus/AMIS）不兼容。
- **4 层 + 双 Web** 工程拆分较粗，nop 用 Maven 多模块 + VFS 更细粒度。
- **SqlSugar CodeFirst 实体标注即建表**与 nop 的"模型驱动生成"理念冲突（nop 是 orm.xml → 代码生成，非 CodeFirst）。
- **看板无独立实体**（散在 Controller）是反模式，nop 看板应基于报表视图实体。
