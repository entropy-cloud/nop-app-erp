# Wimoor ERP 调研报告

> 国内首款百分百开源、支持商用的亚马逊跨境 ERP 系统，基于 SpringBoot + Spring Cloud 微服务架构。**对 nop-app-erp 的电商集成模式、多租户 SaaS 架构、利润计算引擎有独特参考价值**——这些能力在已有的调研项目中（传统制造业 ERP）未被覆盖。

## 1. 基本信息

| 项 | 值 |
|---|---|
| 技术栈 | Spring Boot 2.6.13 + Spring Cloud 2021.0.5 + Alibaba Cloud（Nacos/Seata）+ MyBatis-Plus + MySQL + Redis |
| License | 开源商用 |
| 定位 | 亚马逊跨境一站式 ERP 管理系统 |
| 架构 | Spring Cloud 微服务（Gateway + Admin + Amazon + Amazon-Adv + ERP + Finance + Modules + Common） |

## 2. Amazon API 集成架构

### 2.1 SP-API 认证与客户端构建

Wimoor 围绕 Amazon SP-API（Selling Partner API）构建完整集成层，使用 **Swagger Codegen 生成的 Java SDK**（`amazon-sp-api` 模块）。

**认证引擎**（`AmazonAuthority.java`: `wimoor-amazon/amazon-boot/.../auth/pojo/entity/AmazonAuthority.java`）：
- 存储卖家授权信息：`sellerId`、`refreshToken`、`clientId`、`clientSecret`
- 实现 `RateLimitConfiguration` 接口，跟踪每个 API 的速率限制
- 支持 SOCKS5 代理配置

**API 客户端工厂**（`ApiBuildService.java`: `wimoor-amazon/amazon-boot/.../auth/service/impl/ApiBuildService.java`）：
```text
ApiBuildService.getOrdersV0Api(auth)   → OrdersV0Api
ApiBuildService.getReportsApi(auth)     → ReportsApi
ApiBuildService.getFeedsApi(auth)       → FeedsApi
ApiBuildService.getFbaInventoryApi(auth)→ FbaInventoryApi
ApiBuildService.getFinancesApi(auth)    → FinancesApi
... 20+ API 客户端
```

**认证流程**：LWA (Login with Amazon) Token Exchange → Restricted Data Token（PII 访问）→ Region/Endpoint Mapping

## 4. SaaS/多租户架构

### 4.1 租户隔离模型

**共享数据库 + 逻辑隔离**（shopid 作为租户标识）：
- 大多数业务表以 `shopid` 或 `groupid` 作为租户区分字段
- 无独立数据库/独立 Schema 方案

### 4.2 认证与鉴权

- **Gateway 层鉴权**（`SecurityGlobalFilter.java`）：Token（jsessionid）存储在 Redis
- **URL 级 RBAC**：`system:perm_roles_rule:url:[METHOD:path] → [role1, role2]` 映射
- **用户类型**：admin / manager / member（渐进式权限）
- **数据域限制**：`UserLimitDataType` 字段级数据访问限制

### 4.3 SaaS 计费

| 实体 | 用途 |
|------|------|
| `SysTariffPackages` | 订阅套餐定义 |
| `SysTariffPackagesAppend` | 增值功能 |
| `ManagerLimit` | 租户资源限制 |
| `SysCustomerOrder` / `SysCustomerInvoice` | 客户订阅订单/发票 |

### 5.3 期末结账（策略模式）

**核心设计亮点**——策略模式实现结账处理：

| 组件 | 路径 |
|------|------|
| `IFinClosingTemplateStrategy` | `.../closing/service/strategy/IFinClosingTemplateStrategy.java` |
| `FinClosingTemplateStrategyFactory` | `.../closing/service/strategy/FinClosingTemplateStrategyFactory.java` |
| `FinClosingTemplate` / `FinClosingTemplateItem` | 结账模板定义 |
| `FinClosingTemplateVouchers` | 结账生成凭证 |

结账策略按 `ftype` 分发，覆盖：成本摊销、汇兑损益、损益结转等。

### 5.4 对 nop 的借鉴意义

- **利润计算策略模式**——nop 的 inventory 成本核算可使用类似模式，按计价方法（移动平均/先进先出等）分发策略
- **期末结账策略模式**——nop finance 域的期末处理可直接参照
- **科目层级编码**——`subjectCode` 层级编码（如 1001.01）方便树形展开和汇总

## 6. 值得注意的设计模式

### 6.1 优秀模式

| 模式 | 路径 | 评价 |
|------|------|------|
| **利润计算策略模式** | `wimoor-amazon/amazon-boot/.../profit/service/impl/` | 10 个站点实现 + 子计算器注入，扩展性好 |
| **结账策略模式** | `wimoor-modules/wimoor-finance/.../closing/service/strategy/` | 模板方法 + 策略，领域清晰 |
| **ApiBuildService 工厂** | `wimoor-amazon/.../auth/service/impl/ApiBuildService.java` | 20+ API 客户端统一构建 |

### 6.2 反模式警示

| 反模式 | 位置 | 说明 |
|--------|------|------|
| **Entity 实现基础设施接口** | `AmazonAuthority implements RateLimitConfiguration` | 领域实体耦合速率限制逻辑 |
| **SpringUtil.getBean() 注入** | `AmazonAuthority` | 服务定位器反模式 |
| **Controller 层密集逻辑** | `FinAccountingSubjectsController` | 树构建逻辑不应在 Controller |
| **空 catch 块** | 多处 `me.printStackTrace()` | 错误处理不规范 |
| **混合 JSON 库** | Fastjson/Fastjson2/Jackson/Gson 混用 | 技术债 |

## 7. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Wimoor 证据 | 对 nop 的落地建议 |
|---|--------|-----------|-------------------|
