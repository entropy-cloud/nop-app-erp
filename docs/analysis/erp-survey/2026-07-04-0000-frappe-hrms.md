## 3. Employee Advance 数据模型（`employee_advance.json`）

| 字段 | 类型 | 说明 |
|---|---|---|
| `employee` | Link(Employee) | 借款员工 |
| `posting_date` | Date | 借款日期 |
| `purpose` | Small Text | 借款用途 |
| `advance_amount` | Currency | 申请借款金额 |
| `paid_amount` | Currency | **已支付借款金额**（财务已打款） |
| `claimed_amount` | Currency | **已报销抵扣金额**（经 Expense Claim 核销） |
| `return_amount` | Currency | **已现金还款金额**（未消费部分退回） |
| `pending_amount` | Currency | 员工当前未付借款累计（跨单据） |
| `status` | Select | 状态机（见 §4） |
| `advance_account` | Link(Account) | **借款挂账科目（必须 Receivable 类型）** |
| `exchange_rate` | Float | 多币种汇率 |
| `mode_of_payment` | Link | 付款方式（Cash/Bank） |

### 三金额闭环（核心设计）

```
paid_amount（财务已付借款） = claimed_amount（报销抵扣） + return_amount（现金还款）
```

- 员工借款 1000，报销 700，还回 300 → paid=1000, claimed=700, return=300（闭环）
- 三个金额分别由不同动作驱动：`paid_amount` 由 `make_bank_entry`（付款 JE）+ `Advance Payment Ledger Entry` 回写；`claimed_amount` 由 Expense Claim 报销单核销回写（`update_claimed_amount:186-206`）；`return_amount` 由 `make_return_entry`（还款 JE）驱动。

> 对本项目：`expense-claim.md` 当前只有 EXPENSE_CLAIM/EMPLOYEE_ADVANCE/EMPLOYEE_ADVANCE_SETTLE 三类过账 + 报销抵扣借款净额核销，**缺 return_amount 这一维**（现金还款）。hrms 的三金额闭环是更完整的模型。

## 4. 状态机（`employee_advance.py:81-117`）

| 状态 | 触发条件 |
|---|---|
| Draft | docstatus=0（未提交） |
| Unpaid | 已提交，paid_amount=0 |
| Paid | paid_amount == advance_amount |
| Claimed | claimed_amount == paid_amount（全部报销抵扣） |
| Returned | return_amount == paid_amount（全部现金退回） |
| Partly Claimed and Returned | claimed + return == paid 且两者都 > 0 |
| Cancelled | docstatus=2 |

迁移由 `set_status` 派生计算（非显式迁移动作），基于三金额比例关系自动推导。`set_pending_amount:208-219` 跨单据聚合员工当前未付借款累计（同一员工多张借款单）。

> 设计要点：状态是**派生投影**而非独立存储状态机——金额变化即状态变化。与本项目 `ErpHrSalary.paymentStatus`（派生投影）思路一致。

## 5. 借款/还款的 Journal Entry 机制（`employee_advance.py:232-394`，核心借鉴价值）

### 5.1 借款支付 `make_bank_entry`（`employee_advance.py:232-284`）

借款审批后生成 Bank Entry（JE）：

```
DR  advance_account（员工预支应收科目）   advance_amount
    CR  payment_account（现金/银行科目）        advance_amount
```

JE 行携带 `reference_type=Employee Advance` + `reference_name=<单据号>` + `party_type=Employee` + `party=<员工>` + `is_advance=Yes`。

### 5.2 现金还款 `make_return_entry`（`employee_advance.py:328-394`，本项目缺失的能力）

未消费部分现金退回，生成 Cash/Bank Entry（JE）：

```
DR  bank_cash_account（现金/银行科目）     return_amount
    CR  advance_account（员工预支应收科目）     return_amount
```

JE 行同样携带 `reference_type/reference_name` + `party` + `is_advance=Yes`，`voucher_type` 按 `mode_of_payment`（Cash Entry / Bank Entry）。

### 5.3 薪资扣回 `create_return_through_additional_salary`（`employee_advance.py:309-325`）

第三条还款路径——从下月薪资扣回：生成 `Additional Salary`（金额 = paid − claimed），由薪酬发放时抵扣。

### 5.4 对本项目的关键启示

> **hrms 用 Journal Entry（凭证）直接承载借款支付与现金还款，不依赖独立的 Payment 实体。**

本项目 `expense-claim.md:19` 的 Deferred 原因是"无通用 finance 付款实体（付款为域级 ErpPurPayment/ErpSalReceipt）"。hrms 的解法是：
- 不建独立 Payment 实体，而是**用通用 JE（Journal Entry）承载**任何收付款。
- JE 行的 `reference_type/reference_name` 弱指针关联源单据（借款单），`is_advance=Yes` 标记预付性质。
- 对应到 nop：可直接生成 `ErpFinVoucher`（凭证头+行）承载员工借款的支付与还款，挂 `EMPLOYEE_ADVANCE`/`EMPLOYEE_ADVANCE_SETTLE` businessType + 业财回链 `ErpFinVoucherBillR`，**无需新建通用 Payment 实体**——本项目已有的业财过账引擎完全能覆盖。

## 6. advance_account 科目约束（`employee_advance.py:69-79`）

借款挂账科目**必须是 Receivable 类型**（员工预支 = 其他应收款）。`validate_advance_account_type` 强校验：非 Receivable 抛错。

> 对本项目：对应 `expense-claim.md` 的 EMPLOYEE_ADVANCE 过账"借：其他应收款-员工预支"——hrms 把科目类型约束做成了 doctype 级校验，本项目可经 FactsValidator 实现（参考 `posting.md` §凭证写库前校验扩展点）。

## 7. Advance Payment Ledger Entry（ERPNext 预付台账）

`set_total_advance_paid:119-184` 从 `Advance Payment Ledger Entry`（APLE）聚合 paid/return 金额，按 `against_voucher_type/voucher_no` 反查每笔支付/还款。账户类型 Receivable 时 amount>0 为支付、amount<0 为还款；Payable 反之。多币种时按 exchange_rate 折算。

> 对本项目：APLE 相当于本项目的 `ErpFinArApItem` 辅助账项——员工的借款挂账也是一类辅助账项。本项目 `expense-claim.md` 的"员工应付/预支双方向辅助账"已对齐此模型。

## 8. 取消联动（`employee_advance.py:50-53, 221-229`）

`on_cancel`：`ignore_linked_doctypes = ("GL Entry", "Payment Ledger Entry", "Advance Payment Ledger Entry")` 防止取消时连锁删除已过账的 GL/Payment 行；`check_linked_payment_entry` 按 HR Settings `unlink_payment_on_cancellation_of_employee_advance` 决定是否解除关联付款。

> 对本项目：对应冲销反写闭环（`posting.md` §冲销机制方向一）——取消借款单时经业财回链反查已过账凭证红冲，而非物理删除 GL 行。

## 9. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **三金额闭环模型（paid/claimed/return）** | `employee_advance.json` + `set_status:81-117` | 员工借款实体补 `return_amount` 字段（当前只有借款+报销抵扣两维），支持"未消费现金退回"场景 |
| 2 | **现金还款用 JE/凭证承载（不建 Payment 实体）** | `make_return_entry:328-394` | 解除 `expense-claim.md` Deferred——用 `ErpFinVoucher`（EMPLOYEE_ADVANCE_SETTLE businessType）承载现金还款，无需通用 finance 付款实体。引擎已有业财回链支持 |
| 3 | **薪资扣回作为第三还款路径** | `create_return_through_additional_salary:309-325` | 本项目 HR 薪酬已落地（plan 0831-2），可加 EMPLOYEE_ADVANCE_SETTLE 的"薪资扣回"子路径 |
| 4 | **advance_account Receivable 类型强校验** | `validate_advance_account_type:69-79` | 经 FactsValidator 校验员工预支科目必须是应收类（辅助账 DIRECTION_RECEIVABLE），对齐 `ar-ap-reconciliation.md` 辅助账方向 |
| 5 | **状态派生自金额比例（非独立状态机）** | `set_status:81-117` | 借款状态作为 paid/claimed/return 比例的派生投影（参本项目 `ErpHrSalary.paymentStatus` 派生投影先例） |
| 6 | **跨单据员工未付借款聚合（pending_amount）** | `set_pending_amount:208-219` | 借款审批前校验员工当前累计未付借款，防超额借款 |

## 10. 不建议借鉴的点

- **Frappe Framework 强耦合**：doctype/DocType/`frappe.db`/`frappe.qb` 是 Frappe 私有框架，本项目用 Nop ORM + BizModel，DI 模式与字段定义不可照搬。
- **GL Entry / Payment Ledger Entry 双账本**：ERPNext 维护 GL + Payment Ledger 双账本（见 erpnext.md §8.3）；本项目用单 GL + 辅助账项（ErpFinArApItem），不必引入双账本。
- **状态用字符串 Select 而非 int code**：本项目字典已是 string code（plan 2108-1 重构后），但状态机定义集中度不如本项目 `state-machine.md` 的 10 维度审查。
- **JE 行用 `is_advance=Yes` 标记预付**：本项目用 businessType（EMPLOYEE_ADVANCE）+ 业财回链承载，比布尔标志更语义化。
