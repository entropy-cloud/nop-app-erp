## 4. Odoo 中国本地化的标准做法（OCA 社区模式）

从 README + 模板可推 OCA 的模式：

1. **本地化拆成独立 addons 仓库**（金税/增值税发票/银行对账/中国报表各自独立），**不内建进 Odoo 核心**
2. 按 Odoo 版本分独立分支（18.0 / 17.0 / 16.0...）
3. **每模块自带 `__manifest__.py` 声明 `license`/`depends`/`data`**，可独立安装/卸载
4. OCA 只提供 CI（pre-commit + pylint + ruff + 翻译 widget + codecov + runboat）和规范，不提供业务代码模板

### OCA 覆盖的中国本地化能力（从 README 主题推断）
- 金税接口（Golden Tax）
- 增值税发票（VAT Invoice）
- 银行对账（Bank Reconciliation）
- 中国特色财务报表
- 全套翻译（zh_CN）

> **注意**：以上能力清单是从 README 标题 + OCA 模式**推断**的，非该仓库直接证据。若需精确字段，需另取 `OCA/l10n-china-*` 子仓库（不在本次给的路径内）。

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | OCA 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **中国本地化做成独立可拔模块** | 本仓是 OCA 模板，本地化模块分散在子仓 + 每模块独立 license | nop 设独立 `app-erp-l10n-cn` orm 片段/工程，金税/增值税发票/银行对账/中国报表各自可独立装卸，**不内建在核心 ERP 实体里**——这样金税接口、发票字段、银行对账规则可单独演进，不污染通用进销存模型 |
| 2 | **按版本/分支隔离本地化** | Odoo 按 18.0 分支 | nop 按 `platformVersion` 分独立 orm 版本，本地化随平台版本演进 |
| 3 | **CI 治理优先（pre-commit/pylint/ruff/codecov/runboat 全套）** | `.github/workflows/` + `.pre-commit-config.yaml` | nop-app-erp 已有 AGE 文档结构，可补等价的 lint/规范 CI |
| 4 | **本地化能力最小集合** | README 主题 | 金税接口、增值税发票、银行对账、中国特色报表——这四类是 nop-app-erp 中国本地化的最小集合 |

## 6. 不建议借鉴的点

- **Odoo 的 `__manifest__.py` + OCA 多仓模式**对单一团队过重；nop 用 Maven 模块 + Delta 即可表达本地化。
- **本仓无业务代码**，无法作为字段级参考；需另取 OCA 子仓。
