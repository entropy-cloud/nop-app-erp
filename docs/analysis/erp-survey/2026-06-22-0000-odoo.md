## 2. 模块拆分方式

Odoo 的"addon"是独立的可装卸模块，每个 addon 自带 `__manifest__.py`（声明 `depends`/`data`/`license`）、`models/`、`views/`、`security/`、`data/`、`report/`。模块之间靠 ORM 继承（`_inherit`/`_inherits`）和字段反向关联（`One2many`/`Many2many`）组合。

**模块拆分粒度极细**——按功能维度而非业务域切：

| 模块家族 | 数量 | 代表 addon |
|---|---|---|
| `mrp` | 11 | `mrp` / `mrp_account`（制造过账）/ `mrp_landed_costs` / `mrp_product_expiry` / `mrp_repair` / `mrp_subcontracting`（+ `_account`/`_dropshipping`/`_landed_costs`/`_purchase`/`_repair`） |
| `sale` | 29 | `sale` / `sale_stock` / `sale_purchase` / `sale_mrp` / `sale_management` / `sale_margin` / `sale_loyalty` / `sale_project` / `sale_timesheet` / `sale_expense` / `sale_edi_ubl` |
| `purchase` | 9 | `purchase` / `purchase_stock` / `purchase_mrp` / `purchase_requisition`（+`_sale`/`_stock`）/ `purchase_edi_ubl_bis3` / `purchase_product_matrix` / `purchase_repair` |
| `stock` | 9 | `stock` / `stock_account`（存货会计）/ `stock_landed_costs` / `stock_dropshipping` / `stock_picking_batch` / `stock_delivery` / `stock_maintenance` / `stock_sms` / `stock_fleet` |

> **关键设计**：`stock_account` 把"存货会计"从 `stock` 中独立出来；`sale_stock_margin` 把"销售+库存+毛利"三维交叉功能做成独立 addon。这是 Odoo"功能维度极致细分"哲学的典型。

## 3. 核心业务实体清单

证据均在 `odoo/addons/<module>/models/` 下。

### 库存（`stock`）
- **`stock.move`**（`stock/models/stock_move.py:19`）— 计划层"一次移动意图"，状态 7 态 `draft→waiting→confirmed→partially_available→assigned→done→cancel`；区分需求 `product_uom_qty` 与实际 `quantity`。
- **`stock.move.line`**（`:16`）— 执行层"实际搬动的批次行"，记 lot/包材/实际 qty。
- **`stock.picking`** + **`stock.picking.type`**（`stock_picking.py:539, :21`）— 操作层"一次交接/调拨单"；`picking.type` 定义作业类型（incoming/outgoing/internal/mrp_operation）。
- **`stock.quant`**（`stock_quant.py:20`）— 现状层"某库位某产品的现存量"，`(product, location, lot, package, owner)` 唯一，含 `quantity`/`reserved_quantity`/`available_quantity`。

> **三层语义**：`move = 计划`、`picking = 调度容器`、`move.line = 执行细节`、`quant = 现状快照`。

### 销售（`sale`）
- `sale.order` + `sale.order.line`（`sale_order.py:35`、`sale_order_line.py:13`）。

### 采购（`purchase`）
- `purchase.order` + `purchase.order.line`（`purchase_order.py:22`、`purchase_order_line.py:14`）。

### 制造（`mrp`）
- **`mrp.bom`**（`mrp_bom.py:14-97`）：`product_tmpl_id`/`product_id`、`type` ∈ {`normal` 制造, `phantom` 虚拟件/Kit}、`product_qty`+`product_uom_id`、`bom_line_ids`、`byproduct_ids`、`operation_ids`、`ready_to_produce`（`all_available`/`asap`）、`consumption`（`flexible`/`warning`/`strict`）、`batch_size`。
- **`mrp.bom.line`**（`:684-731`）：`product_id`、`product_qty`+uom、`bom_id`、`sequence`、`operation_id`（在哪道工序消耗）、`child_bom_id`（递归子 BOM）。
- **`mrp.bom.byproduct`**（`:856`）— 联副产品。
- **`mrp.production`**（`mrp_production.py:40`）：工单，`move_raw_ids`（领料消耗）+ `move_finished_ids`（完工入库）通过 `One2many` 关联 `stock.move`。
- **`mrp.workorder`** + **`mrp.workcenter`** + `…productivity`/`…capacity`/`…loss` — 工序/工作中心/产能/损耗。

## 4. 跨模块关联：以 `stock.move` 为枢纽

`stock.move` 字段承载所有跨模块溯源（`stock/models/stock_move.py:75-168`）：

- `location_id` / `location_dest_id` / `location_final_id`（来源/目的/最终库位）
- `move_orig_ids` / `move_dest_ids`（M2M 自关联，构成调拨链/procurement 链 `:98-105`）
- `origin_returned_move_id` / `returned_move_ids`（退货反查 `:155-158`）
- `origin`（源单据字符串 `:130`）、`group_id`（procurement 分组）、`rule_id`（`stock.rule` 触发本移动 `:143`）
- `procure_method` = `make_to_stock` / `make_to_order`（`:131`，决定是否走采购规则）

**触发链路**：
- 采购→库存：`purchase_stock/models/purchase_order.py:375 _create_picking()` → `order_line._create_stock_moves(picking)`。
- 销售→库存：`sale_stock/models/sale_order.py:31` `picking_ids = One2many('stock.picking','sale_id')`。
- 制造→库存：`mrp.production` 的 `move_raw_ids`/`move_finished_ids` 两路 `stock.move`（`mrp_production.py:203,208`）。

## 5. 业财一体机制（两条独立链路）

Odoo 区分**业务→发票（显式开票）**与**业务→存货凭证（自动）**两条链路：

### (a) 销售/采购 → 应收应付凭证（显式）
通过"开票"动作触发：`sale/models/sale_order.py:1550 _create_invoices()` → `_prepare_invoice()`（`:1411`）生成 `account.move`；`sale.order.invoice_ids = Many2many('account.move')`（`:239`）。purchase 模块对偶实现。

### (b) 库存移动 → 存货估值凭证（自动）
在 `stock_account/models/stock_move.py:168 _action_done()` 中调用 `moves._create_account_move()`（`:184`）：

```python
account_move = self.env['account.move'].sudo().create({
    'ref': joined_refs,
    'journal_id': self.company_id.account_stock_journal_id.id,
    'line_ids': [Command.create(aml) for aml in aml_vals_list], ...})
account_move._post()
```

借贷方来自 `_get_account_move_line_vals()`（`:220`）：`location.valuation_account_id` ↔ `product._get_product_accounts()['stock_valuation']`。仅在 `valuation == 'real_time'` 时过账（`:657-659`）。

> **关键观察**：Odoo 的存货估值过账是**自动触发**的，但应收应付凭证需要**显式开票**才产生——这是一个可配置的产品决策。

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Odoo 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **库存移动单作为统一意图枢纽 + 自追溯链** | `stock_move.py:98-158`（`move_orig_ids`/`move_dest_ids`/`origin_returned_move_id`） | nop 的库存移动单应内建"上下游移动关联"，支持采购到货→生产领料→销售出库全链路追溯 |
| 2 | **三层库存模型** move(计划)/picking(调度)/move.line(执行)/quant(现状) | `stock/models/stock_move.py`、`stock_picking.py`、`stock_move_line.py`、`stock_quant.py` | nop 用"移动单（计划/执行合一）+ 库存流水（不可变）+ 库存余额（按 item×warehouse×lot 快照）"三层 |
| 3 | **作业类型参数化 `picking.type`** | `stock_picking.py:21`（code=incoming/outgoing/internal/mrp_operation） | nop 的库存作业类型应参数化（收/发/内/制），绑定默认 location 与科目 |
| 4 | **BOM 标准字段集（可直接照搬）** | `mrp_bom.py:14-97,684-731,856` | nop BOM 表照搬：`type`(normal/phantom)、`product_qty`+uom、`is_active`+`is_default`、`consumption`(flexible/warning/strict)、子件 `operation_id`、联副产品 byproduct |
| 5 | **存货估值独立账户映射** | `stock_account/models/stock_move.py:220`（location.valuation_account ↔ product.stock_valuation） | nop 的产品/库位应配置"存货科目映射"，出入库即生成存货借贷凭证 |
| 6 | **MTS/MTO 采购策略内建于移动单** | `stock_move.py:131`（`procure_method`） | nop 移动单字段支持 `make_to_stock`/`make_to_order`，驱动补货规则 |
| 7 | **功能维度而非业务域的模块细分** | mrp 11 / sale 29 / purchase 9 / stock 9 addon | nop-app-erp 不必照搬 Odoo 极细 addon；但可借鉴"存货会计"独立成模块的思路（见模块拆分对比报告） |
| 8 | **业务单 → 发票是显式动作，与存货自动过账分离** | `sale_order.py:1550` vs `stock_move.py:184` | nop 设计时应明确：存货过账自动，应收应付过账可配置（自动或显式开票） |
| 8a | **移除 workflow engine 的教训 + base.automation 可配置触发器** | `osv/` 仅存 `__init__.py` + `expression.py`；`base_automation/models/base_automation.py:124` | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——Odoo 8.0 彻底移除 workflow engine，改用状态字段 + ORM 方法；base.automation 提供 on_state_set/on_create 等可配置触发器，是"可配置工作流"的轻量替代 |

## 7. 不建议借鉴的点

- **625 addon 的极端细分**适合超大规模社区协作，对单一团队维护成本极高；nop-app-erp 应采用 ERPNext 式粗模块（见 `2026-06-22-0000-erpnext.md`）。
- **`account.move`/`account.move.line` 命名**与"库存移动 stock.move"语义重叠易混淆；nop 应用赤龙式 `Voucher`/`VoucherLine` 命名（见 `2026-06-22-0000-redragon-erp.md`）。
