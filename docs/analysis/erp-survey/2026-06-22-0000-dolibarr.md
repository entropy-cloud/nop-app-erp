# Dolibarr 调研报告

> 极致轻量的 PHP ERP+CRM，单机一键部署。**对 nop-app-erp 的"凭证模板 + 报表对象与实体分离"两点有借鉴价值，但整体技术栈过时，不建议深度参考。**

## 2. 模块拆分方式

**目录式模块化**：每个业务域是 `htdocs/<domain>/` 目录，内含 `class/`、`card.php`（卡片页）、`list.php`、`index.php`、`admin/`。无独立模块元数据文件（不同于 Odoo 的 `__manifest__.py`）。

代表目录（`htdocs/` 下，77 个业务目录）：
- **核心业务**：`product`（商品）、`societe`（第三方：客户/供应商/联系人）、`commande`（销售订单）、`fourn`（采购）、`expedition`（发货）、`delivery`（交付）、`stock`（库存）、`bom`（BOM）、`compta`（财务）、`accountancy`（会计）、`asset`（资产）、`contrat`（合同）、`fichinter`（工单/干预）、`propal`（报价）
- **CRM/协作**：`comm`、`contact`、`adherents`（会员）、`agenda`、`categories`、`ecm`（文档管理）
- **基础设施**：`core`、`admin`、`conf`、`api`、`custom`（用户扩展位）、`themes`、`view`、`tpl`

## 3. 核心业务实体清单

每个实体一个 class 文件，字段以 `public $<field>` 声明。

| 实体 | 文件 | 关键字段 |
|---|---|---|
| **Product** | `product/class/product.class.php` | `ref`、`libelle`/`label`、`description`、`type`(TYPE_PRODUCT/SERVICE)、`price`/`price_ttc`(含税)/`price_min`/`price_min_ttc`、`price_base_type`(HT/TTC)、`price_label`、`weight`、`barcode`、`status` |
| **Société**（第三方） | `societe/class/societe.class.php` | 客户/供应商/联系人一体，含 `client`/`fournisseur` 标志、`price_level`、`tva_intra`(欧盟增值税号) |

## 4. 业财一体机制

- **凭证模板**（`bookkeepingtemplate.class.php` + `bookkeepingtemplateline.class.php`）：与赤龙 ERP 的 `FinVoucherModelHead/Line` 思路一致——预定义借贷模板，业务单据触发时填充金额。
- **BookKeeping 行级结构**：`doc_type`+`doc_ref` 反查源单（类似 ERPNext 的 `voucher_type`+`voucher_no`）。
- 具体触发点未深入（在 `accountancy/journal/` 下），但模板机制是国际通用的业财打通范式。

## 5. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Dolibarr 证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **凭证模板（BookKeepingTemplate）** | `accountancy/class/bookkeepingtemplate.class.php` + `bookkeepingtemplateline.class.php` | 国际通用的业财打通范式；nop 应学赤龙 FinVoucherModel 的更完整实现 |
| 2 | **含税/不含税双价字段内建于商品** | `product.class.php`: `price`/`price_ttc`/`price_base_type`(HT/TTC) | nop 商品主数据内建含税价/不含税价/计价基准类型 |
| 3 | **客户/供应商一体表** | `societe/class/societe.class.php`（`client`/`fournisseur` 标志） | 与管伊佳 `Supplier` 一致；nop 往来单位一体表 + 类型标志 |
| 4 | **模板发票（facture-rec）** | `compta/facture/class/facture-rec.class.php` | 周期性发票可预定义模板，nop 可支持发票模板复用 |
| 4a | **触发器链跨实体工作流** | `interface_20_modWorkflow_WorkflowManager.class.php:37-677` | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——Dolibarr 的 WorkflowManager 通过触发器链实现报价→订单→发票→发货的跨实体自动化，所有规则可通过 admin 开关配置 |

## 6. 不建议借鉴的点

- **PHP 单体架构**与 nop（Java/Quarkus）技术栈不兼容。
- **77 个业务目录**扁平堆砌，无领域边界；nop 应按领域模块化。
- **无模块元数据**（不像 Odoo 的 `__manifest__.py`），装卸不灵活。
- **轻量化取舍**（无多公司、无标准借贷总账）不适合"业财一体化"目标。
