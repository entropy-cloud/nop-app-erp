# 若依 ERP（zccbbg 版）调研报告

> 基于若依脚手架的进销存示例。技术栈最新（SpringBoot 3.2 + JDK17），**对 nop-app-erp 的"进销存三单链 + 价税分离字段 + 财务四件套"有直接可复刻的简洁参考价值。**

## 2. 模块结构与脚手架关系

复用若依脚手架：
- `ruoyi-modules/`：`ruoyi-generator`(代码生成) + `ruoyi-system`(系统管理)
- `ruoyi-common/`：公共

**新增 ERP 业务**全部塞进 `ruoyi-admin-erp/`，包 `com.ruoyi.erp.<域>`，每域统一五件套：
```
controller/domain/{bo,entity,vo}/mapper/service
```
域目录：`base`(基类 constant/domain)、`basic`(主数据)、`purchase`、`sales`、`warehouse`、`financial`。

与 `sys_*` 表的对应：`Merchant` 关联 `sys_user`，权限/字典完全复用若依。

## 3. 表清单（`script/sql/erp.sql`，38 张表，4 组前缀）

### `basic_`（主数据 7 张）
`basic_bank_account`、`basic_brand`、`basic_category`、`basic_goods`、`basic_merchant`、`basic_sku`、`basic_warehouse`

### `purchase_`（采购 6 张）/ `sales_`（销售 6 张）
各 `order`/`order_detail`/`refund`/`refund_detail`/`trade`/`trade_detail`。**"order→trade→refund"三单链**（订单→成交→退货），每单都 head+detail。

### `financial_`（财务 4 张）
`financial_merchant_balance`(商户余额)、`financial_payment_voucher`(付款)、`financial_receipt_voucher`(收款)、`financial_trans_history`(交易流水)

### `gen_`/`sys_`（若依脚手架 14 张）
`gen_table`、`sys_config`/`dept`/`dict_data`/`dict_type`/`logininfor`/`menu`/`notice`/`oper_log`/`oss`/`oss_config`/`post`/`role`/`role_dept`/`role_menu`/`user`

## 4. 关键字段设计

### `sales_order`（`erp.sql` ~第 620 行起）
- `doc_no`(单号，**UNIQUE 索引**)、`doc_date`
- `checked_status`(审核状态)、`stock_status`(库存状态)——**状态双维度**
- `goods_qty`/`processed_qty`(已处理)
- `goods_amount`/`other_expenses_amount`/`discount_amount`/`actual_amount`/`prepay_amount`
- `warehouse_id`、`bank_account_id`

### `sales_order_detail`
- `pid`(父)、`sku_id`
- `price_without_tax`/`tax_amount`/`tax_rate`/`price_with_tax`/`total_amount`
- **含税/不含税分离 + 独立税率字段**

## 5. 进销存业务模型

**订单→成交→退货闭环 + 库存联动 + 资金台账**：
- 业务流转：`*_order`(订单) → `*_trade`(成交/出入库) → `*_refund`(退货)
- 库存：`Inventory`(实时) + `InventoryHistory`(流水) + `MovementDoc`(移拨) + `CheckDoc`(盘点) + `OtherReceiptDoc`/`OtherShipmentDoc`(其他出入)
- 财务：`MerchantBalance`(商户余额台账) ↔ `PaymentVoucher`/`ReceiptVoucher`(收付款凭证) ↔ `TransHistory`(流水)，与 `BankAccount` 联动

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | 若依证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **`*_order → *_trade → *_refund` 三单闭环 + head/detail** | `erp.sql` 的 `purchase_*`/`sales_*` 各 6 表 | nop 进销存可直接复用此单据链（订单→成交/出入库→退货） |
| 2 | **含税/不含税分离 + 独立 tax_rate 字段** | `sales_order_detail`: `price_without_tax`/`tax_amount`/`tax_rate`/`price_with_tax`/`total_amount` | nop 明细表内建税字段三件套，为中国增值税打基础 |
| 3 | **状态双维度：`checked_status` + `stock_status` 分离** | `sales_order` | nop 用两个 int 字段 + `erp/doc-status`/`erp/stock-status` 字典，比单一 status 灵活 |
| 4 | **财务四件套** | `MerchantBalance`(余额台账) + `PaymentVoucher`/`ReceiptVoucher`(凭证) + `TransHistory`(流水) + `BankAccount` | nop 财务模块可直接对应（轻财务路径，类似管伊佳） |
| 5 | **业务全塞单一 admin 模块，每域五件套分层** | `ruoyi-admin-erp/com.ruoyi.erp.<域>` | nop 生成后的 service/web 层可对齐此分层，但应按业务域拆 Maven 模块而非全塞 admin |
| 6 | **单据唯一编号 UNIQUE 索引** | `sales_order` 的 `UNIQUE INDEX idx_bill_no(doc_no)` | nop 实体对单据号加 unique 约束 |

## 7. 不建议借鉴的点

- **若依脚手架**（`sys_*` 表、`ruoyi-common`）与 nop 的 `nop-auth`/`nop-sys` 重复；nop 用平台模块替代。
- **MyBatis** 与 nop ORM 不可混用。
- **38 张表全塞一个 SQL 脚本**无领域边界；nop 按领域拆 orm.xml。
- **无借贷凭证**（轻财务），做不出标准三表——业财一体应学赤龙。
- **业务全塞 `ruoyi-admin-erp`**违背模块化原则。
