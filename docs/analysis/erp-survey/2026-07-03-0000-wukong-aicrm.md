# Wukong-AICRM 调研报告

> 开源版 Salesforce + ChatGPT，悟空 CRM 团队打造的 AI Native CRM 系统。**对 nop-app-erp 的 CRM 模块设计与 AI 集成架构有重要参考价值**，特别是 AI 工具（Tool）系统、SSE 流式对话、多租户架构与第三方集成通用模式。

## 1. 基本信息

| 项 | 值 |

## 2. AI-Native 架构（核心差异化）

Wukong-AICRM 最显著的特征是**以 AI Assistant 作为统一的系统入口**，而非传统的菜单导航优先：

```
User → AI Assistant → CRM / RAG / Workflow → PostgreSQL / Redis / MinIO
```

### 2.1 Chat Application 上下文（6 种）

源码：`backend/src/main/java/com/kakarote/ai_crm/ai/app/`

| 上下文 | 注册类 | 覆盖能力 |
|--------|--------|----------|
| general | `GeneralChatApplication` | 通用对话，无业务上下文绑定 |
| crm | `CrmChatApplication` | 客户/联系人/跟进/产品 |
| project | `ProjectChatApplication` | 项目/看板泳道/任务 |
| knowledge | `KnowledgeChatApplication` | 知识库 RAG 问答 |
| address_book | `AddressBookChatApplication` | 通讯录（内部员工） |
| relation | `RelationChatApplication` | 关系图谱 |

每个实体类型都可开带上下文的对话，由 `ChatApplicationFactory` 根据 `contextType` 分发。

### 2.2 AI 工具（Tool）系统（10 个 LLM 工具）

源码：`backend/src/main/java/com/kakarote/ai_crm/ai/tools/`

| 工具类 | 功能 | 参数 |
|--------|------|------|
| `CustomerTool` | 查询客户 | 统一 `String` 参数 |
| `ContactTool` | 查询联系人 | 同上 |
| `TaskTool` | 查询任务 | 同上 |
| `ScheduleTool` | 查询日程 | 同上 |
| `KnowledgeTool` | 查询知识库 | 同上 |
| `ProductTool` | 查询产品 | 同上 |
| `ProjectTool` | 查询项目 | 同上 |
| `RelationTool` | 查询关系 | 同上 |
| `MailTool` | 查询邮件 | 同上 |
| `CandidateTool` | 查询候选人 | 同上 |

**关键设计**：
- 所有工具参数统一为 `String`，由 LLM 自行提取实体
- `AiToolPermissionAspect`（切面）：执行前校验用户权限
- `AiToolExecutionRecordingAspect`（切面）：记录工具调用历史和耗用 token

### 2.3 SSE 流式响应

`/chat/send` 使用 Server-Sent Events，前端原生 `fetch()` + `TextDecoder` 消费，无 WebSocket 依赖。每行 SSE 包含增量 token 或工具调用结果。

### 2.4 动态 ChatClient

`DynamicChatClientProvider` 避免使用 Spring Boot 自动配置的 `ChatClient`，改为按租户运行时配置创建（不同租户可绑定不同 LLM 模型/Endpoint/API Key）。

### 3.4 基础架构模块

| 模块 | 实现要点 |
|------|----------|
| **认证/授权** | JWT (UUID token) + RBAC + 5 级数据权限范围（本人/本人+下属/部门/部门+子部门/全部） |
| **多租户** | MyBatis-Plus `TenantLineInnerInterceptor` 自动追加 `tenant_id`；约 10 张豁免表 |
| **文件存储** | MinIO（默认）/ 本地文件，预签名 URL |
| **OIDC Provider** | CRM 自身作为 OIDC Provider，支持 MinIO SSO |
| **外部认证** | Google / Outlook / 微信 OAuth 登录，自动建租户 |

### 3.5 第三方集成（通用模式）

企业微信、腾讯会议、邮箱三个集成共享通用架构：

- 凭证按租户加密入库（`SecretTextCipher`）
- 增量同步用游标表（`*_sync_cursor`）
- Webhook 回调用 AES-GCM 解密
- OAuth state/ticket 存 Redis（短 TTL）
- 外部实体可绑定到 CRM `customer`

### 3.6 桌面端（Electron）

`desktop/` 子项目采用"嵌入后端"模式：
- 使用 `embedded-postgres` 和 `embedded-redis`
- WeKnora RAG 离线
- MinIO 关闭，文件存本地
- 仅 LLM API 保持远程
- 打包时内嵌 JRE（~300MB）

## 4. 多租户设计

源码：`backend/src/main/java/com/kakarote/ai_crm/config/auth/`

租户隔离是**自动且隐蔽的**，MyBatis-Plus 拦截器链：

```
Authentication → TenantLineInnerInterceptor → DataPermissionHandler
```

- `TenantLineInnerInterceptor`：自动在查询追加 `tenant_id` 条件
- `MyMetaObjectHandler`：插入时自动填充 `tenantId`
- `JwtAuthenticationTokenFilter`：从 JWT 解析租户，设置到 `TenantContextHolder`
- 租户豁免表：`manager_menu`, `crm_tenant`, `crm_ai_model_pricing` 等

## 5. AI 计费系统（双桶余额模式）

所有 AI 操作分两阶段计量，源码：`backend/src/main/java/com/kakarote/ai_crm/ai/billing/`：

1. **预检**（`ensureQuotaAvailable`）：按估算 token 预检余额
2. **实扣**（`consumeResolvedTokens`）：按模型返回的真实 usage 扣减
- 双桶余额：先扣赠送额度，再扣购买额度
- 自定义模型（`model_source=custom`）不计费
- 计费配置表：`crm_ai_billing_config`（全局）、`crm_ai_model_pricing`（按 provider/model 倍率）

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | Wukong 证据 | 对 nop 的落地建议 |
|---|--------|------------|-------------------|
| 1 | **AI Tool 系统（统一参数 + 权限与审计切面）** | `ai/tools/` 10 个工具，`AiToolPermissionAspect` + `AiToolExecutionRecordingAspect` | nop 的 CRM 模块可引入类似 AI 工具注册机制，将 CRM 实体操作暴露为 LLM 可调用的语义化工具，权限和 token 计量由切面统一处理 |
| 2 | **ChatApplication 上下文路由** | `ChatApplicationFactory` + 6 `AiChatApp` 接口实现 | nop 可将每个业务域（采购/销售/库存）注册为 ChatApplication 上下文，使 AI 助手能理解当前用户所在域 |
| 3 | **SSE 流式对话接口** | `/chat/send` SSE endpoint，前端 `fetch()` + `TextDecoder` | nop 的 AI 助手接口应优先 SSE 而非 WebSocket，简化部署和负载均衡 |
| 4 | **双桶 AI 计费（赠送+购买）** | `AiBillingService` / `BillingConsumptionCacheService` | 若 nop 引入 AI 辅助操作，参考此按 token 计量的双桶余额模式 |
| 5 | **第三方集成通用模式（加密凭证+游标同步+Webhook 解密）** | 企业微信/腾讯会议/邮箱三集成共享架构 | nop 对 EDI/电商平台/银行对账等外部集成的架构可照搬此模式 |
| 6 | **Embedded 后端（Electron 内嵌 PG/Redis/MinIO）** | `desktop/` 嵌入 `embedded-postgres` + `embedded-redis` | nop 若需桌面版，可参考此模式 |
| 7 | **动态 ChatClient（按租户运行时创建）** | `DynamicChatClientProvider` 替代自动配置 | nop 多租户场景中，每个租户可能绑定不同 LLM API Key，应运行时动态创建而非编译期配置 |
| 8 | **5 级数据行级权限** | `DataPermissionHolder` 5 范围 | nop 的 CRM 模块数据权限应同样支持"本人→下属→部门→子部门→全部"层级 |

## 7. Wukong 与 nop-app-erp 的差异提醒

Wukong AICRM **不是 ERP 系统**——它没有以下功能：
- **采购管理**（无 PO/PR/RFQ/供应商管理）
- **库存管理**（无仓库/库位/库存移动/盘点）
- **财务记账**（无凭证/GL/应收应付/成本核算）
- **生产制造**（无 BOM/工单/排产）
- **WMS/物流**（无发货/收货/拣货/装箱）

其核心价值在 **AI 驱动的客户关系管理**，而非进销存+财务一体化。nop-app-erp 可借鉴其 AI-Native 集成模式，但不能作为业务模块（采购/销售/库存/财务/制造）的设计参考。
