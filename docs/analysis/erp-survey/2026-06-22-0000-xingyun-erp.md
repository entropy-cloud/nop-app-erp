# 星云 ERP 调研报告

> 永久免费可商用的国产 ERP，**最大特色是同一套业务代码支持单体和微服务双形态部署**。对 nop-app-erp 的"单体起步→微服务演进"策略与 BPM/MQ 集成有重要参考价值。

## 1. 基本信息

| 项 | 值 |
|---|---|
| 技术栈 | SpringBoot 3.5 · Spring Cloud · JDK 21 · MyBatis-Plus · Vue · MySQL · Redis · RabbitMQ |
| License | 永久免费可商用 |
| 定位 | 中小企业进销存，单体+Cloud 双形态 |
| 源码规模 | 约 9.4 MB · 1328 文件 |

## 2. 模块拆分方式（按业务域 + 部署形态双重切分）

| 模块 | 职责（取自 pom `<name>` 与包名） |
|---|---|
| `xingyun-core` | 核心公共（实体基类、MQ 常量 `com.lframework.xingyun.core`） |
| `xingyun-basedata` | 主数据：客户/供应商/商品/会员/店铺/仓/库位/物流/支付方式/打印模板/地址。包 `com.lframework.xingyun.basedata`，子包 `bo/dto/entity/excel/impl/controller` |
| `xingyun-sc` | "仓库业务服务层"（pom `<name>` 原文）。含采购/销售/零售/库存/调拨/盘点/物流。`controller/{purchase,sale,retail,stock,logistics}` |
| `xingyun-settle` | 结算：对账(`check`)、费用(`fee`)、预收付(`pre`)、结算单(`sheet`)、收支项(`item`)，且**双向**：`CustomerSettle*` + 通用 `Settle*` |
| `xingyun-chart` | 报表/图表，独立模块，**用 MQ 监听器异步消费**业务事件 |
| `xingyun-comp` | 业务组件（公共业务工具，被多模块依赖） |
| `xingyun-api` | **单体入口**（`XingYunApiApplication`） |
| `cloud/xingyun-cloud-api` | **Cloud 入口**（`XingYunCloudApiApplication`） |
| `cloud/xingyun-cloud-gateway` | **Cloud 网关**（`GateWayApplication`） |

## 3. 单体 vs 微服务的演进策略（核心参考价值）

**同一套业务代码**（basedata/sc/settle/chart/comp），两个装配入口：

- **单体** `xingyun-api/pom.xml`：依赖 5 个业务模块 + `rabbitmq-starter`（内嵌 MQ 直连）
- **微服务** `cloud/xingyun-cloud-api/pom.xml`：依赖同样 5 个业务模块，但加 `cloud-starter`，**不含** `rabbitmq-starter`（依赖云端 MQ broker）。网关单独由 `xingyun-cloud-gateway` 承担。

单体入口 `xingyun-api/src/main/java/.../XingYunApiApplication.java`：
```java
@SpringBootApplication(scanBasePackages={"com.lframework.xingyun"})
@MapperScan("com.lframework.xingyun.**.mappers")
@EnableLock(type=REDIS)
```
即：**业务模块只写一次，单体全扫；微服务则按需拆 + 换 starter**。

## 4. BPM 与 MQ 的业务用法

### BPM 强类型集成
- 实体持 `flowInstanceId`：`xingyun-sc/.../entity/PurchaseOrder.java`（`:137` `private Long flowInstanceId;`）
- ServiceImpl 实现 `BpmBizListener`：`xingyun-sc/.../impl/purchase/PurchaseOrderServiceImpl.java` import `FlowInstanceWrapperService`、`BpmBizListener`、`FlowDefinitionExtBizType`
- 配置层：`PurchaseConfigServiceImpl.java` 通过 `FlowDefinitionWrapperService` 配置每个业务类型绑定哪个流程定义

### MQ 异步解耦
- 常量集中：`xingyun-core/.../queue/MqConstants.java`
- 业务消费点：
  - `xingyun-chart/.../listeners/mq/OrderDataToChartListener.java`（订单数据异步入报表）
  - `xingyun-sc/.../listeners/mq/ProductStockWarningStockChangeListener.java`（库存预警）
  - `xingyun-sc/.../listeners/mq/TakeStockPlanStockChangeListener.java`（盘点计划库存变更）
- **报表与库存预警解耦到异步队列**，单体靠内嵌 RabbitMQ 直连，微服务靠云端 MQ broker

## 6. 对 nop-app-erp 的可借鉴设计点

| # | 借鉴点 | 星云证据 | 对 nop 的落地建议 |
|---|---|---|---|
| 1 | **按业务域切 Maven 模块 + 同代码双入口（单体/微服务）** | `xingyun-api`(单体+rabbitmq-starter) vs `cloud/xingyun-cloud-api`(微服务+cloud-starter) | nop 生成后若需微服务化，可仿此"业务模块不变，只换装配入口 + starter"；默认单体 Quarkus app 起步 |
| 2 | **业务模块标准子包布局** `bo/dto/entity/excel/impl/controller/mappers/service/<子域>` | `xingyun-basedata`/`xingyun-sc` | nop 生成代码的包结构可对齐此分层 |
| 3 | **BPM 强类型集成（实体持 flowInstanceId + 监听器回调）** | `PurchaseOrder.java:137` + `PurchaseOrderServiceImpl implements BpmBizListener` | nop 若接工作流（nop-wf），单据实体加 `flowInstanceId` 字段 + 监听器回调，而非散落调用 |
| 3a | **BPM 只管审批不管业务流转（行业共识）** | `V1.17__bpm.sql`（flow_* 表结构）+ `tbl_purchase_config.purchase_require_bpm` | 详见 `2026-06-22-0000-workflow-vs-state-machine.md`——星云验证了"核心流转靠状态 + 审批接 BPM"的设计分层 |
| 4 | **报表/库存预警走 MQ 解耦** | `xingyun-chart/listeners/mq/OrderDataToChartListener.java` + `MqConstants.java` | nop 库存预警/报表同样事件驱动（nop-stream/nop-message），不阻塞主单据事务 |
| 5 | **单据四级拆分 head/detail/bundle/lot** | `SaleOutSheetDetailBundle`/`...Lot` 批次 | nop 销售出库可参照做"明细-组合-批次"三层子表 |
| 6 | **结算四步法（对账→费用→预收付→结算单）** | `xingyun-settle/entity/` 双向 Customer/通用 | nop 财务结算可借鉴此分层 |
| 7 | **审计字段用 ORM 自动填充** | `@TableField(fill=INSERT/INSERT_UPDATE)` | nop 用 ORM 钩子自动填充 created/updatedBy/Time，不手写 |

## 7. 不建议借鉴的点

- **MyBatis-Plus** 与 nop 的 Nop ORM 不可混用；nop 用自己的 ORM + 代码生成。
- **`com.lframework` 包名**是星云私有；nop 用 `app.erp` 包名。
- **单体+Cloud 双形态**对 nop 必要性不高（nop 默认 Quarkus 单体已够；真要微服务再拆）。
- **四级单据拆分**（bundle/lot）对非批次管理场景过度设计。
