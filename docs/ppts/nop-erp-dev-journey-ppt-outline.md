## Part 3：外部信息注入（核心洞察一，7 页 / 12 min）

### P3-1 外部注入三层模型（2 min）
- 核心问题：用户（人类）在整个过程中做了什么？没有做什么？
- 三层模型：
  - **A 类**（明确指明 nop 平台机制）→ 9 次，全部在 06-22 ~ 07-03
  - **B 类**（指明工程/设计原则方向）→ 9 次，散布全程
  - **C 类**（只让 AI 自查/对比）→ 绝大多数，尤其 07-04 之后
- 关键洞察：用户有清晰的边界意识——平台机制明确介入，业务逻辑让 AI 自行对标开源
- **视觉**：三层金字塔，每层高度代表占比（C 最厚），每层 1-2 个典型例子文字

### P3-2 A 类详细案例（一）：全局租户开关（2 min）
- 背景：AI 给每个实体手动加 `useTenant="true"`
- 用户纠正：`nop.orm.enable-tenant-by-default` 全局配置 + `TenantSupport` feature 标签
- AI 为何不知道：docs-for-ai 当时未记录此全局开关
- 修复效应：删除所有手写 useTenant，改为全局配置 + 按需 `no-tenancy` 排除
- **模式提炼**：配置层隐藏机制（application.yaml + xlib feature）是 docs 经常遗漏的知识
- **视觉**：AI 写法 vs 用户校正写法 双栏对比 + 注释标注"节省了 XX 行逐域声明"

### P3-3 A 类案例（二）：跨模块引用机制纠偏（2 min）
- 背景：AI 说"禁止跨模块引用"——以为硬规则
- 用户纠正：`notGenCode` 外部实体引用 ≠ Delta 扩展，平台自身 9 处内置用法
- AI 查不到的原因：漏看了 `entity.xdef` 注释 + `orm-gen.xlib:228` 的平台自身用法
- 修复效应：明确了 11 个跨域引用实体 + 文档补充 docs-for-ai
- **视觉**：AI 的错误推理链 vs 用户的正确证据链（红蓝对比）

### P3-4 A 类全部 9 次速览表（1.5 min）
- 9 行表格：日期 / 纠正对象 / 平台机制 / AI 为什么会错 / 修复效应

- 全部发生在 06-22 ~ 07-03，此后 A 类消失
- **视觉**：紧凑数据表 + 底部时间线标注 A 类出现位置

### P4-1 六种自驱动机制总览（1.5 min）
- 机制 1（最核心）：**计划-执行-审计闭环**——187 份计划，强制双审计

- **核心洞察**：去掉外部用户输入后，项目依靠这套内部信息循环自驱演化
- **视觉**：六边形雷达图（每角一个机制，轴=成熟度/频率）

### P4-2 机制 1 深度展开：计划-审计闭环（2.5 min）
- **标准工作流**：
```
需求/想法 → 计划草案 → 独立草案审计 → 修订 → active → 执行 → 独立结束审计 → completed
```
- **强制双审计**的意义：
  - 草案审计：拦截方向性错误（07-10 拦截 4 个 P0：码值冲突 / BUDGET 污染 / 架构前提错误 / 维度歧义）
  - 结束审计：验证交付完整性 + 自省
- **审计是独立子代理冷重放**：全新会话，不继承主代理上下文，避免确认偏差
- **187 份计划**：每份有状态（draft/active/completed）、范围、退出标准、验证检查清单
- **视觉**：流程图（步骤 = 圆角框，审计步骤高亮红色 + 盾牌图标）+ 底部 187 数字放大

## Part 5：对抗性审计亮点（4 页 / 7 min）

### P5-1 07-05 审计转折故事（2 min）
- **背景**：07-05 一天内做了 3 轮审计（1300 三维 → 1400 对抗性 → 1500 工具化）
- **07-05-1400 的"方法论飞跃"时刻**：
  - 4 路对抗性子代理（全新会话）不只看"方法是否存在"，而是检查"语义是否保真"
  - 发现 C-2：信用额度多币种语义偏差，评分仅 4/10——多币种绕过信用控制
  - 暴露"扩展域是审计盲区"——60 处时间获取蔓延、DAG 违规、表名错误
- **07-05-1500 的工具化反应**：
  - 当天立即固化为 `nop-compliance-checker.sh`（15 条 R 规则）
  - 人工发现的模式 → 可重复运行的脚本
- **关键教训**：文档对比类审计必须与源代码交叉验证；评分偏差事件必须回追溯源
- **视觉**：故事板式——四个子代理的审计结果卡片 + C-2 的评分对比 + 工具截图

### P5-2 07-07 综合审计案例（2 min）
- **配置**：4 路并行子代理 + 主代理复核
- **发现**：4 严重 + 6 高 + 9 中 + 6 低
- **严重问题举例**：
  - 表名双前缀违规（7 域）
  - docStatus 通用集不一致（3 种不同实现）
  - 7 域冗余字典
  - ErrorCode 前缀不统一（5 域 87 码）
- **整改计划 1915-1**：当天全部修复，包含具体 plan 编号
- **自省**：审计评分偏差事件回顾（07-05 误判 REQUIRES_NEW 为矛盾 → 记录为系统性短板）
- **视觉**：问题分类饼图 + 整改前后代码 diff 两栏对比

### P5-3 07-12 竞争杠杆兑现审计（2 min）
- **触发**：用户要求"承诺的功能/优势是否已完整兑现"
- **方法**：3 路并行子代理 + 主代理实时复核
- **8 个"超越点"杠杆的兑现裁决**：
  - 🟢 **完整兑现（4）**：多套账 SchemaPropagator、业财一体三件套(过账/红字/反写)、7 种成本方法、AR-AP open-item
  - 🟡 **有缺口（2）**：模型驱动 Delta 文档数据错误（实体计数 3 处冲突）、制造委外空壳（15 行 CRUD）
  - 🔴 **夸大（2）**：多公司仅结构地基（无数据隔离/权限范围/公司间交易）、域独立部署言过其实（service 层跨域强耦合）
- **纠正 2 处子代理误判**："零 Delta 使用"错（实测 338 个 view.xml 用 `x:extends`）、"finance↔inventory 循环依赖"错（reactor 无环）
- **视觉**：8 行横向砝码图（每行 = 一个杠杆，左标签，右三色进度条）+ 标注误判纠正

### P7-2 5 项风险（2 min）
- **风险 1**（最突出）：文档诚信——实体计数 3 处冲突、"10 份 orm.xml"实为 19、"多公司"夸大
  → 回应：审计识别 → 要求勘误 → 整改计划

  → 回应：07-13 落地完整委外引擎（生命周期 + MRP + GL 过账）

  → 回应：持续演进中（单代理→多路→语义保真→工具化）

### P7-3 健康度总结（1.5 min）

  5. 平台 bug 被主动发现并反哺 nop-entropy

- **视觉**：5 条 + 每人一图标 + 底部大字："Not zero-defect — defect-aware"

## Part 8：总结与展望（3 页 / 6 min）

### P8-1 三个 Takeaway（2 min）
- **Takeaway 1：人机协作的边界**
  - 用户 = 平台权威（A 类，告诉我怎么用平台）
  - 用户 = 工程方向指明（B 类，告诉我往哪个方向走）
  - 用户 ≠ 业务设计者（C 类，让我自己对标开源）
  - 早期密集介入 → 知识转移完成 → 后期放手
  - 这不是"AI 自动写代码"的故事，而是"AI 辅助 + 人类定向 + 系统化质量门控"

- **Takeaway 2：AGE 工作流的有效性**
  - 八阶段递进：每阶段的进入条件和退出门控明确
  - 强制双审计：草案拦截方向错误 + 结束验证交付完整性
  - 内部信息循环：Backlog → Plan → Audit → Bug → Skill，信息不流失
  - 自我纠偏：评分偏差回顾、子代理误判纠正、方法论持续演进

- **Takeaway 3：模型驱动的产品化路径**
  - ORM XML 唯一真相源 → codegen → 154 模块全链构建
  - Delta 定制就绪度 98%（Processor protected 全面开放）
  - 平台共生演化：发现 7 个平台问题 → 修复 nop-entropy → 反哺社区
  - Nop Platform 证明了自己适合构建产品级 ERP

### P8-2 给观众的建议（2 min）
- **如果你是技术团队负责人**：
  - AI 辅助开发不是"把需求扔给 AI"，而是需要有清晰的边界：什么该你定、什么让 AI 自行决定
  - 投资质量门控（审计、测试、工具）比投资"让 AI 写更多代码"更重要
  - 知识转移曲线：前期投入在平台机制纠正上，后期会指数级回报

- **如果你是 Nop 平台用户/潜在用户**：
  - Nop 的模型驱动是真的可以构建产品级应用的——不是玩具
  - Delta 定制是差异化竞争力（98% 场景可纯 Delta 完成）
  - 但 docs-for-ai 有缺口，建议源码级验证 + 社区交流

- **如果你是 AI 辅助开发的研究者**：
  - 三层注入模型（A/B/C）可以作为组织级 AI 采用的分析框架
  - 审计体系的 8 阶段演进展示了"AI 辅助开发的质量保障"如何从零建立
  - 内部信息循环系统是可复用的工程模式
