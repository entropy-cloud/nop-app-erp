nop-app-erp 的实验

nop-app-erp 是探索 AI 全自动开发边界的实验性项目。核心追问：如果没有人，AI 还能产出同样的系统吗？

实验记录：全量抽取 28 天日志，得到 28 条明确人类干预 + 80 次设计决策记录

继续追问：这 28 条干预 + 80 次决策，多大比例可以被另一 AI agent 替代？

初步判断：约 90% 的知识内容属于通用可编码类型，约 10% 涉及方向性/战略性判断

为什么需要更大一层的 Loop

  • Plan Loop 只管理"一次变更"

    谁决定下一个？谁确保全路线图被执行？谁空闲时触发审计？

  • Mission Driver = AGE 框架中 harness 层的自动化实现

    在 Plan Loop 之外加一个编排层。完全依赖人 → 手动、不可重复。

  • 时间尺度：天/周级

    一次完整循环处理 1-7 个计划、耗时 4-12 小时。整个路线图的流转需要数天到数周。它自动运行的底气，来自 attractor 已被定义好

关键设计：REVIEW 在 EXEC 之前 / EXEC 用 forEach / DRAFT 含自审计 / DEEP_AUDIT（全部完成不是停止信号，是升级审计的信号）

——这就是 Boris Cherny 说的"我的工作是写 Loop"，但前提是 Loop 在已定义的 attractor 内工作

Mission Driver 的实现架构——通用 Flow DSL 引擎

关键设计一：注入机制

AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符，引擎从 mission.json 配置中读取真实路径并注入。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt

关键设计二：隔离的顶级步骤

CHECK / REVIEW_PLANS / EXEC_PLANS / DRAFT_PLANS / DEEP_AUDIT 相互不传参数。所有状态通过 Plan 文件（> Plan Status: + [x] 复选框）和 Audit 文件持久化到磁盘。因此可以：单独执行任一步骤、从任意步骤开始向后执行、恢复不依赖内存——扫盘即知"当前在哪"

关键设计三：零 IPC，全文件

没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实

对人介入的支持

DRAFT_PLANS 持续起草新 Plan；REVIEW_PLANS 自动拾取 Status=draft 的 Plan（不区分来源）；人可随时修改 Plan 状态/路线图，下轮循环自动适配

真实运行记录：07-10 一次完整循环

5 步自动循环，每步都是 AGE harness 层的具体实现。

注意看时间尺度：从 CHECK(08:00) 到 DEEP_AUDIT(16:05) 约 8 小时，完成了一次完整的 Mission Driver 循环——处理 4 个计划、经历 5 个步骤。而整个路线图（18 域 + 业财 + 扩展域）从 06-22 到 07-13 横跨 22 天，Mission Driver 在这期间完成了数十次这样的循环。

两层 Harness 的关系——以及它们不能做的事

Plan Loop

单次变更控制。每次变更从起草到关闭的完整管理

Mission Driver Loop

编排控制。自动遍历路线图，决定"下一个做什么"，空闲时自动审计

但它们不能回答：路线图本身正确吗？"正确的结构"是什么？

07-12 竞争杠杆审计发现的"2 个夸大 + 2 个缺口"——不是因为 Harness 没执行好，是因为方向定义层（Attractor）有过问题（文档夸大"多公司 ERP"、制造委外曾是空壳）。Harness 执行得越好，方向错了损失越大。

过渡到 Act 4："所以我们需要解决的，不是第三个 Loop——而是一个完全不同性质的东西：系统长期收敛的目标结构（Attractor）。"

动力系统语言——为什么需要 Attractor

混淆 1：吸引子 ≠ 边界

边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。边界抓不住这种偏离。

混淆 2：吸引子 ≠ 更强的护栏

护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。护栏视角：每次合规；吸引子视角：方向在漂移。

混淆 3：吸引子 ≠ 控制目标

它是控制的目标因，不在控制之中。没有先定义 attractor，Harness 没有统一含义。反例：团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。结果：所有控制机制高效地固化了一套错误基线。

对 AI 自主开发研究的含义

提供了一个可公开审计的实证案例：在限定条件下（通用产品、无真实客户约束），AI 可以自主完成从 ORM 模型到 E2E 测试的全栈开发。之前的讨论停留在"AI 能不能写代码"——这里的答案是"能，而且能完成需求分析、设计决策和架构选择"

"全自动 AI 开发"的适用条件：

业务规则由通用最佳实践/公开标准决定

不依赖特定企业的流程偏好或历史包袱

参照系（竞品行为、会计准则、工程模式）公开可获取

设计决策的正确性可由内在一致性论证

这些条件恰好定义了一个"产品化通用软件"的范畴

从 28 条干预到全自动的最后一公里

需要构建三个基础设施：

平台知识注入 agent

定时运行合规审计脚本 + 读 Nop 平台源码变更，自动检测平台用法偏差

质量深度门控 agent

对每个 draft Plan 增加"深度校核"维度——竞品对比 / 替代方案分析 / 残留风险评估

领域知识注入 agent

读会计准则/ERP 实践文档/行业规范，定期生成设计修正建议

三个 Takeaway

Takeaway 1

把"变更"从对话变成契约

Plan Loop 给每次变更一个关闭契约（draft→审查→执行→审计→completed）

这是 Human On The Loop 的前提——AI 自主执行完整一轮，不等人

Takeaway 2

把"编排"从记忆变成自动

Mission Driver 自动遍历路线图、拾取 Plan、审查执行、空闲时审计

Plan 够了吗 → 不够 → Mission Driver 来编排——这是从 In 到 On 的关键跨越

Takeaway 3

把"方向"从个人变成文件

Attractor 承载于 owner docs，可版本、可审计、可继承

吸引子不能外包给 AI——定义方向是人的工作
