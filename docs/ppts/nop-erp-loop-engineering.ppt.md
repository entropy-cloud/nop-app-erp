## S02 · 变革点

**演讲词：**

传统软件工程的智能时代转型已经到来。

左边这张图展示了传统信念——人定义需求、AI 执行；人控制方向、AI 填充细节；人做架构决策、AI 写代码。这是今天大多数 AI 辅助开发的默认模式。

但 nop-app-erp 做的是一个更激进的实验。核心追问是：**如果没有人，AI 还能产出同样的系统吗？**

我们从 22 天的开发日志中抽取了全部记录，得到 **28 条明确的人类干预**和 **80 次设计决策**。这 80 次决策来自 grill-with-docs 设计审查的逐题决议——83 个问题中，80 项 AI 独立裁定，仅 4 题需要用户方向性纠正。

然后我们追问了一个更直白的问题——这 28 条干预和 80 次决策，AI agent 能替代其中的多少？

初步判断：大约 90% 的知识内容属于通用可编码类型，只有大约 10% 涉及方向性或战略性的判断。这意味着什么？意味着绝大部分用户介入是可以被自动化取代的。

**过渡：** 那这个项目具体是什么？翻到下一页。

## S03 · nop-app-erp 是什么

**演讲词：**

看屏幕上的三圆交叠图。

nop-app-erp 是三个圈的交叠：**Nop Platform** 提供低代码平台能力，**AGE 工作流** 提供 AI 辅助开发方法论，**ERP 应用层** 是业务产品本身。注意它不是框架核心，而是构建在 nop-entropy 之上的应用层产品。

规模数据：18+1 个业务域（18 个业务域加 1 个跨域通知派发子系统），154 个 Reactor Maven 模块，352 个自有实体（外加 110 个跨域引用桩），开发周期 22 天。

这些数字都不是我随口说的——全部经过独立审计校准，后面我会展示审计体系。

## S06 · 从 Prompt → Loop → AGE

**演讲词：**

（**语速放慢，逐字强调**）

这是今天的核心理论框架。

当前业界对 Loop Engineering 的讨论很热。有人说"我不再写 prompt 了，改成设计写 prompt 的系统"；有人提出了三层 Loop——Agentic Coding 分钟级、Developer Feedback 小时级、External Feedback 天/周级。

但这些讨论有一个共同隐含前提：**方向是已知的**。

AGE 框架追问了一个更根本的问题：方向谁定义的？谁来保证方向本身是对的？

看右边的图。业界的三层 Loop 全在 Harness 层——反馈控制 Loop。AGE 的贡献不是又多了一层 Loop，而是在下面加了方向定义层：状态空间 → 吸引子 → 轨迹 → 控制。

（**停顿**）

**关键句：Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。** 前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。

## S07 · Vibe Coding Loop

**演讲词：**

在引入 Plan Loop 之前，我们先看看传统 AI 开发的痛点——Vibe Coding。

屏幕上方这个无限循环箭头：用户说 → AI 写 → 用户改 → AI 再写 → ... 无限循环，没有退出条件。

它有四大缺失。第一，**无基线**——AI 从对话记忆推断代码状态，不知道当前状态是什么。第二，**无范围**——不知道要做什么，边界模糊，经常做多或做少。第三，**无退出标准**——不知道何时算"完成"，"差不多了"就算完。第四，**无审计**——没有人独立验证"是否正确"，AI 自己说自己做完了。

**过渡：** 那怎么解决？核心思路就是把"对话"变成"契约"——Plan Loop。

---

## S08 · 从 In 到 On 的关键：Plan

**演讲词：**

所以我们需要 Plan。Plan 不是任务清单，是**关闭契约**。

它包含三个核心要素：**当前基线（Current Baseline）**——从仓库读的，不靠记忆；**目标集（Goals + Non-Goals）**——明确做什么、更重要的，不做什么；**退出标准（Exit Criteria + Closure Gates）**——可观察的完成条件和关门检查表。还有一个辅助项——审查记录（Draft Review Record），记录迭代过程。

Plan 的生命周期是这样的：**起草 → 独立草案审查（必须用全新会话，fresh session）→ active（可执行）→ 执行阶段（含验证）→ 独立结束审计（也是全新会话）→ completed。**

红色标注的两个环节——独立草案审查和独立结束审计——是这套机制的核心。**生成与验收分离**：审计的人完全不看之前的聊天记录，从零开始审仓库。这是从 Human In The Loop（人在环中）升级到 Human On The Loop（人在环上）的前提。

这套 Plan 加审计加验证的联合机制，就是 Harness 中的 **Plan Loop**——控制单次变更轨迹的执行层。

## S09 · 实例：Plan 1100-1 销售定价引擎

**演讲词：**

理论讲完了，来看一个真实的 Plan 实例——1100-1 销售定价引擎。

看屏幕上的时间线。从左到右：draft → 审查第一轮发现"客户维度 Decision 缺失"，被打回 needs_revision。修订后第二次审查通过，进入 active 状态。然后执行 Phase 1 到 Phase 4，覆盖 ORM、SPI、测试、前端。最后 mvn test 全绿，结束审计 PASS，completed。

这个批次共有 4 个 Plan。看到屏幕上的红色圆点了吗？独立审查共拦截了 4 个 P0 缺陷——**码值冲突、BUDGET 污染实际财务、GlBalance 前提错误、维度歧义。全部在编码前拦截。**

（**语气加重，稍作停顿**）

如果没有 Plan Loop，这 4 个缺陷会等到实施甚至运行阶段才暴露，修复成本高 5 到 10 倍。Plan Loop 的本质就是用编码前的审查成本换运行期的修复成本——10 倍杠杆。

**过渡：** Plan Loop 解决了单次变更的问题。但来了 187 次变更呢？

## S10 · 为什么需要更大一层的 Loop

**演讲词：**

Plan Loop 让单次变更有了关闭契约，但 187 次变更呢？

问题来了：谁来排顺序？谁来保证每件事都做完了？谁在空闲时触发审计？谁把 Plan 的状态同步回路线图？

全凭人做——手动、不可重复、全靠注意力撑着。Plan Loop 仍然需要人逐个触发，没有编排层，本质上还是 Human In The Loop。

**Mission Driver** 就是解决这个问题的——在 Plan Loop 之外加一个自动编排层，实现从 In 到 On 的跨越。

看屏幕上的嵌套图：Mission Driver Loop 是天/周级的大 Loop，里面嵌套着多个 Plan Loop。它的时间尺度是 4 到 12 小时一次完整循环，整个路线图横跨 22 天。

## S11 · Mission Driver 的 5 步闭环

**演讲词：**

Mission Driver 的核心是 5 步闭环。我先快速过一遍，然后用具体例子展开。

第一步 **CHECK**——健康检查，确认 mvn 全绿。第二步 **REVIEW_PLANS**——自动捡起所有 draft 状态的计划，逐个审查，标记为 active。第三步 **EXEC_PLANS**——对每个 active 计划执行，含构建验证和结束审计。第四步 **DRAFT_PLANS**——读路线图、选工作项、起草新计划、自审计、标记 active。第五步 **DEEP_AUDIT**——全部完成后自动启动深度审计，不是停止，是升级。

（**指向右边子流程**）

看 EXEC_PLANS 的展开：对每个 Plan → 执行 Phase → BUILD_VERIFY → CLOSURE_AUDIT。DRAFT_PLANS 的展开：读路线图 → 选工作项 → 起草 → 自审计 → 标记 active。

几个关键设计：REVIEW 在 EXEC 之前避免重复起草；EXEC 用 forEach，每个 Plan 执行完重新扫描；DEEP_AUDIT 触发条件是"全 done"——全部完成不是停止信号，是升级审计的信号。

这就是 Boris Cherny 说的"我的工作是写 Loop"——但前提是 Loop 在已定义的 attractor 内工作。

## S12 · Mission Driver 的实现架构

**演讲词：**

（**简化、口语化**）

Mission Driver 的实现其实很简单：一份 JSON 配置、五个隔离的执行步骤、全部存文件——没有 web 服务器，没有 IPC。

三个关键设计值得一说。

第一，**注入机制**。就像考试——prompt 是试卷模板，mission.json 是密封线里的个人信息。AI 只负责答卷，不管这个卷子是给哪个项目的。换项目只需改配置，不改 prompt。

第二，**隔离的顶级步骤**。CHECK、REVIEW_PLANS、EXEC_PLANS、DRAFT_PLANS、DEEP_AUDIT 之间不传参数。所有状态通过 Plan 文件和审计文件存到磁盘。所以你可以从任意步骤开始执行，断点了扫一下盘就知道"当前在哪"——不依赖内存。

第三，**零 IPC，全文件**。没有进程间通信，没有 web server。这是 AGE"仓库等于唯一真相源"原则在工具层的落实。

对人介入的支持很干净：DRAFT_PLANS 持续起草，REVIEW_PLANS 自动拾取任何 draft 状态的 Plan——不管是你手写的还是 AI 起草的。人随时改 Plan 状态或路线图，下轮自动适配。

## S14 · 两层 Harness 的关系——以及它们不能做的事

**演讲词：**

回顾一下两层 Loop 的关系。

Plan Loop 控制单次变更。Mission Driver Loop 控制编排。它们构成了 Harness 层，能高效地"按路线图执行"。

但有一个它们不能回答的问题：**路线图本身正确吗？"正确的结构"是什么？**

（**语速放慢**）

讲一个真实的教训。07-12 竞争杠杆审计，我们让子代理核实 8 个"超越点"是否兑现。结果发现：2 个夸大（文档说"多公司 ERP"，实际只有地基；说"域独立部署"，言过其实）和 2 个缺口（制造委外是空壳、Delta 文档数据错误）。这些不是 Harness 没执行好——是方向定义层出了问题。Harness 执行得越好，方向错了损失越大。

所以我们需要解决的，不是第三个 Loop——而是一个完全不同性质的东西：系统长期收敛的目标结构，我们称之为 **Attractor**。

（**停顿 2 秒，环视观众**）

注意屏幕右侧这个脉冲跳动的橙色圆点——它像心脏一样持续跳动，象征吸引子是一个活的、引领系统持续收敛的目标结构，不是静态文档。

**方向不是 Loop。环变暗，轴点亮。**

## S15 · 动力系统语言——为什么需要 Attractor

**演讲词：**

（**先建立直觉，再给概念**）

想象我们在沙漠中行走。状态空间是所有可能的方向——每一刻代码、配置、文档的快照。Attractor 是绿洲——系统长期要走向的地方。轨迹是我们实际走过的路。控制就是把走偏的路线拉回绿洲方向的机制——就是 Plan Loop 加 Mission Driver Loop。

吸引子不是文档——文档是承载者。吸引子也不是代码——代码是投影。吸引子是类似"所有实现必须从 ORM XML 单一生成"这种**结构不变量**。

在 nop-app-erp 中，我们定义了三个具体的吸引子：

- **ORM XML 唯一真相源**——所有持久化代码从 XML 生成，手写绕过等于编译失败。这是数据空间的收敛。
- **DAG 加 R/S/P 跨域契约**——模块依赖是无环有向图，跨域引用分三类：只读桩、同步接口、弱关联字符串。共 369 条白名单。这是模块空间的收敛。
- **三轴正交状态分离**——单据状态（docStatus）、审批状态（approveStatus）、过账状态（posted）三轴独立，互不污染。这是业务状态空间的收敛。

这三个不变量贯穿了 22 天所有设计的始终。

（**过渡：三种常见混淆**）

澄清三个最容易混淆的点。

第一，**吸引子不等于边界**。边界禁止行为——违反就报错。吸引子定义汇聚目标——单次偏离不报错但持续偏离让结构腐烂。规定"不准用 JPA"是边界，AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。边界抓不住这种偏离。

第二，**吸引子不等于更强的护栏**。护栏在执行层检查每次动作，吸引子在方向层评估多次累积。10 次 PR 代码规范都通过，但测试套件已与旧实现深度耦合——护栏视角是"每次合规"，吸引子视角是"方向在漂移"。

第三，**吸引子不等于控制目标**。它是控制的前提。团队有完美的 CI/CD 加 review 加测试门禁，但没人回答过"系统该向什么架构收敛"——结果所有控制机制高效地固化了一套错误基线。

## S16 · 用户其实在做什么——定义和演进吸引子

**演讲词：**

好，理论讲完了。那问题回到：用户在整个 22 天里到底做了什么？

看这张表。用户介入可以分为三类。

**A 类**，9 次，集中在 06-22 到 07-03。用户明确指明平台机制——比如"租户开关是全局配置，不是每个实体手写"。这修正了吸引子的技术基底。

**B 类**，9 次。用户指明工程原则方向——比如"过账方法应挂聚合根 BizModel"。这修正了吸引子的结构原则。

**C 类**，绝大多数，07-04 后为主。用户只让 AI 自查对比——比如"对照 erp-survey 检查功能覆盖"。这是在已定义的吸引子内让 AI 自行扩张。

看表格下方的时间线——红色 A 类从左到右逐渐消失，绿色 C 类成为绝对主力。**关键洞察：用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 能自动扩张。**

## S18 · 8 阶段演进

**演讲词：**

项目的 22 天可以清晰地划分为 8 个阶段。

看这张甘特图。06-22 到 06-25 是 **S1 设计期**，4 天——竞品调研加 ORM 建模加 codegen 首次闭环。S2 是深化期，grill 83 题消除设计歧义。S3 做的是**三层分离和 Processor 模式**——让业务逻辑从上到下分层。S4 是**模式复用和批量深化**——把 S3 的经验快速复制到所有扩展域。S5 是业财加运营成熟度。S6 是审计整改。S7 是 E2E 测试基础设施。S8 是深化收尾。

节奏特征是两头重：前 4 天设计建基座，后 4 天竞争核实加产品化打磨。中间是 Mission Driver 批处理模式铺开。

每个阶段都有清晰的进入条件和退出门控——S1 退出后构建闭环，S2 退出后设计完备，S3 退出后核心域业务逻辑通过，以此类推。没有跳跃。

## S19 · 六种自驱动机制

**演讲词：**

除了外部用户输入，项目还建立了六种自驱动机制——看屏幕上的六边形拓扑图。

第一，**计划-审计闭环**——187 份计划，强制双审计。第二，**路线图驱动**——3 个子路线图逐项推进。第三，**审计-整改驱动**——9 轮审计，每轮发现问题后强制整改。第四，**测试回归防护**——约 2890 个单元测试加 260 多个 E2E。第五，**竞品对标驱动**——约 24 个开源项目深度调研作为功能完整性参照系。第六，**文档一致性驱动**——ORM XML 是唯一真相源，散文不重复字段定义。

这六种机制之间的拓扑关系是：计划-审计闭环居中，右侧是路线图驱动和竞品对标，左侧是测试回归防护和文档一致性，顶端是审计-整改驱动。它们共同构成了项目的自演化引擎——即使没有外部输入，项目也能持续向吸引子收敛。

## S20 · 对 AI 自主开发研究的含义

**演讲词：**

nop-app-erp 提供了一个可公开审计的实证案例。

在限定条件下——通用产品、没有真实客户约束——AI 可以自主完成从 ORM 模型到 E2E 测试的全栈开发。

注意这不是"AI 能写代码"的又一个证明——这个结论已经不需要更多证明了。这里的关键答案是：**AI 能完成需求分析、设计决策和架构选择**。

当然，这有适用条件：

第一，业务规则由通用最佳实践或公开标准决定。第二，不依赖特定企业的流程偏好或历史包袱。第三，参照系（竞品行为、会计准则、工程模式）公开可获取。第四，设计决策的正确性可由内在一致性论证。

这些条件恰好定义了一个"产品化通用软件"的范畴。AI 全自动开发不是万能药，但在产品化通用软件领域，边界条件已经满足。

## S21 · 从 28 条干预到全自动的最后一公里

**演讲词：**

回到开头的问题：28 条用户干预，我们能自动化多少？

我判断大约 90% 可以被替代。那剩下的 10% 怎么解决？需要构建三个基础设施。

第一，**平台知识注入 agent**——定时运行合规审计脚本，读平台源码变更，自动检测用法偏差。第二，**质量深度门控 agent**——对每个草案增加深度校核维度，包括竞品对比、替代方案分析、残留风险评估。第三，**领域知识注入 agent**——读会计准则、ERP 实践文档、行业规范，定期生成设计修正建议。

当前到下一阶段：先做可替代性分析，然后进入下一阶段——把 A 类和 B 类干预的 90% 自动化。

**过渡：** 讲了这么多，三个核心信息送给大家。

## S22 · 三个 Takeaway

**演讲词：**

**Takeaway 1：把"变更"从对话变成契约。** Plan Loop 给每次变更一个关闭契约——起草、审查、执行、审计、完成。这是 Human On The Loop 的前提——AI 自主执行完整一轮，不等人。

**Takeaway 2：把"编排"从记忆变成自动。** Mission Driver 自动遍历路线图、拾取 Plan、审查执行、空闲时审计。Plan 够了吗？不够——Mission Driver 来编排。这是从 In 到 On 的关键跨越。

（**停顿**）

**Takeaway 3：把"方向"从个人变成文件。** Attractor 承载于 owner docs，可版本控制、可审计、可继承。**吸引子不能外包给 AI——定义方向是人的工作。**

## S23 · 落地路径——三种项目模式

**演讲词：**

如果你想把这套方法用在自己的项目上，有三种模式。

**模式 A：已有项目。** 先加 Plan Loop——修改 PR 模板，每次变更加 Plan 加独立审计。先定义一个 Attractor——核心架构文档。再看是否需要 Mission Driver。最小改动模式。

**模式 B：新项目。** 从 AGE 模板开始，它已经包含了 Plan Loop、Mission Driver 和 Attractor 骨架。先花 1 到 2 天定义 Attractor——这个步骤不能外包给 AI。然后运行 Mission Driver。全自动模式。

**模式 C：平台或框架。** Attractor 定义最重要——平台方向决策权不能交给 AI。Plan Loop 用于框架自身开发控制。方向决策模式。

### Q&A 预判

1. **"你们的经验能复制到其他项目吗？"**
   → 控制结构可复制（Plan Loop + Mission Driver + Attractor 骨架），但方向定义（吸引子内容）不能复制，每个项目不同

2. **"AI 全自动开发什么时候能完全实现？"**
   → 在产品化通用软件领域已经接近；在依赖特定企业流程的领域还有距离

3. **"Plan 审查的成本是多少？"**
   → 典型 Plan 4-12 小时完成，审查时间约占 20%。编码前的审查成本换运行期 5-10 倍的修复成本节省——10 倍杠杆

4. **"如果团队没有 Nop 平台呢？"**
   → AGE 方法论的吸引子概念和 Plan Loop 控制机制不依赖特定平台。任何技术栈都可以用

5. **"22 天听起来太顺利了——有没有失败的尝试？"**
   → 有。07-12 竞争杠杆审计发现的 2 个夸大声明和 2 个缺口就是失误案例。制造委外曾是空壳，文档一度夸大"多公司 ERP"。但 Plan Loop 的本质正是从这些失败中学习——审计发现问题 → 起草新 Plan → 执行修复 → 闭合。不是不犯错误，是 1-3 天闭合
