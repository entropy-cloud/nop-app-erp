Act 4：最外层控制 — Attractor Evolution Cycle

定义方向 · 吸引子是系统应长期回归的稳定结构

什么是 Attractor——dynamical systems 视角

🧲 Attractor（吸引子）

系统应长期回归的稳定结构

🛤️ Trajectory（轨迹）

每轮生成-验证-修正的实际路径

🌌 State Space

系统可能进入的所有状态

🎮 Control（控制）

将轨迹拉回 attractor 的机制
Plan Loop + Mission Driver

三个最易混淆的点：吸引子 ≠ 边界（禁止行为 vs 汇聚目标）· 吸引子 ≠ 护栏（执行层 vs 方向层）· 吸引子 ≠ 控制目标（先于控制存在）

nop-app-erp 的三个核心吸引子

模型驱动

不变量：所有实现从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者：module-*/model/*.orm.xml

状态一致性

不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者：docs/design/*/state-machine.md

过账契约

不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者：docs/design/finance/posting.md

吸引子不是文档——文档是承载者。吸引子是"所有实现必须从 ORM XML 生成"这个结构不变量。

用户其实在做什么——定义和演进吸引子

AGE 文章 §9："humans define new attractors; AI expands rapidly around the established attractor."

介入类型频次对吸引子的影响
A 类9 次，06-22~07-03修正吸引子的技术基底——明确指明平台机制
B 类9 次修正吸引子的结构原则——指明工程方向
C 类绝大多数，07-04 后为主在吸引子内让 AI 自行扩张——只让 AI 自查对比

用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 能自动扩张。

【全场高潮】三层汇聚——从割裂到嵌套

最外层：吸引子演进（周级）
—— 定义方向

↓ 方向变化产生新工作项

中间层：Mission Driver（天级）
—— 自动编排

↓ 每个工作项走 Plan Loop

最内层：Plan（小时级）
—— 单次变更关闭

↑ 完成后状态同步回路线图

Plan 关闭每一次变更。Mission 编排每一个计划。Attractor 定义方向。
三层都在，系统才能自运转。缺一层，要么没方向，要么没执行，要么没闭环。

核心健康指标不是"零缺陷"，而是：

  • 缺陷被发现后几乎全部在 1-3 天内闭合

  • 审计会自我纠偏（纠正子代理误判 + 回顾评分偏差）

有人看到 22 天 154 模块，说 AI 真强——这是错误归因。

真正可复制的是三层控制 Loop 的设计模式：

Plan Loop

docs/plans/ + fresh session 审计契约 —— 任何项目都能用

Mission Driver Loop

tools/mission-driver/ engine.js + flow JSON —— AGE 模板已包含

Attractor Evolution Cycle

docs/design/ + docs/architecture/ 骨架 —— 承载吸引子的 owner docs 体系

AGE 模板包含这三层的完整实现：cp -r ~/attractor-guided-engineering-template/ ./my-project/ 即开即用

三个 Takeaway

把"变更"从对话变成契约

对话不可追溯 → Plan Loop 给每次变更一个关闭契约（draft → review → audit → close）

同时：人机有清晰边界——A 类=平台权威，B 类=方向指南，C 类=放手自查

把"编排"从记忆变成自动

不需要人记住下一个做什么 → Mission Driver Loop 自动选工作项、起草、验证、审计

同时：对抗性审计保障质量——8 阶段演进从单代理到语义保真

把"方向"从个人变成文件

架构判断是稀缺资源，不应只在一个人脑子里 → 吸引子承载于 owner docs，可版本、可审计、可继承

同时：模型驱动 + 平台共生——ORM XML 唯一真相源，反哺 nop-entropy

给不同角色的建议

  • 技术团队负责人

    投资质量门控（审计/测试/工具）比投资"让 AI 写更多代码"回报更高。知识转移前期投入，后期指数回报。

  • Nop 平台用户/潜在用户

    模型驱动能构建产品级应用，Delta 定制是差异化竞争力（~98% 纯 Delta）。但 docs-for-ai 有缺口，建议源码级验证。

  • AI 辅助开发研究者

    三层注入模型（A/B/C）可作为组织级 AI 采用的分析框架。审计 8 阶段演进 + 内部信息循环是可复用工程模式。

结尾：不是回扣，是挑战

"Plan 关闭每一次变更。Mission 编排每一个计划。Attractor 定义方向。三者都到位，系统才会自运转。"

但 AGE 文章说得很清楚：吸引子不能外包给 AI。

定义方向是你的工作。AI 可以在吸引子内高速扩张，但吸引子需要人来定义。

所以今天讲的不是一个"AI 写代码 4000 行/天"的故事。

这是一个"人定义方向 · AI 执行扩张 · 系统检查偏离"的故事。

"100 个变更后，系统还在正确的结构上吗？—— 是的——如果三层 Loop 都在。"
