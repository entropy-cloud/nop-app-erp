## 一、项目总体特征

### 1.1 一句话定位

nop-app-erp 是基于 Nop Platform（nop-entropy 低代码平台）构建的**产品化通用 ERP 参考应用**，采用 **AGE（Attractor-Guided Engineering，吸引子引导工程）工作流**进行 AI 辅助开发。它不是框架核心项目，而是构建在 nop-entropy 之上的应用层产品。

### 1.3 三个本质特征

1. **模型驱动（Model-First）**：一切从 ORM XML 源模型出发，经 `nop-cli gen` 代码生成产出全链路模块（model → codegen → dao → meta → service → web → app → api）。ORM XML 是唯一真相源。
2. **文档先行（Docs-as-Truth）**：设计文档（`docs/design/`）、架构文档（`docs/architecture/`）、需求文档（`docs/requirements/`）在任何代码变更之前先行定型，代码实现严格对齐文档。
3. **对抗性审计（Adversarial Audit）**：每个非平凡计划必须通过独立子代理的草案审计和结束审计；全项目经历多轮综合审计与竞争对标审计。

### 1.4 三层嵌套控制 Loop（AGE 框架）

22 天的开发并非线性推进，而是由三层嵌套控制 Loop 共同驱动：

| 层级 | 类型 | 时间尺度 | 职责 | 实现载体 |
|------|------|----------|------|----------|
| **内层** | Plan Loop | 小时级（4-12h） | 单次变更的关闭契约 | `docs/plans/` + 独立草案/结束审计 |
| **外层** | Mission Driver Loop | 天-周级（~8h/循环） | 自动路线图编排 + 自审计 | `flows/mission-driver.json` + `missions/erp.json` |
| **轴心** | Attractor | 贯穿全程 | 系统长期收敛的目标结构 | `docs/design/` + `docs/architecture/` owner docs |

#### 1.4.1 Plan Loop——将"变更"从对话变成契约

Plan 是 AI 自主执行的基本单位。每一份 Plan 包含：

- **Current Baseline**：从仓库读取的当前状态快照，不依赖对话记忆
- **Goals + Non-Goals**：明确做什么和不做什么
- **Exit Criteria**：可观察的关闭条件
- **Closure Gates**：最终验证检查清单
- **Draft Review Record**：独立审查迭代记录

Plan 生命周期：`draft → 独立草案审查（fresh session）→ active → 执行（含验证）→ 独立结束审计（fresh session）→ completed`

**关键规则**：生成与验收分离——审计必须由独立子代理在全新会话中执行，不继承执行上下文。这是从 Human In The Loop 升级到 Human On The Loop 的前提。

**实证**：189 份 Plan 全部经过此生命周期。典型 Plan 4-12 小时完成。最短 ~4h（07-10 的 4 个计划同日完成）。没有以分钟计的计划——每条都必须经过审查→执行→验证→审计门控链。

**编码前缺陷拦截**：Plan 1100-1（销售定价引擎）批次 4 个 Plan 的独立审查共拦截 4 个 P0 缺陷——码值冲突、BUDGET 污染实际财务、GlBalance 前提错误、维度歧义。全部在编码前拦截。如果没有 Plan Loop，这 4 个缺陷在实施/运行期才暴露，修复成本高 5-10 倍。

#### 1.4.2 Mission Driver Loop——将"编排"从记忆变成自动

Plan Loop 让单次变更有了关闭契约，但 189 次变更需要一个人来决定"下一个做什么"——这本质上还是 Human In The Loop。Mission Driver 在 Plan Loop 之外加一个自动编排层，实现从 In 到 On 的跨越。

**5 步闭环**：

```
    ┌──────────┐
    │  CHECK   │ 健康检查（mvn 全绿确认）
    └────┬─────┘
         ▼
    ┌──────────┐
    │REVIEW    │ draft Plan → active（自动拾取）
    │_PLANS    │
    └────┬─────┘
         ▼
    ┌──────────┐
    │EXEC_PLANS│ forEach active Plan → 执行 → 验证 → 审计
    └────┬─────┘
         ▼
    ┌──────────┐
    │DRAFT     │ 读路线图 → 选工作项 → 起草 → 自审计 → active
    │_PLANS    │
    └────┬─────┘
         ▼
    ┌──────────┐                                       ┌────────────┐
    │DEEP_AUDIT│ ← 空闲时自动启动深度审计（全部完成     │ 发现缺陷    │
    └──────────┘   不是停止信号，是升级审计的信号）      │ →新Plan    │
                                                        └────────────┘
```

**关键设计**：
- **注入机制**：AI step 的 prompt 模板含 {{planGuide}} 等占位符，引擎从 mission.json 配置注入。换项目只需改配置，不改 prompt。
- **隔离的顶级步骤**：相互不传参数，可单独执行任一步骤，断点扫盘恢复。
- **零 IPC，全文件**：所有状态通过 Plan 文件/Audit 文件持久化到磁盘。

**对人介入的支持**：DRAFT_PLANS 持续起草新 Plan；REVIEW_PLANS 自动拾取任何 Status=draft 的 Plan（不区分来源——手工创建或自动起草都一样）；人可随时修改 Plan 状态/路线图，下轮循环自动适配。

**07-10 真实运行记录**：
```
08:00 CHECK — mvn 全绿
08:05 REVIEW_PLANS — 4 个 draft → all active
09:30 EXEC_PLANS — 4 个 Plan 全绿 ✅
16:00 DRAFT_PLANS — 路线图全 done
16:05 DEEP_AUDIT — 自动启动深度审计
```
一次循环 ~8h，处理 4 个计划。整个路线图（06-22 → 07-13）由数十次这样的循环完成遍历。

#### 1.4.3 Attractor——将"方向"从个人变成文件

Plan Loop + Mission Driver Loop 构成了 Harness（控制层），它们能高效地"按路线图执行"。但有一个它们不能回答的问题：**路线图本身正确吗？"正确的结构"是什么？**

Attractor（吸引子）回答的就是这个问题——它定义系统应向哪里长期收敛，为控制 Loop 提供方向。

**nop-app-erp 的三个核心 Attractor 维度**：

1. **ORM XML 唯一真相源（数据空间）**：所有持久化代码从 XML 生成，手写绕过 = 编译失败
2. **DAG + R/S/P 跨域契约（模块空间）**：模块依赖为 DAG，跨域引用分三类（R 只读桩/S 同步 I*Biz/P 弱关联字符串），369 条白名单
3. **三轴正交状态分离（业务状态空间）**：docStatus（域状态机）× approveStatus（四态统一）× posted（过账引擎独占）

**三种常见混淆**（AGE 文章原文）：
- **吸引子 ≠ 边界**：边界禁止行为，吸引子定义汇聚目标。规定"不准用 JPA"（边界），AI 用 JDBC 写 SQL——没违反边界但破坏了 ORM 唯一真相源。边界抓不住这种偏离。
- **吸引子 ≠ 更强的护栏**：护栏在执行层检查每次动作，吸引子在方向层评估多次累积是否靠拢。10 次 PR 代码规范都通过（护栏），但测试套件已与旧实现深度耦合。护栏视角：每次合规；吸引子视角：方向在漂移。
- **吸引子 ≠ 控制目标**：它是控制的目标因，不在控制之中。团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"——所有控制机制高效地固化了一套错误基线。

**与 Andrew Ng 三层 Loop 的关系**：Ng 的三层（Agentic Coding → Developer Feedback → External Feedback）全在 Harness 层。AGE 的贡献不在又多一层 Loop，而在发现了 Loop 的前提——方向定义层。

#### 动力系统语言的四层结构

AGE 框架借用了动力系统（dynamical systems）的语言来建模 AI 开发过程：

1. **状态空间（State Space）**：系统可能进入的所有状态——每一刻的代码、配置、文档的快照
2. **吸引子（Attractor）**：系统长期收敛的目标结构——ORM XML 唯一真相源、DAG 跨域契约、三轴状态分离
3. **轨迹（Trajectory）**：每轮生成-验证-修正的实际路径——每个 Plan 的 draft→active→completed
4. **控制（Control）**：将轨迹拉回吸引子的机制——Plan Loop + Mission Driver Loop

四者关系：状态空间包含所有可能方向，吸引子定义"应向哪收敛"，轨迹记录实际演化路径，控制确保轨迹不偏离吸引子。

#### 全场架构总图

将以上内容浓缩为一张架构图：

```
方向层（不是 Loop，是结构不变量）：
  Attractor 的三个维度（§1.4.3）→ 定义"系统应向哪里收敛"
─────────────────────────────────────────────────────────
Harness 层（控制 Loop）：
  内层：Plan Loop（小时级）→ 每次变更的关闭契约
  外层：Mission Driver Loop（天/周级）→ 自动编排 + 自审计
─────────────────────────────────────────────────────────
关系：Harness 在 Attractor 内运行，Attractor 为 Harness 提供方向
```

**与 Andrew Ng 三层 Loop 的关系**：Ng 的三层（Agentic Coding → Developer Feedback → External Feedback）全在 Harness 层。AGE 的贡献不在又多一层 Loop，而在发现了 Loop 的前提——方向定义层。

**关键句**："Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"

## 三、外部信息注入分析

这是本次分析的核心维度之一：**外部在什么时候注入了什么信息，注入的具体程度如何，这些信息如何塑造了项目的走向**。

### 3.1 核心区分：介入的三个层次

用户的每一次介入并非同等"具体"。按**介入的具体程度**可分为三个层次，这直接决定了 AI 的自由度：

| 层次 | 特征 | 用户给了什么 | AI 自由度 | 占比 |
|------|------|-------------|-----------|------|
| **A. 明确指明 nop 平台机制** | 用户直接说出平台怎么工作、该用哪个机制 | 具体机制名称 + 配置项 + 源码位置 | 极低——AI 只需执行核实 | 少但关键 |
| **B. 指明工程/设计原则方向** | 用户给出原则或提出质疑，但不给具体平台 API | 原则/质疑/方向 | 中——AI 裁定具体实现 | 少 |
| **C. 只让 AI 自查/对比** | 用户只提要求或问题，完全不暗示答案 | 任务/问题 | 高——AI 自行调研决定 | 绝大多数 |

**关键洞察**：用户有清晰的边界意识——**nop 平台已提供的机制，明确介入（A 类）；ERP 业务逻辑和功能完整性，让 AI 自查（C 类）**；只在实现偏离工程原则时给出方向性纠偏（B 类）。一般情况下用户只是让 AI 自己去检查完整性、对照开源项目，并没有明确指出如何去做；但关于 nop 平台已经提供的机制，则需要用户明确指明（因为 AI 对平台的了解不如用户）。

用 AGE 的语言总结：用户不是在"监督 AI 写代码"，而是在**定义和演进吸引子**。A 类介入修正吸引子的技术基底（ORM 建模、平台 API 选择），B 类介入修正吸引子的结构原则（聚合根、文档分层），C 类介入在已定义的吸引子内让 AI 自行扩张。一旦吸引子定型，用户介入自然归零。

### 3.2 A 类：明确指明 nop 平台机制

这类纠正的共同特征是：**用户比 AI 更了解 nop 平台，直接说出平台机制名称或指出该用哪个机制**。AI 自己查 docs-for-ai 是查不到部分内容的——有些是源码级机制，有些是 docs-for-ai 当时根本没写。

| # | 用户原话/意图 | 明确指明的平台机制 | 为何 AI 自己查不到 | 日期 |
|---|--------------|-------------------|-------------------|------|
| A1 | "有一个全局开关可以缺省为每个未指定 useTenant 的表启用租户" | `nop.orm.enable-tenant-by-default` 配置 + `orm-gen.xlib` 的 `TenantSupport` feature 标签 | **docs-for-ai 当时未记录此全局开关**，只记录了按实体声明 useTenant。用户直接点破了配置层的隐藏机制 | 06-22 |
| A2 | "把 Delta 扩展（ext:baseClass）与 notGen 外部表引用搞混了。nop 的'外部表引用'是 `<entity notGenCode='true'>` 机制，不是 Delta" | `entity.xdef` 的 `@notGenCode` 注释 + `orm-gen.xlib:228` 平台自身用法 | AI 第一次调研时**漏看了这两处源码证据**，看到 9 个内置模块零跨引用就误推出"硬规则禁止跨模块引用"。用户指出后才找到正确证据 | 06-22 |
| A3 | "需对照 docs-for-ai 检查字段类型选择，确认 Nop 数据类型只有 StdSqlType 枚举定义的类型" | `StdSqlType` 是唯一合法类型枚举；`INT` 非法（应为 `INTEGER`）；`dictName=` 非标准（应用 `ext:dict=`）；dict value 必须 10/20/30 | AI 之前用了非标准写法（dictName、INT、dict value 0/1/2），因为没有逐条对照 docs-for-ai 的类型规范 | 06-22 |
| A4 | "主键设计节只展示了 VARCHAR(36) 一种方案，过于强制" | 平台支持两种主键方案：`VARCHAR(36)+tagSet="seq"` / `BIGINT+tagSet="seq-default"` | docs-for-ai 当时只展示了 UUID 一种方案，AI 照搬。用户知道还有自增方案 | 06-22 |
| A5 | "domain 是类型模板，用于定义可复用的列类型配置，不是用于标记业务语义" | `domain` 的平台语义——`OrmModelInitializer.syncDomains()` 将 domain 属性复制到 column | **AI 误解 domain 为业务语义标记**。用户纠正后 AI 才去查源码确认 | 06-22 |
| A6 | "code 是 Java 常量后缀，与字典名前缀组合生成常量" | `_{moduleClassPrefix}DaoConstants.java.xgen` 的常量生成机制。`code` → `{DICT_PREFIX}_{CODE}` | **AI 混淆了 code 与 value**。用户纠正后 AI 才理解 code 是生成 Java 常量的后缀 | 06-22 |
| A7 | "只有主键是 id 的情况下，设置 ext:allowIdAsColName 才有意义" | `id` 是系统保留属性名；`allowIdAsColName` 的精确适用条件 | AI 之前写了过于宽泛的规则，用户精确定界 | 06-22 |
| A8 | "所有字典改 string、所有引用字段改 varchar" | 对齐平台 `use-approval` 机制（审批状态字段需 string 字典） | **用户知道平台约束（use-approval 要求 string），AI 当时不知道**。AI 后来才在调研中确认原因 | 07-03 |
| A9 | "设计文档包含过多细粒度技术设计，与 docs-for-ai 和 AGE 模板规范不一致" | 指明 docs-for-ai 为规范来源——设计文档不应含 Java/XML/YAML 代码块 | AI 之前在设计文档里写了大量 Java 注解示例和 XML 定义，因为没有对照 docs-for-ai 的文档规范 | 06-22 |

**A 类模式**：全部发生在**项目早期（06-22 ~ 07-03）**，集中在 ORM 建模和平台 API 层面。用户充当了"平台知识权威"，纠正 AI 对 nop 平台的误解。一旦这些机制被纠正确认并沉淀到文档/skills，后期就不再需要这类介入。

**深层原因**：AI 对 nop 平台的认知来源于 docs-for-ai，但 docs-for-ai 不是全部真相——配置层（application.yaml + beans.xml + xlib feature 机制）有文档未覆盖的开关，源码层有文档未展开的机制。用户拥有源码级平台知识，能直接点破 AI 的盲区。

### 3.3 B 类：指明工程/设计原则方向

这类纠正的特征是：**用户给出原则或提出质疑，但不告诉具体用哪个平台 API**。AI 需要自己想清楚怎么做。

| # | 用户原话 | 指明的原则 | AI 自行决定的部分 | 日期 |
|---|---------|-----------|------------------|------|

### 3.5 知识转移曲线——全场核心洞察

这是一个贯穿整个 22 天的动态过程，揭示了一个根本规律：

```
06-22  ███████████  A 类密集 + B + C 起步
06-29        ██████████  C 类（grill 83 问，仅 4 题需纠正）
07-01  ██    ██      最后两次 A 类 + 两次关键 B 类
07-03  █               最后一次 A 类
07-04+         ████████████  几乎全 C 类
```

**双 Y 轴趋势**（X=时间，左 Y=A 类频次↓，右 Y=AI 自主度↑）：
- A 类频次从 06-22 的高峰（9 次/前 12 天）逐渐归零（07-03 后为 0）
- AI 自主度从 06-22 的 ~25%（需要 A 类频繁纠正）逐渐上升到 07-13 的 ~100%
- 两条曲线在 **06-29 ~ 07-01** 交叉——此后 AI 自主成为主要工作模式

**三阶段总结**：
- **早期（06-22 ~ 06-26）**：用户是"nop 平台知识权威"，A 类介入频繁。AI 通过被纠正 + 自行源码核实，快速建立平台机制认知。9 条明确 A 类干预（详见 §3.2 表格）中的多数在此阶段。
- **中期（06-29 ~ 07-01）**：两次关键 B 类介入（过账引擎 DDD 原则 + 架构文档元规则）确立了工程范式边界。grill-with-docs 83 问仅 4 题方向性纠正。
- **后期（07-04 ~ 07-13）**：平台知识已沉淀完毕，用户退到"只提要求不给方法"的 C 类模式。AI 的 19 个可复用提示词 + docs-for-ai 补充 + lessons 沉淀构成了自运转的知识基座。

**核心结论**：
> "用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 能自动扩张。"
> "用户不是在写代码，而是在转移平台知识。一旦知识转移完成，他的介入频率自动归零。"

这条曲线揭示了 AGE 工作流的本质：人的稀缺注意力应用在**定义吸引子**上，而非监督执行。一旦吸引子定型，AI 在吸引子内的自主扩张几乎不需要人介入。

## 四、自身演化的驱动力分析

这是本次分析的第二个核心维度：**去掉外部用户输入后，项目自身依靠什么信息不断演化**。

### 4.1 七种自驱动机制

七种机制分为两层：**Harness 层机制**（Plan Loop + Mission Driver Loop，见 §1.4）和**支撑层机制**（路线图/审计-整改/测试回归/竞品对标/文档一致性）。

#### Harness 机制 1：Plan Loop——计划-执行-审计闭环

这是 AGE Harness 层中最内层的控制 Loop。项目的默认工作流是：
```
需求/想法 → 计划草案 → 独立草案审计（fresh session）→ 修订 → active → 执行 → 独立结束审计（fresh session）→ completed
```

Plan 是 AI 自主执行的基本单位。每一份 Plan 包含关闭契约五要素：Current Baseline / Goals + Non-Goals / Exit Criteria / Closure Gates / Draft Review Record。它让每次变更从"对话"变成"契约"——有边界、有基线、有退出标准、可审计、可恢复。

- **189 份计划文件**，每份有明确的状态（draft/active/completed）、范围、退出标准、验证检查清单。
- **强制双审计**：草案审计拦截方向性错误（如 07-10 拦截 4 个 P0：码值冲突/BUDGET 污染/架构前提错误/维度歧义）；结束审计验证交付完整性。
- **审计是独立子代理在全新会话冷重放**，不继承主代理上下文，避免确认偏差。
- **生成与验收分离**：这是从 Human In The Loop 到 Human On The Loop 的前提——AI 自主执行完整一轮，不等人。

> 与 Mission Driver 的关系：当 Mission Driver 的 EXEC_PLANS 步骤运行时，它对每个 active Plan 执行 plan-execution subflow，走的就是 Plan Loop。两层 Loop 在此嵌套。

#### Harness 机制 2：Mission Driver Loop——自动路线图编排

Plan Loop 之上是 Mission Driver Loop，它解决的是"谁来决定下一个做什么"的问题。Mission Driver 是一个通用 Flow DSL 引擎，核心设计：

- **5 步闭环**：CHECK → REVIEW_PLANS → EXEC_PLANS → DRAFT_PLANS → DEEP_AUDIT
- **注入机制**：prompt 模板含 {{planGuide}} 占位符，引擎从配置注入，换项目只需改配置不改 prompt
- **隔离的顶级步骤**：相互不传参数，可单独执行任一步骤，扫盘恢复
- **零 IPC，全文件**：无 web server，状态通过 Plan/Audit 文件持久化

对人介入的支持：DRAFT_PLANS 持续起草；REVIEW_PLANS 自动拾取任何 Status=draft 的 Plan（不区分来源）；人可随时修改 Plan 状态/路线图，下轮循环自动适配。

> "这就是 Boris Cherny 说的'我的工作是写 Loop'——但前提是 Loop 在已定义的 attractor 内工作。"

#### 支撑机制 3：路线图驱动（阶段性推进）

三个子路线图，由 mission driver 按顺序逐项推进：
```
crud-roadmap（18 域全 done） → core-business-roadmap（M1/M4/M5 done） → extended-roadmap（M2/M3 done）
```
路线图定义了"做什么"的顺序，避免随机开发。

#### 支撑机制 4：审计-整改驱动（强制纠偏）

每隔 2-3 天进行一次综合审计，发现问题后强制整改：
- 07-02：最佳实践合规审计（单代理，D1-D5 整改，引入复审修正机制）
- 07-04：批处理架构审计（nop-job 无 DAG 能力的架构澄清）
- 07-05-1300：三维综合审计（3 路并行子代理，评分 7.3→~8.0）
- 07-05-1400：4 路对抗性跨审查（发现 11 项新问题 C-1~C-11，评分修正为 ~7.0）
- 07-05-1500：工具化合规检查器（`nop-compliance-checker.sh`，15 条 R 规则）
- 07-06：Use Case 实现率审计（189 UC 中 67-78% 完全实现，master-data 最弱 14%；审计源内部摘要与表格数字存在自洽性误差）
- 07-07：综合审计（4 路并行子代理，4 严重 + 6 高整改）
- 07-09：代码质量与文档一致性分析（3 子代理纠正 4 处事实错误）
- 07-12：竞争杠杆兑现审计（3 路并行子代理，发现 3 高风险文档诚信问题）

**关键**：审计不仅发现问题，还**自省审计质量**（07-07 回顾记录了 07-05 审计的评分偏差事件）。

**审计体系本身的演进**（这是一个被 logs 系统性遗漏的重要维度）：

| 阶段 | 审计 | 方法 | 演进点 |
|------|------|------|--------|
| 1 | 07-02-0000 | 单代理 + rg 扫描 + 抽样精读 | 首次系统性合规审计，**引入复审修正机制**（D3/D5 初版错误后修正） |
| 2 | 07-04-0000 | 单代理 + 6 维度逐项证据 | **结束审计**范式（验证产物存在而非业务正确性） |
| 3 | 07-05-1300 | **3 路并行子代理** + 主代理交叉核对 | 首次多代理并行，覆盖手写代码/测试/设计-实现差距三维 |
| 4 | 07-05-1400 | **4 路对抗性子代理（新会话）** + 主代理复核 | **方法论飞跃**：从"方法存在"到"语义保真"——发现 C-2 信用额度多币种语义偏差（4/10 最低分），暴露"扩展域是审计盲区" |
| 5 | 07-05-1500 | 独立 grep + **启发式检测工具** | **工具化**：人工经验固化为可重复运行的 15 条 R 规则 |
| 6 | 07-06 use-case | 单代理 + UC 断言对照 | **粒度切换**：从代码合规到业务实现率 |
| 7 | 07-07-1900 | 4 路并行子代理 | 综合审计 + 自省审计质量偏差 |
| 8 | 07-12-1504 | 3 路并行子代理 + 主代理实时复核 | 竞争杠杆兑现核实（纠正 2 处子代理误判） |

这是 **Harness 层的自进化**——不只是审计代码和设计文档，审计方法论本身也在被审计和改进。07-07 回顾记录的评分偏差事件（07-05 审计曾误判 REQUIRES_NEW 为矛盾）是这一自省能力的典型例证。

**5 个关键演进特征**：
1. **从单代理到多路对抗性子代理**（07-05-1300 三路 → 07-05-1400 四路对抗性）
2. **从"方法存在"到"语义保真"**（07-05-1400 关键转折——信用额度语义保真度 4/10，多币种绕过信用控制）
3. **从人工经验到工具固化**（07-05-1500 `nop-compliance-checker.sh`）
4. **从合规审计到业务实现率审计**（07-06 切换粒度到 UC 级）
5. **子代理"缺失"断言必须由主代理用构建事实复核**（07-12 纠正 2 处误判）

#### 支撑机制 5：测试发现 bug 驱动（回归防护）

- 单元测试（~2900 个）+ Playwright E2E（260+ spec）构成双重防护网。
- 测试发现回归 → 记录到 `docs/bugs/` → 修复 → 验证。
- **测试分层逐步深化**：冒烟层 → 数值断言层 → visual 层，每层能捕获上层无法发现的缺陷（典型：visual 层捕获 `$var` 损坏，冒烟 + 数值层均漏过）。

#### 支撑机制 6：竞品对标驱动（完整性验证）

- `docs/analysis/erp-survey/` 下约 24 个开源项目深度调研（基于源码实测，非营销文案），持续作为功能完整性参照系。
- 10 大主题归纳 + 14 个设计场景首选参考，**设计来源明确标注背书项目**：凭证三件套←赤龙、库存三层←Odoo、异步过账←Metasfresh、多币种双字段←iDempiere、声明式状态机←Tryton。
- 每个新功能引用竞品"标配"作为实现必要性依据。
- 竞争杠杆审计核实 8 个"超越点"是否真正兑现——**4 个完整兑现**（多套账/业财一体三件套/7 种成本方法/AR-AP open-item）、**2 个大部分兑现有缺口**（模型驱动 Delta 文档数据错误/制造委外空壳）、**2 个措辞夸大**（多公司仅结构地基行为层缺失/域独立部署言过其实）。

#### 支撑机制 7：文档一致性驱动（真相源维护）

- ORM XML 是唯一真相源，散文不重复字段定义。
- 文档与代码漂移 → 一致性修复计划（如 07-09 文档-代码一致性修复、07-11 分析报告一致性修复）。
- 跨引用质量 9/10（最佳实践级），带章节锚点的相对路径。

### 4.2 自驱动信息的内部循环

项目建立了一套完整的**内部信息循环系统**，使演化不依赖外部持续输入：

```
                    ┌──────────────────────────────────────┐
                    │                                      │
```

### 5.3 健康度总体判断

**核心健康指标不是"零缺陷"，而是**：
1. 缺陷被发现后**几乎全部在 1-3 天内闭合**
2. 审计会**自我纠偏**（纠正子代理误判、记录评分偏差事件）
3. 既报告问题也报告亮点，避免单一偏差
4. 文档诚信问题（夸大、计数漂移）被独立识别并要求勘误，而非掩盖
5. 平台 bug 被主动发现并反哺 nop-entropy

这是一个**暴露问题而非隐藏问题**的成熟工程文化。

### 5.4 Attractor 合规度（与"方向"的一致性）

三个核心 Attractor 维度在项目各阶段的合规度演进：

| Attractor 维度 | S1 设计期 | S3 核心逻辑 | S5 业财一体 | S8 深化收尾 |
|---------------|-----------|-------------|-------------|-------------|
| ORM XML 唯一真相源 | ✅ 确立 | ✅ 维持 | ✅ 维持 | ✅ 维持 |
| DAG + R/S/P 跨域契约 | ✅ 确立（369 白名单） | ⚠️ 发现 2 处 DAG 违规 | ✅ 修复 | ✅ 全绿 |

**关键发现**：三个 Attractor 维度在 S1 设计期通过竞品调研+架构文档基本确立，但"三轴正交状态分离"直到 S3 核心逻辑期才通过过账引擎设计真正落地。这说明 **Attractor 不是一次定义的，而是随项目演化逐步收敛的**——这也解释了为什么用户需要在 S3 中期还有两次关键 B 类介入（DDD 聚合根重定义 + 架构文档元规则）。

## 七、落地路径——三种项目模式

基于 nop-app-erp 的经验，将 Loop Engineering × Attractor 应用于其他项目的路径：

### 模式 A：已有项目（最小改动）
- **立即生效**：修改 PR 模板，每次变更附加 Plan + 独立审计
- **逐步推进**：先定义一个 Attractor（核心架构文档），再看是否需要 Mission Driver
- **门槛最低**：不改变现有开发流程，在已有流程上叠加 Plan Loop + 审计门控

### 模式 B：新项目（从第一天起全自动）
- **起步**：从 AGE 模板开始（含 Plan Loop + Mission Driver + Attractor 骨架）
- **关键投入**：先花 1-2 天定义 Attractor（不能外包给 AI）
- **运行**：然后运行 Mission Driver 自动编排

### 模式 C：平台/框架项目
- **方向决策权**：Attractor 定义最重要——平台方向决策权不能交给 AI
- **开发控制**：Plan Loop 用于框架自身开发控制
- **优先顺序**：Attractor 先于 Harness

## 八、全自动 AI 开发的适用条件

nop-app-erp 的实证表明，当前技术条件下"全自动 AI 开发"可行，但需满足四个条件：

1. **业务规则由通用最佳实践/公开标准决定**——不依赖特定企业的定制流程
2. **不依赖特定企业的流程偏好或历史包袱**——没有遗留系统的兼容约束
3. **参照系（竞品行为、会计准则、工程模式）公开可获取**——AI 可自行学习和对标
4. **设计决策的正确性可由内在一致性论证**——而非依赖人工经验判断

这些条件恰好定义了一个**"产品化通用软件"的范畴**。超出此范畴的场景（企业专有流程、遗留数据迁移、团队历史偏好等）需要不同程度的人机协作增强。

## 九、可用于 PPT 大纲的核心叙事线

基于以上分析，提炼出几条可选的叙事线（供后续制作大纲时选择）：

### 叙事线 A（主推荐）：Loop Engineering × Attractor——从 In 到 On 的跨越

- **高潮**：知识转移曲线——A 类频次随时间归零、AI 自主度同步上升，两条曲线交叉后用户退场。"用户不是在写代码，而是在转移平台知识。"

- 强调 Model-First、Delta 定制、平台共生演化
- 适合 Nop Platform / 低代码主题

### 叙事线 D：对抗性审计如何保障代码质量
- 强调多层审计体系、自我纠偏、文档诚信
- **核心素材**：审计体系自身的 5 阶段演进（单代理→多路对抗→语义保真→工具化→业务实现率）；07-05-1400 的 4 路对抗性子代理从"方法存在"升级到"语义保真"是方法论飞跃；07-12 纠正 2 处子代理误判的教训
