A 类案例（一）：全局租户开关

AI 写法（错误）

每个实体手动加 useTenant="true"，逐域重复声明

用户校正写法

nop.orm.enable-tenant-by-default 全局配置 + TenantSupport feature 标签，按需 no-tenancy 排除

为什么 AI 不知道

docs-for-ai 当时未记录此全局开关

修复效应

删除所有手写 useTenant，改为全局配置 + 按需排除

模式提炼：配置层隐藏机制（application.yaml + xlib feature）是 docs 经常遗漏的知识。

A 类案例（二）：跨模块引用机制纠偏

AI 的错误推理链

"禁止跨模块引用"

把 notGenCode 外部实体引用当成了 Delta 扩展，误以为是硬规则

漏看的证据

漏看 entity.xdef 注释 + orm-gen.xlib:228 的平台自身用法

用户的正确证据链

notGenCode ≠ Delta

外部实体引用是平台支持的合法机制，平台自身有 9 处内置用法

修复效应

明确 11 个跨域引用实体 + 补充 docs-for-ai

仅 orgId 物理列 + 字典 + 树，无数据隔离/权限/公司间交易
回应：诚实降级声明为"结构地基"，排入后续

风险 3：制造委外曾为空壳

15 行 CRUD
回应：07-13 落地完整委外引擎（生命周期 + MRP + GL 过账）

风险 4：审计方法论改进空间

扩展域在 07-05 前是盲区；子代理断言的复核机制
回应：持续演进中（单代理→多路→语义保真→工具化）

风险 5：logs 与 analysis/discussions 错位

logs 强于"做了什么"，弱于"为什么这么做"
回应：建议增加"重大审计发现登记日志"的约定

不是"我们有这么多问题"，而是"我们知道这些问题、在管理它们"

健康度总结

核心健康指标不是"零缺陷"，而是：

1. 缺陷被发现后几乎全部在 1-3 天内闭合

2. 审计会自我纠偏（纠正子代理误判 + 回顾评分偏差）

3. 同时报告亮点与问题，避免单一偏差

4. 文档诚信问题被独立识别并要求勘误，而非掩盖

5. 平台 bug 被主动发现并反哺 nop-entropy

这是一个暴露问题而非隐藏问题的成熟工程文化

Not zero-defect — defect-aware

三个 Takeaway

Takeaway 1：人机协作的边界

用户 = 平台权威（A 类）

用户 = 工程方向指明（B 类）

用户 ≠ 业务设计者（C 类，让 AI 自己对标开源）

早期密集介入 → 知识转移完成 → 后期放手

不是"AI 自动写代码"，而是"AI 辅助 + 人类定向 + 系统化质量门控"

Takeaway 2：AGE 工作流的有效性

八阶段递进：进入/退出门控明确

强制双审计：草案拦截方向错误 + 结束验证交付完整性

内部信息循环：Backlog → Plan → Audit → Bug → Skill，信息不流失

自我纠偏：评分偏差回顾、子代理误判纠正、方法论持续演进

Takeaway 3：模型驱动的产品化路径

ORM XML 唯一真相源 → codegen → 154 模块全链构建

Delta 定制就绪度 98%（Processor protected 全面开放）

平台共生演化：发现 7 个平台问题 → 修复 nop-entropy → 反哺社区

Nop Platform 证明了自己适合构建产品级 ERP

给观众的建议

技术团队负责人

AI 辅助开发不是"把需求扔给 AI"，而是需要有清晰的边界：什么该你定、什么让 AI 自行决定

投资质量门控（审计、测试、工具）比投资"让 AI 写更多代码"更重要

知识转移曲线：前期投入在平台机制纠正上，后期会指数级回报

Nop 平台用户/潜在用户

Nop 的模型驱动是真的可以构建产品级应用的——不是玩具

Delta 定制是差异化竞争力（98% 场景可纯 Delta 完成）

但 docs-for-ai 有缺口，建议源码级验证 + 社区交流

AI 辅助开发研究者

三层注入模型（A/B/C）可以作为组织级 AI 采用的分析框架

审计体系的 8 阶段演进展示了"AI 辅助开发的质量保障"如何从零建立

内部信息循环系统是可复用的工程模式
