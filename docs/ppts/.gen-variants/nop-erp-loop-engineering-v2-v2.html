从 In 到 On 的关键：Plan

为什么是 Plan？

不是"一段代码"或"一个文件"——Plan 是可独立执行的基本单位

有边界

Goals + Non-Goals → AI 知道做什么、不做什么

有基线

Current Baseline → 从仓库读取，不依赖对话记忆

有退出标准

Exit Criteria → AI 知道"什么时候做完"

可审计

独立子代理可在不继承上下文的情况下验证完成度

可恢复

Plan 状态（draft/active/completed）+ 复选框，中断后扫盘即恢复

Plan Loop 闭环：生成与验收分离——审计必须由独立子代理（fresh session）执行

draft
独立草案审查
active
执行（含验证）
独立结束审计
completed

这套 Plan + 审计 + 验证的联合机制，我们称为 Harness（控制机制）——它是控制轨迹的执行层

实例：Plan 1100-1 销售定价引擎

draft → 审查 iter1（"客户维度 Decision 缺失"，needs_revision）
  → 修订 → 审查 iter2（accept）
    → active → Phase 1-4（ORM + SPI + 测试 + 前端）
      → mvn test 全绿 → 结束审计 PASS → completed

189 份 Plan 的轨迹

每份 Plan 可追溯

为什么做（Source → 路线图）/ 做前状态（Current Baseline）/ 决策记录（Decision + 备选方案）/ 完成状态（Closure Gates + 审计记录）

时间尺度实证

典型 Plan 生命周期 4-12 小时（同一天完成）。最短 ~4h（07-10 的 4 个计划同日完成），最长 ~2 天

没有以分钟计的计划——每条都必须经过审查→执行→验证→审计的门控链

189 个 Plan，X=日期，颜色=状态 | 典型周期 4-12h，最短 ~4h，最长 ~2 天

Mission Driver

在 Plan Loop 之外加一个自动编排层，实现从 In 到 On 的跨越

Mission Driver 的运行实例与 5 步循环

07-10 真实运行记录

08:00 CHECK（mvn 全绿）
08:05 REVIEW_PLANS（4 个 draft → all active）
09:30 EXEC_PLANS（4 个 active Plan 全绿 ✅）
16:00 DRAFT_PLANS（路线图全 done）
16:05 DEEP_AUDIT（自动启动深度审计）

关键设计

REVIEW 在 EXEC 之前（避免重复起草）
EXEC 用 forEach（一次返回后重新扫描）
DRAFT 含自审计
DEEP_AUDIT——全 done 不是停止信号，是升级审计的信号

关于时间

一次完整循环 ~8 小时，整个路线图遍历横跨 22 天

通用 Flow DSL 引擎

三个层级，一个引擎：

注入机制

AI step 的 prompt 含 {{planGuide}} 等占位符，引擎从配置注入。AI 的思考内容来自外部的 plan guide 和 context docs，不来自 prompt 本身

隔离的顶级步骤

顶级步骤相互不传参数。可单独执行任一（--step DEEP_AUDIT），可从任一步开始向后执行，中断后扫盘恢复

零 IPC，全文件

没有 web server，状态通过 Plan/Audit 文件持久化

对人介入的支持：DRAFT_PLANS 持续从路线图起草新 Plan（不限批次）；REVIEW_PLANS 自动拾取任何 Status=draft 的 Plan（不区分来源——手工创建或自动起草都一样）；人可随时修改 Plan 状态/路线图，下轮循环自动适配

人的三类介入

全量抽取 22 天开发周期的日志，得到 28 条明确人类干预 + 80 次设计决策决议

A 类（9 次）

明确指明平台机制
06-22~07-03 密集

用户直接说出平台怎么工作
例："全局租户开关是 nop.orm.enable-tenant-by-default"——AI 查 docs-for-ai 查不到，因为当时没写

B 类（9 次）

指明工程原则方向

用户给原则，不给 API
例："过账方法应挂聚合根 BizModel"——AI 自行裁定凭证=聚合根，零 ORM 变更

C 类（绝大多数，07-04 后为主）

只让 AI 自查

只提要求，不暗示答案
例："对照 erp-survey 检查功能覆盖"

用户不是"AI 监督员"——他是平台知识权威（A 类）+ 方向指南针（B 类），业务逻辑让 AI 对开源自行对标（C 类）

06-29 ██████████ C 类（grill 83 问，仅 4 题需纠正）

用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了

维度一

ORM XML 唯一真相源
所有持久化代码从 XML 生成，手写绕过 = 编译失败

维度二

DAG + R/S/P 跨域契约
模块依赖为 DAG，跨域引用分三类（R 只读桩/S 同步 I*Biz/P 弱关联字符串），369 条白名单

维度三

三轴正交状态分离
docStatus（域状态机）× approveStatus（四态统一）× posted（过账引擎独占）

三者是同一个 Attractor 的三个维度：系统应向声明式、模块隔离、状态正交的结构收敛

状态空间（所有可能状态）→ 吸引子（长期稳定的结构）→ 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

⚠️ 最常见的三类混淆：① 吸引子 ≠ 边界 ② 吸引子 ≠ 更强的护栏 ③ 吸引子 ≠ 控制目标

Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。

三个 Takeaway

Takeaway 1

把"变更"从对话变成契约

Plan Loop 给每次变更一个关闭契约（draft→审查→执行→审计→completed）

这是 Human On The Loop 的前提——AI 自主执行完整一轮，不等人

Takeaway 2

把"编排"从记忆变成自动

Mission Driver 自动遍历路线图、拾取 Plan、审查执行、空闲时审计

Plan 够了吗 → 不够 → Mission Driver 来编排——这是从 In 到 On 的跨越器

Takeaway 3

把"方向"从个人变成文件

Attractor 承载于 owner docs，可版本、可审计、可继承

吸引子不能外包给 AI——定义方向是人的工作

落地路径——三种项目模式

模式 A：已有项目

最小改动

加 Plan Loop：修改 PR 模板，每次变更加 Plan + 独立审计
先定义一个 Attractor（核心架构文档），再看是否需要 Mission Driver

模式 B：新项目

从第一天起 AI 全自动

从 AGE 模板开始（含 Plan Loop + Mission Driver + Attractor 骨架）
先花 1-2 天定义 Attractor（不能外包给 AI）
然后运行 Mission Driver

模式 C：平台/框架

方向决策权不能交给 AI

Attractor 定义最重要——平台方向决策权不能交给 AI
Plan Loop 用于框架自身开发控制
