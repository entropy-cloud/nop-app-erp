行业趋势

Fresh Session 原则——Harness 成败的关键

同一个上下文不能既执行又判断完成——这是 AGE 对 Harness 设计的第一要求。

同一上下文 ❌

过渡

Plan Loop 解决了单次变更怎么关闭。但 189 次变更——谁决定顺序、谁编排、谁在空闲时触发审计？这是 Mission Driver 的工作。

为什么需要更大一层的 Loop

Plan Loop 的局限

Plan Loop 只管理"一次变更"：谁决定下一个？谁确保全路线图被执行？谁空闲时触发审计？

Mission Driver = AGE 框架中 harness 层的自动化实现

在 Plan Loop 之外加一个编排层。时间尺度上是天/周级（遍历整个路线图）：一次完整循环处理 1-7 个计划、耗时 4-12 小时，但整个路线图的流转需要数天到数周。实践中接近业内讨论的自动化编排，但思想根源不同：它自动运行的底气，来自 attractor 已被定义好。

真实运行记录：07-10 一次完整循环

08:00 CHECK（mvn 全绿）
08:05 REVIEW_PLANS（4 个 draft → all active）
09:30 EXEC_PLANS（4 个 active Plan 全绿 ✅）
16:00 DRAFT_PLANS（路线图全 done）
16:05 DEEP_AUDIT（自动启动深度审计）

5 步自动循环，每步都是 AGE harness 层的具体实现

注意看时间尺度：从 CHECK(08:00) 到 DEEP_AUDIT(16:05) 约 8 小时，完成了一次完整的 Mission Driver 循环——处理 4 个计划、经历 5 个步骤。而整个路线图（18 域 + 业财 + 扩展域）从 06-22 到 07-13 横跨 22 天，Mission Driver 在这期间完成了数十次这样的循环。

一次循环 ~8h，路线图横跨 22 天

Mission Driver 的 5 步闭环

第一层（5 步）

CHECK → REVIEW_PLANS → EXEC_PLANS → DRAFT_PLANS → DEEP_AUDIT → 回到 REVIEW_PLANS

第二层 EXEC_PLANS 展开

对每个 active Plan 走执行 → BUILD_VERIFY → CLOSURE_AUDIT

第三层 DRAFT_PLANS 展开

读路线图 → 选工作项 → 起草 Plan → 自审计 → 标记 active

关键设计：REVIEW 在 EXEC 之前 / EXEC 用 forEach / DRAFT 含自审计 / DEEP_AUDIT（全 done 不是停止信号，是升级审计的信号）

"这就是 Boris Cherny 说的'我的工作是写 Loop'——但 Mission Driver 之所以能自动运行，前提是这条 Loop 是在已定义的 attractor 内工作。"

Mission Driver 的实现架构——通用 Flow DSL 引擎

关键设计一：注入机制

AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符，引擎从 mission.json 配置中读取真实路径并注入。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt。

关键设计二：隔离的顶级步骤

CHECK / REVIEW_PLANS / EXEC_PLANS / DRAFT_PLANS / DEEP_AUDIT 相互不传参数。所有状态通过 Plan 文件（> Plan Status: + [x] 复选框）和 Audit 文件持久化到磁盘。因此可以：单独执行任一步骤、从任意步骤开始、恢复不依赖内存。

关键设计三：零 IPC，全文件

没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实。

三个层级，一个引擎

配置层：missions/erp.json → 定义 roadmapPath、planGuide、commands、prompts
流程层：flows/mission-driver.json → 定义 5 步状态机 + 转换条件
执行层：AI step → 启动 opencode 子进程 → 写 Plan/审计文件 → 引擎从日志抠 marker 推进

审计发现 → 闭环

两层 Harness 的关系——以及它们不能做的事

Plan Loop

单次变更控制。每次变更从起草到关闭的完整管理。

Mission Driver Loop

编排控制。自动遍历路线图，决定"下一个做什么"，空闲时自动审计。

但它们不能回答：路线图本身正确吗？"正确的结构"是什么？07-12 竞争杠杆审计发现的"2 个夸大 + 2 个缺口"——不是因为 Harness 没跑好，是因为方向定义层（Attractor）有过问题（文档夸大"多公司 ERP"、制造委外曾是空壳）。Harness 跑得越好，方向错了损失越大。

为什么 Harness 不够

回顾 AGE 的核心洞察：当前所有 Harness 类机制（guardrail、verification、review、feedback loop）都默认一个前提——系统正确的方向是已知的。

传统协作

状态检查 + 程序员的隐性方向感

AI 协作

状态检查全通过 + 无隐性方向感 = 轨迹漂移

核心句

"在 AI 协作中，所有状态层检查都能通过，但系统整体可以在持续漂移。"

实例：nop-chaos-flux 的 Plan 76

测试全部通过，但测试套件本身已与旧实现耦合——从状态视角看一切正常，从轨迹视角看已经在漂移。

动力系统语言——为什么需要 Attractor

状态空间（所有可能状态）→ 吸引子（长期稳定的结构）
  → 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

三个最易混淆的点

吸引子 ≠ 边界

边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。

吸引子 ≠ 更强的护栏

护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。

吸引子 ≠ 控制目标的另一种说法

它是控制的目标因，不在控制之中。反例：团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。

Attractor 的三层结构

结构层（attractor 本身）→ 承载层（工程载体）→ 实现层（瞬时投影）

nop-app-erp 的三个核心 Attractor 实证

吸引子 1：模型驱动

结构不变量：所有实现必须从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者：module-*/model/app-erp-*.orm.xml

验证：mvn clean install 触发 codegen

吸引子 2：状态一致性

结构不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者：docs/design/*/state-machine.md

验证：审计对照状态机 vs 代码实现

吸引子 3：过账契约

结构不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者：docs/design/finance/posting.md

验证：新增业务类型必须注册 + 测试验证

它们不是"设计文档"——设计文档是承载者。真正的吸引子是"所有实体的字段定义必须从 ORM XML 一个源生成"这个结构不变量。文档写错了可以改，结构不变量改了意味着重新 codegen。

人究竟在做什么——定义吸引子

用户的三类介入映射到吸引子的不同层面：

A 类 明确指明平台机制

9 次，06-22 ~ 07-03。修正吸引子的技术基底。例："全局租户开关是 nop.orm.enable-tenant-by-default"

B 类 指明工程原则方向

9 次。修正吸引子的结构原则。例："过账方法应挂聚合根 BizModel"（07-01——用一句话改变了整个过账引擎架构）

C 类 只让 AI 自查

绝大多数，07-04 后为主。在吸引子内让 AI 自行扩张。

"用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 可以在吸引子内自动扩张。这就是 'Attractor Before Harness'——人定义方向，Harness 自动执行扩张和纠偏。"

方向层

健康度——defect-aware 比 zero-defect 更有意义

风险 文档诚信

文档夸大"多公司 ERP"

风险 多组织缺失

风险 委外空壳

制造委外曾是空壳

风险 审计盲区

风险 logs 缺决策记录

核心健康指标：所有缺陷在 1-3 天内闭合 / 审计自纠偏 / 同时报告亮点与问题 / 文档诚信被独立识别并要求勘误 / 主动反哺平台。这不是 Harness 失败——恰恰是 Harness 和 Attractor 在工作的证据。没有 Attractor 定义方向、Harness 检测偏离，这些问题不会被发现。

Not zero-defect — defect-aware（不是零缺陷——缺陷可感知、偏差可见、纠偏可执行）。因为 Attractor 让偏差可见，Harness 让纠偏可执行。

可复制的不是代码，是"Harness + Attractor"结构

22 天 154 模块 → 说"AI 真强"是错误归因。真正可复制的是两层 Harness + Attractor 定义方法的组合：

Plan Loop

Harness：Plan 模板 + fresh session 审计——任何项目都能用

Mission Driver Loop

Harness：自动编排 + 自审计——已包含在 AGE 模板

Attractor 定义方法

owner docs + precedence 体系——不是 Loop，是方向定义

AGE 模板包含这三者的完整实现：cp -r ~/app/attractor-guided-engineering-template/ ./my-project/

落地路径——三种项目模式

模式 A 已有项目

想引入 AI 辅助开发。加 Plan Loop（最小改动）：修改 PR 模板，每次变更加 Plan + 独立审计。先定义一个 Attractor（核心架构文档），再看是否需要 Mission Driver。

模式 B 新项目

从第一天起 AI 全自动。从 AGE 模板开始（含两层 Harness + Attractor 承载骨架）。先花 1-2 天人定义第一个 Attractor（不能外包给 AI）。然后运行 Mission Driver。

模式 C 平台/框架项目

如 nop-entropy。Attractor 定义最重要——平台方向的决策权不能交给 AI。Plan Loop 用于框架自身的开发控制。Mission Driver 可后加。

需人工 三种模式共同前提：人定义吸引子不能外包给 AI

实践建议——给不同角色的核心提取

技术负责人

投资方向定义（Attractor）先于投资自动化（Harness）。方向错了，Loop 跑得越快偏离越远。AGE 说的"吸引子不能外包给 AI"是铁律。

AI 工具使用者

Loop Engineering 的工具成熟度已经够用（Codex/Claude Code 的 automations/worktrees/skills/sub-agents）。但注意工具都是 Harness——你需要自己定义 Attractor。

AI 开发工具构建者

"独立子代理 fresh session 审计"这个 Harness 模式还没被充分工具化。能自动验证"代码是否与 owner docs 一致"的检查器是机会点。

Loop Engineering 让 AI 全自动开发成为可能——但只有 Attractor-Guided Engineering 让全自动开发不偏离方向。

"吸引子定义方向。Harness 控制轨迹。人定义吸引子。AI 在吸引子内扩张。系统检查偏离。三者都在，系统才会自运转。"

"在 AI 时代，真正稀缺的并非更多会写代码的 agent，而是能够率先回答'系统该向哪里收敛'的人。"
