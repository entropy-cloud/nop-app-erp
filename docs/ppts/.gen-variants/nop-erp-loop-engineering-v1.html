基于 Nop Platform · AGE 工作流 · 2026-07-13 基线

最近大家都在聊 Loop Engineering。但如果你仔细看——所有 Loop 都在回答'怎么做'。没人回答'往哪做'。今天我们把两个问题一起回答。

一个让 Loop 失效的问题

问题不在 AI 不聪明，在结构

写完代码的 AI 太擅长给自己的作业打高分——同一个上下文里既制造又检查。天然确认偏差。

传统 Loop 能解决这个问题吗？能——如果你加了 fresh session 独立审计。但更深的问题是：审计的基准是什么？靠什么判断"正确"？

核心问题：做 100 个变更后，系统还在正确的结构上吗？——这需要先定义"正确的结构"是什么

从 Prompt Engineering 到 Loop Engineering 再到 AGE

行业趋势

多位业界实践者（Boris Cherny/Claude Code、Addy Osmani/Google、Andrew Ng/DeepLearning.AI）共同提出同一方向——从手写 prompt 转向设计自动循环系统（Loop）
Cherny："我不再写 prompt 了。我设计的 Loop 在 prompt Claude。"
Osmani："Loop Engineering = 替换你自己作为 prompt 的那个人。"
Andrew Ng 的三层 Loop：Agentic Coding Loop（分钟级）→ Developer Feedback Loop（小时级）→ External Feedback Loop（天/周级）

但都有一个隐含前提："系统正确的方向是已知的。"

Harness 默认为已知方向提供执行保障。AGE 问更深一层：方向已知是谁定义的？靠什么保证方向本身是对的？

AGE 的回答：Loop Engineering 是 harness 层的突破，但需要 attractor 先于 harness。这不是"对 Loop 的补充完善"，而是不同层次的问题——先有吸引子定义"系统应向哪里收敛"，后有 Loop 做"怎么收敛"。

"Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"

Plan Loop 的普适模式

提取为可复用的 Harness 模式：

Plan（起草 + 独立审查）→ Act（执行 Phase）→ Verify（测试/构建/审计）→ 关闭或重试

验证信号层级：编译（最弱）→ 测试 → 独立审计 → 竞争对标（最强）

任何项目都能用的最小 Plan Loop：一次变更加一个 Plan 模板 + 独立子代理审计

过渡："Plan Loop 解决了单次变更怎么关闭。但 189 次变更——谁决定顺序、谁编排、谁在空闲时触发审计？这是 Mission Driver 的工作。"

为什么需要更大一层的 Loop

Plan Loop 只管理"一次变更"：谁决定下一个？谁确保全路线图被执行？谁空闲时触发审计？

全凭人做 → 手动、不可重复。Mission Driver = AGE 框架中 harness 层的自动化实现——在 Plan Loop 之外加一个编排层。

Mission Driver 不是概念创新——它是 AGE harness 理念的自然发展，在 nop-app-erp 中自然发展到自动化形态。时间尺度上是天/周级（遍历整个路线图）：一次完整循环处理 1-7 个计划、耗时 4-12 小时，但整个路线图的流转需要数天到数周（如 crud-roadmap 18 域 + core-business-roadmap M1/M4/M5 + extended-roadmap M2/M3 的完整推进横跨了 06-22 到 07-13）。实践中接近业内讨论的自动化编排，但思想根源不同：它自动运行的底气，来自 attractor 已被定义好。

Mission Driver 的实现架构——通用 Flow DSL 引擎

台上讲：Mission Driver 不是一个定制的"AI 编排脚本"，而是一个通用 Flow DSL 引擎。三个关键设计决定了它的可复用性。引擎底层细节（状态机 marker 抠取、进程模型、断点恢复）放在附录或 speaker notes，台上聚焦三个设计决策即可。

三个层级，一个引擎：

配置层：missions/erp.json → 定义 roadmapPath、planGuide、commands、prompts

流程层：flows/mission-driver.json → 定义 5 步状态机 + 转换条件

执行层：AI step → 启动 opencode 子进程 → 写 Plan/审计文件 → 引擎从日志抠 marker 推进

关键设计一：注入机制

AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符，引擎从 mission.json 配置中读取真实路径并注入。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt。

关键设计二：隔离的顶级步骤

CHECK / REVIEW_PLANS / EXEC_PLANS / DRAFT_PLANS / DEEP_AUDIT 相互不传参数。所有状态通过 Plan 文件（> Plan Status: + [x] 复选框）和 Audit 文件持久化到磁盘。可以单独执行任一步骤、从任意步骤开始向后执行、恢复不依赖内存。

关键设计三：零 IPC，全文件

没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实。

引擎细节（marker 抠取、进程模型、断点恢复）→ 附录 / speaker notes

两层 Harness 的关系——以及它们不能做的事

Plan Loop

单次变更控制。每次变更从起草到关闭的完整管理。

Mission Driver Loop

编排控制。自动遍历路线图，决定"下一个做什么"，空闲时自动审计。

两层 Harness 解决了"怎么执行、怎么编排"——非常有效。

但它们不能回答：路线图本身正确吗？"正确的结构"是什么？

07-12 竞争杠杆审计发现的"2 个夸大 + 2 个缺口"——不是因为 Harness 没跑好，是因为方向定义层（Attractor）有过问题（文档夸大"多公司 ERP"、制造委外曾是空壳）。Harness 跑得越好，方向错了损失越大。

过渡到 Act 4："所以我们需要解决的，不是第三个 Loop——而是一个完全不同性质的东西：结构不变量（Attractor）。"

为什么 Harness 不够

回顾 AGE 的核心洞察：当前所有 Harness 类机制（guardrail、verification、review、feedback loop）都默认一个前提——系统正确的方向是已知的

在 AI 协作中，这个兜底机制被抽走了：AI 不是更快的程序员，而是结构上不同的扰动源——高频、高幅度、无跨 session 方向感、但局部高度合理。

"在 AI 协作中，所有状态层检查都能通过，但系统整体可以在持续漂移。"

nop-chaos-flux 的 Plan 76 实例：测试全部通过，但测试套件本身已与旧实现耦合——从状态视角看一切正常，从轨迹视角看已经在漂移。

传统协作

状态检查 + 程序员的隐性方向感

AI 协作

状态检查全通过 + 无隐性方向感 = 轨迹漂移

动力系统语言——为什么需要 Attractor

AGE 引入动力系统本体论，把轨迹问题从"偶发的修补对象"升级为"一等公民对象"：

状态空间（所有可能状态）→ 吸引子（长期稳定的结构）

→ 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

这不是"换个更优雅的说法"，而是新本体论的基础语言——四个概念相互定义，缺一个其他三个就无法严格表达。

三个最易混淆的点：

≠ 吸引子 ≠ 边界

边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。

≠ 吸引子 ≠ 更强的护栏

护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。

≠ 吸引子 ≠ 控制目标的另一种说法

它是控制的目标因，不在控制之中。没有先定义 attractor，Harness 没有统一含义。反例：完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。

Attractor 的三层结构：

结构层（attractor 本身）：少量高阶不变量——职责怎么分、边界怎么立

→ 承载层（工程载体）：可版本化、可审计的 owner docs

→ 实现层（瞬时投影）：当前代码

关键：attractor 不是文档（文档是承载），attractor 不是代码（代码是投影）。

nop-app-erp 的三个核心 Attractor 实证

吸引子 1：模型驱动

结构不变量：所有实现必须从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者：module-*/model/app-erp-*.orm.xml

验证：mvn clean install 触发 codegen

吸引子 2：状态一致性

结构不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者：docs/design/*/state-machine.md

验证：审计对照状态机 vs 代码实现

吸引子 3：过账契约

结构不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者：docs/design/finance/posting.md

验证：新增业务类型必须注册 + 测试验证

它们不是"设计文档"——设计文档是承载者。真正的吸引子是"所有实体的字段定义必须从 ORM XML 一个源生成"这个结构不变量。文档写错了可以改，结构不变量改了意味着重新 codegen。

健康度——defect-aware 比 zero-defect 更有意义

5 项风险诚实声明（文档诚信、多组织缺失、委外空壳、审计盲区、logs 缺决策记录）。

核心健康指标：所有缺陷在 1-3 天内闭合 / 审计自纠偏 / 同时报告亮点与问题 / 文档诚信被独立识别并要求勘误 / 主动反哺平台。

这不是 Harness 失败——恰恰是 Harness 和 Attractor 在工作的证据。没有 Attractor 定义方向、Harness 检测偏离，这些问题不会被发现。

Not zero-defect — defect-aware（不是零缺陷——缺陷可感知、偏差可见、纠偏可执行）。因为 Attractor 让偏差可见，Harness 让纠偏可执行。

可复制的不是代码，是"Harness + Attractor"结构

22 天 154 模块 → 说"AI 真强"是错误归因。

真正可复制的是两层 Harness + Attractor 定义方法的组合：

  • Plan Loop（Harness：Plan 模板 + fresh session 审计）

    任何项目都能用

  • Mission Driver Loop（Harness：自动编排 + 自审计）

    已包含在 AGE 模板

  • Attractor 定义方法（owner docs + precedence 体系）

    不是 Loop，是方向定义

AGE 模板包含这三者的完整实现：cp -r ~/app/attractor-guided-engineering-template/ ./my-project/

"Loop Engineering 让 AI 全自动开发成为可能——但只有 Attractor-Guided Engineering 让全自动开发不偏离方向。"

"吸引子定义方向。Harness 控制轨迹。人定义吸引子。AI 在吸引子内扩张。系统检查偏离。三者都在，系统才会自运转。"

"在 AI 时代，真正稀缺的并非更多会写代码的 agent，而是能够率先回答'系统该向哪里收敛'的人。"
