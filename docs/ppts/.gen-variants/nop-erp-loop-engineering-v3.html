从 Prompt Engineering 到 Loop Engineering 再到 AGE

  • 行业趋势 前沿实践

    多位业界实践者（Boris Cherny/Claude Code、Addy Osmani/Google、Andrew Ng/DeepLearning.AI）共同提出同一方向——从手写 prompt 转向设计自动循环系统（Loop）
Cherny："我不再写 prompt 了。我设计的 Loop 在 prompt Claude。"
Osmani："Loop Engineering = 替换你自己作为 prompt 的那个人。"
Andrew Ng 的三层 Loop：Agentic Coding Loop（分钟级）→ Developer Feedback Loop（小时级）→ External Feedback Loop（天/周级）

  • 但都有一个隐含前提

    "系统正确的方向是已知的。"
    Harness 默认为已知方向提供执行保障
    AGE 问更深一层：方向已知是谁定义的？靠什么保证方向本身是对的？

  • AGE 的回答

    Loop Engineering 是 harness 层的突破，但需要 attractor 先于 harness。这不是"对 Loop 的补充完善"，而是不同层次的问题——先有吸引子定义"系统应向哪里收敛"，后有 Loop 做"怎么收敛"。
"Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"
——全篇核心句（S24 回扣）

Mission Driver 的实现架构——通用 Flow DSL 引擎

台上讲：Mission Driver 不是一个定制的"AI 编排脚本"，而是一个通用 Flow DSL 引擎。三个关键设计决定了它的可复用性。引擎底层细节（状态机 marker 抠取、进程模型、断点恢复）放在附录或 speaker notes。

  • 三个层级，一个引擎

    配置层：missions/erp.json → 定义 roadmapPath、planGuide、commands、prompts
    流程层：flows/mission-driver.json → 定义 5 步状态机 + 转换条件
    执行层：AI step → 启动 opencode 子进程 → 写 Plan/审计文件 → 引擎从日志抠 marker 推进

  • 关键设计一：注入机制

    AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符，引擎从 mission.json 配置中读取真实路径并注入。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt

  • 关键设计二：隔离的顶级步骤

    CHECK / REVIEW_PLANS / EXEC_PLANS / DRAFT_PLANS / DEEP_AUDIT 相互不传参数。所有状态通过 Plan 文件（> Plan Status: + [x] 复选框）和 Audit 文件持久化到磁盘。因此可以：单独执行任一步骤 / 从任意步骤开始向后执行 / 恢复不依赖内存——扫盘即知"当前在哪"

  • 关键设计三：零 IPC，全文件

    没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实

为什么 Harness 不够

  • AGE 的核心洞察

    当前所有 Harness 类机制（guardrail、verification、review、feedback loop）都默认一个前提——系统正确的方向是已知的

  • 在 AI 协作中，这个兜底机制被抽走了

    AI 不是更快的程序员，而是结构上不同的扰动源——高频、高幅度、无跨 session 方向感、但局部高度合理

  • 核心句

    "在 AI 协作中，所有状态层检查都能通过，但系统整体可以在持续漂移。"

  • nop-chaos-flux 的 Plan 76 实例

    测试全部通过，但测试套件本身已与旧实现耦合——从状态视角看一切正常，从轨迹视角看已经在漂移

传统协作

状态检查 + 程序员的隐性方向感

AI 协作

状态检查全通过 + 无隐性方向感 = 轨迹漂移

动力系统语言——为什么需要 Attractor

  • AGE 引入动力系统本体论

    状态空间（所有可能状态）→ 吸引子（长期稳定的结构） → 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

  • 三个最易混淆的点

吸引子 ≠ 边界——边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。边界抓不住这种偏离
吸引子 ≠ 更强的护栏——护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。护栏视角：每次合规；吸引子视角：方向在漂移
吸引子 ≠ 控制目标的另一种说法——它是控制的目标因，不在控制之中。没有先定义 attractor，Harness 没有统一含义。反例：团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。结果：所有控制机制高效地固化了一套错误基线

Attractor 的三层结构

结构层
attractor 本身：少量高阶不变量——职责怎么分、边界怎么立
承载层
可版本化、可审计的 owner docs
实现层
当前代码（瞬时投影）

关键：attractor 不是文档（文档是承载），attractor 不是代码（代码是投影）

nop-app-erp 的三个核心 Attractor 实证

吸引子 1：模型驱动

结构不变量：所有实现必须从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者：module-*/model/app-erp-*.orm.xml

验证：mvn clean install 触发 codegen

吸引子 2：状态一致性

结构不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者：docs/design/*/state-machine.md

验证：审计对照状态机 vs 代码实现

吸引子 3：过账契约

结构不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者：docs/design/finance/posting.md

验证：新增业务类型必须注册 + 测试验证

它们不是"设计文档"——设计文档是承载者。真正的吸引子是"所有实体的字段定义必须从 ORM XML 一个源生成"这个结构不变量。文档写错了可以改，结构不变量改了意味着重新 codegen

人究竟在做什么——定义吸引子

  • 用户的三类介入 A 类 B 类 C 类

    A 类（明确指明平台机制，9 次，06-22 ~ 07-03）：修正吸引子的技术基底。例："全局租户开关是 nop.orm.enable-tenant-by-default"
    B 类（指明工程原则方向，9 次）：修正吸引子的结构原则。例："过账方法应挂聚合根 BizModel"（07-01——用一句话改变了整个过账引擎架构）
    C 类（只让 AI 自查，绝大多数，07-04 后为主）：在吸引子内让 AI 自行扩张
06-22 ███████████ A 类密集 + B + C 起步
06-29 ██████████ C 类（grill 83 问，仅 4 题需纠正）

用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 可以在吸引子内自动扩张。这就是 'Attractor Before Harness'——人定义方向，Harness 自动执行扩张和纠偏。

落地路径——三种项目模式

模式 A：已有项目

想引入 AI 辅助开发

  • 加 Plan Loop（最小改动）：修改 PR 模板，每次变更加 Plan + 独立审计
  • 先定义一个 Attractor（核心架构文档），再看是否需要 Mission Driver

模式 B：新项目

从第一天起 AI 全自动

  • 从 AGE 模板开始（含两层 Harness + Attractor 承载骨架）
  • 先花 1-2 天人定义第一个 Attractor 不能外包给 AI
  • 然后运行 Mission Driver

模式 C：平台/框架项目

如 nop-entropy

  • Attractor 定义最重要——平台方向的决策权不能交给 AI
  • Plan Loop 用于框架自身的开发控制
  • Mission Driver 可后加

实践建议——给不同角色的核心提取

技术负责人

AI 工具使用者
