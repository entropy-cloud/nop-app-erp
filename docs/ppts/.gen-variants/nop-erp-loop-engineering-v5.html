AGE 框架中 harness 层的自动化实现——在 Plan Loop 之外加一个编排层

Mission Driver 的实现架构——通用 Flow DSL 引擎

关键设计一：注入

AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt

关键设计二：隔离步骤

5 步相互不传参数。所有状态通过 Plan 文件持久化到磁盘。可单独执行任一步骤、从任意步骤开始、恢复不依赖内存——扫盘即知"当前在哪"

关键设计三：零 IPC，全文件

没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实

两层 Harness 的关系——以及它们不能做的事

Plan Loop——单次变更控制

每次变更从起草到关闭的完整管理

Mission Driver Loop——编排控制

自动遍历路线图，决定"下一个做什么"，空闲时自动审计

但它们不能回答：路线图本身正确吗？"正确的结构"是什么？

07-12 竞争杠杆审计发现的"2 个夸大 + 2 个缺口"——不是因为 Harness 没跑好，是因为方向定义层（Attractor）有过问题。Harness 跑得越好，方向错了损失越大

为什么 Harness 不够

AGE 的核心洞察

当前所有 Harness 类机制（guardrail、verification、review、feedback loop）都默认一个前提——系统正确的方向是已知的

"在 AI 协作中，所有状态层检查都能通过，但系统整体可以在持续漂移。"

传统协作

状态检查 + 程序员的隐性方向感

→ 稳健

AI 协作

状态检查全通过 + 无隐性方向感

= 轨迹漂移 ❌

实例：nop-chaos-flux 的 Plan 76——测试全部通过，但测试套件本身已与旧实现耦合。从状态视角看一切正常，从轨迹视角看已经在漂移

动力系统语言——为什么需要 Attractor

AGE 引入动力系统本体论：

状态空间（所有可能状态）→ 吸引子（长期稳定的结构）
  → 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

这不是"换个更优雅的说法"——四个概念相互定义，缺一个其他三个就无法严格表达。用旧本体论的语言（架构、约束、目标、流程）翻译，必然产生信息损失

三个最易混淆的点：

吸引子 ≠ 边界

边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。边界抓不住这种偏离

吸引子 ≠ 更强的护栏

护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。护栏视角：每次合规；吸引子视角：方向在漂移

吸引子 ≠ 控制目标的另一种说法

它是控制的目标因，不在控制之中。没有先定义 attractor，Harness 没有统一含义。反例：团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。结果：所有控制机制高效地固化了一套错误基线

nop-app-erp 的三个核心 Attractor 实证

吸引子 1：模型驱动

结构不变量：所有实现必须从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者 module-*/model/app-erp-*.orm.xml

验证 mvn clean install 触发 codegen

吸引子 2：状态一致性

结构不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者 docs/design/*/state-machine.md

验证 审计对照状态机 vs 代码实现

吸引子 3：过账契约

结构不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者 docs/design/finance/posting.md

验证 新增业务类型必须注册 + 测试验证

它们不是"设计文档"——设计文档是承载者。真正的吸引子是"所有实体的字段定义必须从 ORM XML 一个源生成"这个结构不变量。文档写错了可以改，结构不变量改了意味着重新 codegen

人究竟在做什么——定义吸引子

三类介入映射：

A 类 — 明确指明平台机制（9 次，06-22 ~ 07-03）

修正吸引子的技术基底。例："全局租户开关是 nop.orm.enable-tenant-by-default"

B 类 — 指明工程原则方向（9 次）

修正吸引子的结构原则。例："过账方法应挂聚合根 BizModel"（07-01——用一句话改变了整个过账引擎架构）

C 类 — 只让 AI 自查（绝大多数，07-04 后为主）

在吸引子内让 AI 自行扩张

知识转移曲线

"用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 可以在吸引子内自动扩张。这就是 'Attractor Before Harness'——人定义方向，Harness 自动执行扩张和纠偏。"

完整架构：两层 Harness + 三个 Attractor

方向层（不是 Loop，是结构不变量）：
三个 Attractor → 定义"系统应向哪里收敛"
    ↓
Harness 层（控制 Loop）：
内层：Plan Loop（小时级）→ 每次变更的关闭契约
外层：Mission Driver Loop（天/周级）→ 自动编排 + 自审计
    ↑
两者关系：Harness 在 Attractor 内运行，Attractor 为 Harness 提供方向目标

与 Andrew Ng 框架的关系

Andrew Ng 的三层全是反馈控制 Loop——它们属于 Harness 层的细分。AGE 的贡献不在 Loop 层，而在发现了 Loop 的前提——方向定义层（Attractor）

不是"AGE 比 Loop Engineering 多一层"，而是"Loop Engineering 在 Harness 层面上的实践，需要在 Attractor 层面上补充方向"

"Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"

健康度——defect-aware 比 zero-defect 更有意义

5 项风险诚实声明

文档诚信 · 多组织缺失 · 委外空壳 · 审计盲区 · logs 缺决策记录

核心健康指标

所有缺陷在 1-3 天内闭合 / 审计自纠偏 / 同时报告亮点与问题 / 文档诚信被独立识别并要求勘误 / 主动反哺平台
