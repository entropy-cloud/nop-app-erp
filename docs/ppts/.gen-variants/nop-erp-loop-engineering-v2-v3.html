Vibe Coding = Human In The Loop

  • 现在的主流 AI 编程模式：用户说 → AI 写 → 用户读 → 用户再说 → AI 再写 → ...

  • 人在循环内部：每一轮迭代都需要人的注意力才能推进。没有人的指令，AI 不会进入下一轮

  • 人是循环的节拍器——循环的节奏由人的注意力决定

  • 这不是"坏"的——它适合快速原型、单文件脚本、局部修复

  • 但对于产品级系统（352 实体、154 模块、~2890 测试），这种模式不可持续

  • 核心问题：这是一个不可关闭的循环——没有基线、没有范围、没有退出标准、没有审计

从 In 到 On 的关键：Plan

  • Human On The Loop 需要 AI 能自主执行完整的一轮，而不是每步等人

  • 这就要求一个可独立执行的基本单位——Plan

  • 为什么是 Plan，不是"一段代码"或"一个文件"？

    有边界（Goals + Non-Goals）→ AI 知道做什么、不做什么
    有基线（Current Baseline）→ 从仓库读取，不依赖对话记忆
    有退出标准（Exit Criteria）→ AI 知道"什么时候做完"
    可审计 → 独立子代理可在不继承上下文的情况下验证完成度
    可恢复 → Plan 状态（draft/active/completed）+ 复选框，中断后扫盘即恢复

生成与验收分离：审计必须由独立子代理（fresh session）执行，不继承执行上下文

这套 Plan + 审计 + 验证的联合机制，我们称为 Harness（控制机制）——它是控制轨迹的执行层

实例：Plan 1100-1 销售定价引擎

draft → 审查 iter1（"客户维度 Decision 缺失"，needs_revision）
    → 修订 → 审查 iter2（accept）
        → active → Phase 1-4（ORM + SPI + 测试 + 前端）

  • 如果没 Plan Loop：这 4 个缺陷在实施/运行期才暴露，修复成本高 5-10 倍

通用 Flow DSL 引擎

配置层：missions/erp.json
  → roadmapPath、planGuide、commands
流程层：flows/mission-driver.json
  → 5 步状态机
执行层：AI step
  → spawn opencode run
  → 写 Plan/审计文件
  → 引擎抠 marker 推进

  • 关键设计一：注入机制

    AI step 的 prompt 含 {{planGuide}} 等占位符，引擎从配置注入。AI 的思考内容来自外部的 plan guide 和 context docs，不来自 prompt 本身

  • 关键设计二：隔离的顶级步骤

    顶级步骤相互不传参数。可单独执行任一（--step DEEP_AUDIT），可从任一步开始向后执行，中断后扫盘恢复

  • 关键设计三：零 IPC，全文件

    没有 web server，状态通过 Plan/Audit 文件持久化

对人介入的支持：DRAFT_PLANS 持续从路线图起草新 Plan（不限批次）；REVIEW_PLANS 自动拾取任何 Status=draft 的 Plan（不区分来源——手工创建或自动起草都一样）；人可随时修改 Plan 状态/路线图，下轮循环自动适配

用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了

状态空间（所有可能状态）→ 吸引子（长期稳定的结构）
    → 轨迹（实际演化路径）→ 控制（将轨迹拉回吸引子的机制）

吸引子不是"多一层 Loop"——它是所有 Loop 的前提。没有方向，Harness 没有统一含义

维度一：ORM XML 唯一真相源

数据空间——所有持久化代码从 XML 生成，手写绕过 = 编译失败

维度二：DAG + R/S/P 跨域契约

模块空间——模块依赖为 DAG，跨域引用分三类（R 只读桩/S 同步 I*Biz/P 弱关联字符串），369 条白名单

维度三：三轴正交状态分离

业务状态空间——docStatus（域状态机）× approveStatus（四态统一）× posted（过账引擎独占）

三者是同一个 Attractor 的三个维度：系统应向声明式、模块隔离、状态正交的结构收敛

最常见的三类混淆：

  • 吸引子 ≠ 边界

    边界禁止行为，吸引子定义汇聚目标。例：规定"不准用 JPA"（边界），AI 用 JDBC 写 SQL——没违反边界但破坏了 ORM 唯一真相源

  • 吸引子 ≠ 更强的护栏

    护栏检查每次动作，吸引子评估多次累积。例：10 次 PR 代码规范都通过，但测试套件已与旧实现耦合

  • 吸引子 ≠ 控制目标

    它是控制的前提，不在控制之中。例：完美的 CI/CD + review 门禁，但没人定义过架构方向→所有控制高效地固化了错误基线

三个 Takeaway

Takeaway 1：把"变更"从对话变成契约

Plan Loop 给每次变更一个关闭契约（draft→审查→执行→审计→completed）

这是 Human On The Loop 的前提——AI 自主执行完整一轮，不等人

Takeaway 2：把"编排"从记忆变成自动

Mission Driver 自动遍历路线图、拾取 Plan、审查执行、空闲时审计

Plan 够了吗 → 不够 → Mission Driver 来编排——这是从 In 到 On 的跨越器

Takeaway 3：把"方向"从个人变成文件

Attractor 承载于 owner docs，可版本、可审计、可继承

吸引子不能外包给 AI——定义方向是人的工作
