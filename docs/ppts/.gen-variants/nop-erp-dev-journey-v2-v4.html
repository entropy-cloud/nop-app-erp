外部注入三层模型

  • A 类：给机制名 + 配置项 + 源码位置

    9 次，全在 06-22~07-03。AI 自由度极低。

  • B 类：给原则或质疑，不给 API

    9 次，散布全程。AI 中等自由度。

  • C 类：只提要求，不暗示答案

    绝大多数，尤其 07-04 后。AI 高自由度。

关键洞察：用户有清晰边界意识——平台机制明确介入，业务逻辑让 AI 对标开源自查。

文档诚信事件：一个深度案例

发现的问题

  • 实体计数不一致

    设计文档 145 → 架构文档 447 → 实际 352 自有 + 110 引用桩

  • "10 份 orm.xml" → 实为 19 份

  • "多公司 ERP" → 仅结构地基

我们的回应

不掩盖
独立审计识别
要求勘误
整改计划 1504-1 专门修正

最强的不是"零缺陷"，而是 "暴露问题而非隐藏问题" 的工程文化。

5 项风险 · 诚实声明

  • 文档诚信（最突出）

    计数漂移 / "多公司"夸大 → 审计识别 + 勘误 + 整改计划

    已响应

  • 多组织行为层缺失

    仅 orgId 列 + 字典 + 树 → 诚实降级为"结构地基"，排入后续

    未完全解决

  • 制造委外曾空壳

    07-13 落地完整委外引擎（生命周期 + MRP + GL 过账）

    已解决

  • 审计方法论改进空间

    扩展域曾是盲区；持续演进（单代理→多路→语义保真→工具化）

    持续改进

  • logs 弱于"为什么"

    强于"做了什么"，弱于决策依据 → 建议增"重大审计发现登记"约定

    持续改进

不是"我们有很多问题"，而是"我们知道并在管理它们"。

defect-aware：健康度总结

  • 缺陷发现后几乎全在 1-3 天内闭合

  • 审计会自我纠偏（纠正子代理误判 + 回顾评分偏差）

  • 同时报告亮点与问题，避免单一偏差

  • 文档诚信问题被独立识别并要求勘误，而非掩盖

  • 平台 bug 被主动发现并反哺 nop-entropy

Not zero-defect — defect-aware.

给不同角色的一句建议

  • 技术团队负责人

    投资质量门控（审计/测试/工具）比投资"让 AI 写更多代码"回报更高；知识转移前期投入，后期指数回报。

  • Nop 平台用户 / 潜在用户

    模型驱动能构建产品级应用（非玩具），Delta 定制是差异化竞争力（~98% 纯 Delta）；但 docs-for-ai 有缺口，建议源码级验证。

  • AI 辅助开发研究者

    三层注入模型（A/B/C）可作组织级 AI 采用的分析框架；审计 8 阶段演进 + 内部信息循环是可复用工程模式。
