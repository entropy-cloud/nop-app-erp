预告答案：他在做三件事，且随时间主动退场。

外部注入三层模型

  • A 类 — 明确指明 Nop 平台机制

    给机制名 + 配置项 + 源码位置。AI 自由度极低。9 次，全在 06-22∼07-03

  • B 类 — 指明工程/设计原则方向

    给原则或质疑，不给 API。AI 中等自由度。9 次，散布全程

  • C 类 — 只让 AI 自查/对比

    只提要求，不暗示答案。AI 高自由度。绝大多数，尤其 07-04 后

关键洞察：用户有清晰边界意识——平台机制明确介入，业务逻辑让 AI 对标开源自查。

A 类案例：用户当"平台知识权威"

案例一：全局租户开关

AI 写法：每个实体手写 useTenant="true"

用户校正：nop.orm.enable-tenant-by-default 全局配置 + TenantSupport feature 标签

AI 查不到，因 docs-for-ai 当时未记录此全局开关。

案例二：跨模块引用机制

AI 写法：误以为"禁止跨模块引用"是硬规则

用户校正：notGenCode 外部实体引用 ≠ Delta 扩展，平台自身 9 处内置用法

AI 漏看 entity.xdef 注释 + orm-gen.xlib:228。

共同模式：A 类全部命中 docs-for-ai 未覆盖或覆盖不全的平台细节（配置层/源码层隐藏机制）。

最强的不是"零缺陷"，而是"暴露问题而非隐藏问题"的工程文化。

5 项风险 · 诚实声明

  • 1. 文档诚信（最突出）已识别

    计数漂移 / "多公司"夸大 → 审计识别 + 勘误 + 整改计划

  • 2. 多组织行为层缺失诚实降级

    仅 orgId 列 + 字典 + 树，无隔离/权限/公司间交易 → 诚实降级为"结构地基"，排入后续

  • 3. 制造委外曾空壳已响应

    15 行 CRUD → 07-13 落地完整委外引擎（生命周期 + MRP + GL 过账）

  • 4. 审计方法论改进空间持续演进

    扩展域曾是盲区；子代理断言需复核 → 单代理→多路→语义保真→工具化

  • 5. logs 弱于"为什么"已知

    强于"做了什么"，弱于决策依据 → 建议增"重大审计发现登记"约定

不是"我们有很多问题"，而是"我们知道并在管理它们"。

健康度总结

  • 缺陷发现后几乎全在 1-3 天内闭合

  • 审计会自我纠偏（纠正子代理误判 + 回顾评分偏差）

  • 同时报告亮点与问题，避免单一偏差

  • 文档诚信问题被独立识别并要求勘误，而非掩盖

  • 平台 bug 被主动发现并反哺 nop-entropy

Not zero-defect — defect-aware.

三个 Takeaway

Takeaway 1 · 人机协作有边界

用户 = 平台权威（A 类）/ 方向指南（B 类）/ 非业务设计者（C 类）

早期密集介入 → 知识转移完成 → 后期放手

八阶段递进，进入条件 + 退出门控明确

Takeaway 3 · 模型驱动可行

ORM XML 唯一真相源 → codegen → 154 模块全链构建

Delta 定制就绪度 98% / 平台共生反哺

给不同角色的一句建议

  • 技术团队负责人

    投资质量门控（审计/测试/工具）比投资"让 AI 写更多代码"回报更高；知识转移前期投入，后期指数回报。

  • Nop 平台用户 / 潜在用户

    模型驱动能构建产品级应用（非玩具），Delta 定制是差异化竞争力（~98% 纯 Delta）；但 docs-for-ai 有缺口，建议源码级验证。

  • AI 辅助开发研究者

    三层注入模型（A/B/C）可作组织级 AI 采用的分析框架；审计 8 阶段演进 + 内部信息循环是可复用工程模式。
