一个让 Loop 失效的问题

  • 开场故事

  • 核心问题

传统 Loop 能解决吗？ 能——如果你加了 fresh session 独立审计。但更深的问题是：审计的基准是什么？靠什么判断"正确"？

从 Prompt Engineering 到 Loop Engineering 再到 AGE

  • 行业趋势

    多位业界实践者（Boris Cherny/Claude Code、Addy Osmani/Google、Andrew Ng/DeepLearning.AI）共同提出同一方向——从手写 prompt 转向设计自动循环系统（Loop）
    Cherny："我不再写 prompt 了。我设计的 Loop 在 prompt Claude。"
    Osmani："Loop Engineering = 替换你自己作为 prompt 的那个人。"
    Andrew Ng 的三层 Loop：Agentic Coding Loop（分钟级）→ Developer Feedback Loop（小时级）→ External Feedback Loop（天/周级）

  • 隐含前提

    "系统正确的方向是已知的。" Harness 默认为已知方向提供执行保障。AGE 问更深一层：方向已知是谁定义的？靠什么保证方向本身是对的？

"Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"

没有 Loop 会怎样

  • 传统 AI 开发

    用户说 → AI 写 → 用户改 → AI 再写 → ... ∞ 无退出条件

四大缺失

  • 无基线

    不知道从哪里开始，不知道当前状态

  • 无范围

    不知道边界在哪里，不知道什么时候该停

  • 无退出标准

    没有定义"完成"意味着什么

  • 无审计

    没有独立验证，自写自检

Plan 作为关闭契约

  • Plan 不是任务清单，是关闭契约

    Current Baseline / Goals + Non-Goals / Exit Criteria / Closure Gates / Draft Review Record

  • Plan Loop 闭环

    draft → 独立草案审查（fresh session）→ active → 执行 Phase（含验证）→ 独立结束审计（fresh session）→ completed

  • 独立子代理 fresh session

    不继承执行上下文，冷启动重查仓库——这是 Harness 的核心价值：生成与验收分离

189 份 Plan 的轨迹可追溯性

  • 每份 Plan 可追溯

    为什么做 → Source 指向路线图 / 做前状态 → Current Baseline / 决策记录 → Decision + 备选方案 + 被否原因 / 完成状态 → Closure Gates + 独立审计记录

  • 轨迹信息外部化

    关键决策在文件里，不在临时对话里

  • 时间尺度实证

    典型 Plan 生命周期 4-12 小时（同一天内完成）。最短约 4-6 小时（07-10 的 4 个计划：销售定价/信用冻结/到岸成本/预算管理同日完成），最长约 2 天（跨域 FK 名称解析接力）。没有以分钟计的计划——每条都必须经过 draft → 独立审查（1-2 轮）→ execute（含 build 验证 ~10 min）→ closure audit 的门控链

为什么需要更大一层的 Loop

  • Plan Loop 只管理"一次变更"

    谁决定下一个？谁确保全路线图被执行？谁空闲时触发审计？

  • Mission Driver = AGE 框架中 harness 层的自动化实现

    在 Plan Loop 之外加一个编排层。全凭人做 → 手动、不可重复。

  • 时间尺度：天/周级

    一次完整循环处理 1-7 个计划、耗时 4-12 小时。整个路线图的流转需要数天到数周。它自动运行的底气，来自 attractor 已被定义好

关键设计

REVIEW 在 EXEC 之前 / EXEC 用 forEach / DRAFT 含自审计 / DEEP_AUDIT（全 done 不是停止信号，是升级审计的信号）

Cherny 的 Loop

"我的工作是写 Loop"——但 Mission Driver 之所以能自动运行，前提是这条 Loop 是在已定义的 attractor 内工作

Mission Driver 的实现架构——通用 Flow DSL 引擎

台上讲：Mission Driver 不是一个定制的"AI 编排脚本"，而是一个通用 Flow DSL 引擎。三个关键设计决定了它的可复用性。引擎底层细节（状态机 marker 抠取、进程模型、断点恢复）放在附录或 speaker notes，台上聚焦三个设计决策即可。

关键设计一：注入机制

AI step 的 prompt 模板含 {{planGuide}}、{{roadmapPath}} 等占位符，引擎从 mission.json 配置中读取真实路径并注入。AI 的思考内容不来自 prompt，而来自外部注入的 plan guide、roadmap、context docs。换项目只需改配置，不改 prompt

关键设计二：隔离的顶级步骤

CHECK / REVIEW_PLANS / EXEC_PLANS / DRAFT_PLANS / DEEP_AUDIT 相互不传参数。所有状态通过 Plan 文件（> Plan Status: + [x] 复选框）和 Audit 文件持久化到磁盘。因此可以：单独执行任一步骤、从任意步骤开始向后执行、恢复不依赖内存——扫盘即知"当前在哪"

关键设计三：零 IPC，全文件

没有 IPC、没有 web server。Mission Driver 主进程只做状态机推进，AI 通过文件 I/O 交互。这是 AGE "仓库 = 唯一真相源"原则在工具层的落实

DEEP_AUDIT——Harness 能自省

  • 07-07 综合审计

    4 路并行子代理 → 4 严重 + 6 高 → 1-3 天全部闭合

两层 Harness 的关系——以及它们不能做的事

Plan Loop

单次变更控制。每次变更从起草到关闭的完整管理

Mission Driver Loop

编排控制。自动遍历路线图，决定"下一个做什么"，空闲时自动审计

但它们不能回答：路线图本身正确吗？"正确的结构"是什么？

07-12 竞争杠杆审计发现的"2 个夸大 + 2 个缺口"——不是因为 Harness 没跑好，是因为方向定义层（Attractor）有过问题（文档夸大"多公司 ERP"、制造委外曾是空壳）。Harness 跑得越好，方向错了损失越大。

过渡到 Act 4："所以我们需要解决的，不是第三个 Loop——而是一个完全不同性质的东西：结构不变量（Attractor）。"

为什么 Harness 不够

  • AGE 核心洞察

    当前所有 Harness 类机制（guardrail、verification、review、feedback loop）都默认一个前提——系统正确的方向是已知的

  • AI 是结构上不同的扰动源

    AI 不是更快的程序员，而是结构上不同的扰动源——高频、高幅度、无跨 session 方向感、但局部高度合理

  • 核心句

    "在 AI 协作中，所有状态层检查都能通过，但系统整体可以在持续漂移。"

  • nop-chaos-flux 的 Plan 76 实例

    测试全部通过，但测试套件本身已与旧实现耦合——从状态视角看一切正常，从轨迹视角看已经在漂移

动力系统语言——为什么需要 Attractor

混淆 1：吸引子 ≠ 边界

边界回答"什么不能做"，违反即报错。吸引子回答"系统应向哪里长期收敛"，单次偏离不报错但持续偏离让结构腐烂。反例：规定"不准用 JPA"（边界），AI 绕过用 JDBC 写 SQL——没违反边界但破坏了"ORM 模型唯一真相源"这个吸引子。边界抓不住这种偏离。

混淆 2：吸引子 ≠ 更强的护栏

护栏在执行层检查每次动作，吸引子在方向层评估多次动作累积是否靠拢。反例：10 次 PR 的代码规范都通过（护栏），但测试套件已与旧实现深度耦合。护栏视角：每次合规；吸引子视角：方向在漂移。

混淆 3：吸引子 ≠ 控制目标

它是控制的目标因，不在控制之中。没有先定义 attractor，Harness 没有统一含义。反例：团队有完美的 CI/CD + code review + 测试门禁，但没人回答过"系统该向什么架构收敛"。结果：所有控制机制高效地固化了一套错误基线。

nop-app-erp 的三个核心 Attractor 实证

吸引子 1：模型驱动

结构不变量：所有实现必须从 ORM XML 生成，不允许手写绕过的 Entity/DAO

承载者：module-*/model/app-erp-*.orm.xml
验证：mvn clean install 触发 codegen

吸引子 2：状态一致性

结构不变量：每域的审批/过账/状态迁移必须与状态机一致

承载者：docs/design/*/state-machine.md
验证：审计对照状态机 vs 代码实现

吸引子 3：过账契约

结构不变量：所有过账走同一 SPI 契约，红字用同向取负约定

承载者：docs/design/finance/posting.md
验证：新增业务类型必须注册 + 测试验证

它们不是"设计文档"——设计文档是承载者。真正的吸引子是"所有实体的字段定义必须从 ORM XML 一个源生成"这个结构不变量。文档写错了可以改，结构不变量改了意味着重新 codegen。

人究竟在做什么——定义吸引子

  • A 类（明确指明平台机制，9 次，06-22 ~ 07-03）

    修正吸引子的技术基底。例："全局租户开关是 nop.orm.enable-tenant-by-default"

  • B 类（指明工程原则方向，9 次）

    修正吸引子的结构原则。例："过账方法应挂聚合根 BizModel"（07-01——用一句话改变了整个过账引擎架构）

  • C 类（只让 AI 自查，绝大多数，07-04 后为主）

    在吸引子内让 AI 自行扩张

"用户后期介入归零不是因为 AI 学会了写代码，而是因为吸引子已经定义好了。方向对了，AI 可以在吸引子内自动扩张。这就是 'Attractor Before Harness'——人定义方向，Harness 自动执行扩张和纠偏。"

"不能把定义新 attractor 的责任默认外包给 AI。" ——AGE

完整架构：两层 Harness + 三个 Attractor

  • 方向层（不是 Loop，是结构不变量）

    三个 Attractor → 定义"系统应向哪里收敛"

  • Harness 层（控制 Loop）

    内层：Plan Loop（小时级）→ 每次变更的关闭契约
    外层：Mission Driver Loop（天/周级）→ 自动编排 + 自审计

与 Andrew Ng 框架的关系：

Andrew Ng 的三层全是反馈控制 Loop——它们属于 Harness 层的细分。AGE 的贡献不在 Loop 层，而在发现了 Loop 的前提——方向定义层（Attractor）。不是"AGE 比 Loop Engineering 多一层"，而是"Loop Engineering 在 Harness 层面上的实践，需要在 Attractor 层面上补充方向"。

"Loop Engineering 让 AI 跑得更快更稳。Attractor 让 Loop 知道往哪里跑。前者是实践突破，后者是哲学基础。两者都需要，但不能混淆。"

  • Plan Loop（Harness：Plan 模板 + fresh session 审计）

  • Mission Driver Loop（Harness：自动编排 + 自审计）

落地路径——三种项目模式

模式 A：已有项目

想引入 AI 辅助开发

加 Plan Loop（最小改动）：修改 PR 模板，每次变更加 Plan + 独立审计
先定义一个 Attractor（核心架构文档），再看是否需要 Mission Driver

模式 B：新项目

从第一天起 AI 全自动

从 AGE 模板开始（含两层 Harness + Attractor 承载骨架）
先花 1-2 天人定义第一个 Attractor（不能外包给 AI）
然后运行 Mission Driver

模式 C：平台/框架项目

如 nop-entropy

Attractor 定义最重要——平台方向的决策权不能交给 AI
Plan Loop 用于框架自身的开发控制
Mission Driver 可后加

三条路径的共同起点：先定义 Attractor，再运行 Harness。人定义方向，AI 自动执行。

实践建议——给不同角色的核心提取

技术负责人

投资方向定义（Attractor）先于投资自动化（Harness）。方向错了，Loop 跑得越快偏离越远。AGE 说的"吸引子不能外包给 AI"是铁律。

AI 工具使用者

Loop Engineering 的工具成熟度已经够用（Codex/Claude Code 的 automations/worktrees/skills/sub-agents）。但注意工具都是 Harness——你需要自己定义 Attractor。

AI 开发工具构建者

"独立子代理 fresh session 审计"这个 Harness 模式还没被充分工具化。能自动验证"代码是否与 owner docs 一致"的检查器是机会点。

吸引子定义方向。Harness 控制轨迹。人定义吸引子。AI 在吸引子内扩张。系统检查偏离。三者都在，系统才会自运转。

结尾：从 Loop 到 Attractor——这条路走通了

"Loop 让 AI 跑得更快。Attractor 让 Loop 知道往哪跑。"
