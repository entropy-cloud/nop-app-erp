# nop-app-erp 开发历程 PPT 大纲 v2（1 小时版）

> 用途：逐页 PPT 结构和内容指导，每页标注建议视觉类型、预估时间与讲者过渡语。
> 关系：本文件是 `nop-erp-dev-journey-ppt-outline.md`（v1）的重写版本，v1 保持不变。
> 素材来源：同 `nop-erp-dev-journey-base-material.md`。
>
> **v2 设计原则（相对 v1 的关键变更）**
> 1. **单一命题（thesis-driven）**：全篇一句主张贯穿——"AI 辅助开发的本质 = 知识转移 + 系统化质量门控"。第 2 页点题，知识转移曲线处兑现，结尾回扣。
> 2. **去重合并**：v1 中审计出现在 6+ 处、竞品对标出现在 4 处、"8 阶段"框架用了两次。v2 把审计收敛为一个专门幕（Act 4），竞品对标收敛为一处主讲 + 一处兑现核实，时间线 8 阶段与审计 8 阶段明确区分。
> 3. **提前抛钩子**：把最强的"知识转移曲线"在开场埋一句悬念，Act 2 高潮兑现。
> 4. **压缩规模**：34 页正文，目标 55 分钟 + 5 分钟 Q&A（v1 的 42 页在 60 分钟内偏挤）。
> 5. **数字对齐诚信主题**：全篇采用经审计校准的口径（187 计划 + 2 指南；352 自有实体 + 110 引用桩；154 模块；~2890 Java 测试；260 E2E），并把"数字漂移被独立识别"作为诚信文化的正面证据讲，而非回避。
> 6. **诚信内容合并**：v1 的 P5-4（文档诚信）与 Part 7（健康度）合并为 Act 4 的高潮与收束，避免两处重复讲同一批事实。
>
> 叙事线：时间骨架（22 天）为脊柱 × 单一命题（知识转移 + 质量门控）为灵魂，审计文化与平台共生为两翼。
> 总页数：34 页正文 + 附录，建议 60 分钟（含 Q&A 预留 5 分钟）。

## Act 2：命题一 —— 人机协作的边界（6 页 / 13 min）

### S08 一个反直觉的问题（1 min）
- 抛问：整个过程里，用户（人）写了多少代码？——**几乎为零**。那他做了什么？
- 预告答案：他在做三件事，且随时间**主动退场**。
- **视觉**：留白 + 一行大字问句 + 三个待揭示的空占位

### S09 外部注入三层模型（2 min）
- 按“介入的具体程度”把用户每次介入分三层：
  - **A 类（明确指明 nop 平台机制）**：给机制名 + 配置项 + 源码位置。AI 自由度极低。→ 9 次，全在 06-22~07-03
  - **B 类（指明工程/设计原则方向）**：给原则或质疑，不给 API。AI 中等自由度。→ 9 次，散布全程
  - **C 类（只让 AI 自查/对比）**：只提要求，不暗示答案。AI 高自由度。→ 绝大多数，尤其 07-04 后
- **关键洞察**：用户有清晰边界意识——**平台机制明确介入，业务逻辑让 AI 对标开源自查**。
- **视觉**：三层金字塔，层高=占比（C 最厚），每层 1 例

### S10 A 类案例：用户当“平台知识权威”（2.5 min）
- **案例一 · 全局租户开关**：AI 给每个实体手写 `useTenant="true"` → 用户点破 `nop.orm.enable-tenant-by-default` 全局配置 + `TenantSupport` feature 标签。AI 查不到，因 docs-for-ai 当时未记录此全局开关。
- **案例二 · 跨模块引用机制**：AI 误以为“禁止跨模块引用”是硬规则 → 用户纠正 `notGenCode` 外部实体引用 ≠ Delta 扩展，平台自身 9 处内置用法。AI 漏看了 `entity.xdef` 注释 + `orm-gen.xlib:228`。
- **共同模式**：A 类全部命中 **docs-for-ai 未覆盖或覆盖不全的平台细节**（配置层/源码层隐藏机制）。
- **视觉**：左右双栏“AI 写法 vs 用户校正写法”，两案例各一组，红蓝对比

### S11 B 类案例：用户当“方向指南针”（1.5 min）
- 两次关键 B 类（07-01），决定工程范式：
  - **过账引擎 DDD 聚合根原则**：“产生记录应有主实体，方法挂聚合根 BizModel” → AI 自行裁定凭证=聚合根，`ErpFinPostingService` 降为 Processor，零 ORM 变更
  - **架构文档元规则**：“架构文档只写通用设计原则，不针对具体业务做技术设计” → AI 自行产出 Processor 扩展模式 ADR
- **一句话**：“用户不告诉答案，只告诉方向。”
- **视觉**：指南针图标 + 两条“方向 → AI 自定实现”箭头

### S12 【全场高潮】知识转移曲线（3 min）
- 时间线密度图（精美化 ASCII → 图形）：
  ```
  06-22  ███████████  A 类密集 + B + C 起步
  06-29        ████████████  C 类（grill 83 问，仅 4 题需纠正）
  07-01  ██   ██       最后两次 A 类 + 两次关键 B 类
  07-03  █                  最后一次 A 类
  ```

- **三阶段讲解**：

  2. **转折**（07-01）：两次 B 类确立工程范式边界。
  3. **后期**（07-04+）：用户退到“只提要求不给方法”。AI 的 19 个 skills + docs-for-ai 补充 + lessons 构成自运转基座。

### S13 grill-with-docs：83 题设计审查如何印证边界（3 min）
- 5 轮结构（漏斗，每轮 Q 数递减）：
  - Q1-Q40 设计正确性（状态机命名 / 会计期间 / 业财契约）
  - Q41-Q63 产品化可配置性（11 个“写死”→ 可配置，5 条设计原则）
  - Q64-Q76 集成/迁移/ORM 缺口（webhook / 数据迁移 / 种子数据）
  - Q77-Q83 横切缺口（金额精度 / 时区 / 审批委派 / 汇率来源）

- **印证边界的关键数字**：83 题中仅 **4 题**（Q71/Q77/Q78/Q79）需用户方向性纠正，其余 76 题 AI 自行审查发现并推荐 → 用户=逐轮确认者，非指挥者。

- **过渡到 Act 3**：“知识转移完成后，是什么让项目继续自己往前走？”

## Act 3：命题二 —— 让项目自转的引擎（5 页 / 9 min）

### S14 六种自驱动机制总览（1.5 min）
- 去掉外部输入后，项目靠六种内部机制自驱：

  3. **审计-整改驱动** —— 9 轮审计（Act 4 展开）

  5. **竞品对标驱动** —— 24 项目作完整性参照系
  6. **文档一致性驱动** —— ORM XML 唯一真相源
- **视觉**：六边形雷达图（每角一机制，轴=成熟度/频率）

### S15 机制 1 深展：计划-审计闭环（2.5 min）
- **标准工作流**：
  ```
  需求/想法 → 计划草案 → 独立草案审计 → 修订 → active → 执行 → 独立结束审计 → completed
  ```
- **强制双审计的意义**：
  - 草案审计拦方向性错误（07-10 拦下 4 个 P0：码值冲突 / BUDGET 污染 / 架构前提错误 / 维度歧义）
  - 结束审计验交付完整性 + 自省
- **独立子代理冷重放**：全新会话，不继承主代理上下文，规避确认偏差
- **187 计划**：每份带状态 / 范围 / 退出标准 / 验证清单
- **视觉**：流程图（审计步骤高亮红色 + 盾牌图标）+ 187 数字放大

### S16 【核心架构图】内部信息循环（2 min）

## Act 4：质量文化 —— 对抗性审计的演进与诚信（6 页 / 12 min）

### S19 【核心洞察图】审计体系的 8 阶段演进（2.5 min）
- 纵向演进（阶梯上升，关键转折点 07-05-1400 打大红星）：

| 阶段 | 时间 | 方法 | 演进点 |
|------|------|------|--------|
| 1 | 07-02 | 单代理 + rg 扫描 + 抽样精读 | 首次合规审计，引入复审修正机制 |
| 2 | 07-04 | 单代理 + 6 维度逐项证据 | 结束审计范式定型 |
| 3 | 07-05-1300 | 3 路并行子代理 + 主代理交叉 | **首次多代理并行** |
| 4 | 07-05-1400 | 4 路对抗性子代理 + 主代理复核 | **方法论飞跃：从“方法存在”到“语义保真”** |
| 5 | 07-05-1500 | 工具化合规检查器 | **人工经验固化为工具**（15 条 R 规则） |
| 6 | 07-06 | 单代理 + UC 断言对照 | **粒度切换：代码合规 → 业务实现率** |
| 7 | 07-07 | 4 路并行 + 自省 | 综合审计 + 审计质量回顾 |
| 8 | 07-12 | 3 路并行 + 主代理实时复核 | 竞争兑现核实 + 纠正 2 处子代理误判 |

- **5 个演进特征**：单代理→多路对抗 / “方法存在”→“语义保真” / 人工→工具化 / 代码合规→业务实现率 / 子代理“缺失”断言须主代理复核
- **视觉**：阶梯图，每阶=一阶段，07-05-1400 大红星

### S20 转折故事：07-05 的“方法论飞跃”（2 min）
- **背景**：一天内 3 轮审计（1300 三维 → 1400 对抗性 → 1500 工具化）
- **1400 的飞跃时刻**：4 路对抗性子代理不只看“方法是否存在”，而查“语义是否保真” → 发现 C-2 信用额度多币种语义偏差，评分仅 **4/10**（多币种绕过信用控制）；暴露“扩展域是审计盲区”（60 处时间获取蔓延 / DAG 违规 / 表名错误）
- **1500 的工具化反应**：当天固化为 `nop-compliance-checker.sh`（15 条 R 规则），人工模式 → 可重复脚本
- **教训**：文档对比类审计必须与源码交叉验证；评分偏差事件必须回溯归因
- **视觉**：故事板——4 子代理结果卡片 + C-2 评分对比 + 工具截图

### S21 竞争兑现核实：8 个“超越点”的裁决（2 min）
- **触发**（07-12，用户要求）：“承诺的功能/优势是否已完整兑现？” → 3 路并行子代理 + 主代理实时复核
- **裁决**：
  - 🟢 **完整兑现（4）**：多套账 SchemaPropagator / 业财一体三件套（过账·红字·反写）/ 7 种成本方法 / AR-AP open-item
  - 🟡 **有缺口（2）**：模型驱动 Delta 文档数据错误 / 制造委外空壳（15 行 CRUD）
  - 🔴 **夸大（2）**：多公司仅结构地基（无数据隔离/权限范围/公司间交易）/ 域独立部署言过其实（service 层跨域强耦合）
- **主代理纠正 2 处子代理误判**：“零 Delta 使用”错（实测 338 个 view.xml 用 `x:extends`）/“finance↔inventory 循环依赖”错（reactor 无环）

### S22 文档诚信事件：一个深度案例（1.5 min）

- **视觉**：三组 before/after + 底部闭环（审计→勘误→整改→闭合）

### S23 5 项风险 · 诚实声明（2 min）
- 每条“风险 → 项目回应”，左侧红/绿标签（未解决/已响应）：
  1. **文档诚信**（最突出）：计数漂移 / “多公司”夸大 → 审计识别 + 勘误 + 整改计划
  2. **多组织行为层缺失**：仅 orgId 列 + 字典 + 树，无隔离/权限/公司间交易 → 诚实降级为“结构地基”，排入后续
  3. **制造委外曾空壳**（15 行 CRUD）→ 07-13 落地完整委外引擎（生命周期 + MRP + GL 过账）
  4. **审计方法论改进空间**：扩展域曾是盲区；子代理断言需复核 → 持续演进（单代理→多路→语义保真→工具化）
  5. **logs 弱于“为什么”**：强于“做了什么”，弱于决策依据 → 建议增“重大审计发现登记”约定
- **语气**：不是“我们有很多问题”，而是“我们知道并在管理它们”
- **视觉**：5 条各两行 + 红绿标签

### S24 defect-aware：健康度总结（1.5 min）
- 核心健康指标不是“零缺陷”，而是：
  1. 缺陷发现后几乎全在 1-3 天内闭合
  2. 审计会自我纠偏（纠正子代理误判 + 回顾评分偏差）
  3. 同时报告亮点与问题，避免单一偏差
  4. 文档诚信问题被独立识别并要求勘误，而非掩盖
  5. 平台 bug 被主动发现并反哺 nop-entropy
- **底部大字**：**Not zero-defect — defect-aware.**
- **过渡到 Act 5**：“最后一个反哺——项目和平台之间发生了什么？”

## 收尾（3 页 / 6 min）

### S28 三个 Takeaway（2 min）
- **Takeaway 1 · 人机协作有边界**
  - 用户 = 平台权威（A 类）/ 方向指南（B 类）/ 非业务设计者（C 类）
  - 早期密集介入 → 知识转移完成 → 后期放手
  - 这不是“AI 自动写代码”，而是“AI 辅助 + 人类定向 + 系统化质量门控”
- **Takeaway 2 · AGE 工作流有效**
  - 八阶段递进，进入条件 + 退出门控明确
  - 强制双审计：草案拦方向错误 + 结束验交付完整
  - 内部信息循环 + 自我纠偏：评分偏差回顾、子代理误判纠正、方法论持续演进
- **Takeaway 3 · 模型驱动的产品化路径可行**
  - ORM XML 唯一真相源 → codegen → 154 模块全链构建
  - Delta 定制就绪度 98%（Processor protected 全面开放）
  - 平台共生：发现 7 个平台问题 → 反哺 nop-entropy
- **视觉**：三张 Takeaway 卡片，顶部回扣 S02 命题句

### S29 给不同角色的一句建议（2 min）
- **技术团队负责人**：投资质量门控（审计/测试/工具）比投资“让 AI 写更多代码”回报更高；知识转移前期投入，后期指数回报。
- **Nop 平台用户 / 潜在用户**：模型驱动能构建产品级应用（非玩具），Delta 定制是差异化竞争力（~98% 纯 Delta）；但 docs-for-ai 有缺口，建议源码级验证。
- **AI 辅助开发研究者**：三层注入模型（A/B/C）可作组织级 AI 采用的分析框架；审计 8 阶段演进 + 内部信息循环是可复用工程模式。
- **视觉**：三行，每行 角色图标 + 一句话（比 v1 每角色三条更精炼）
