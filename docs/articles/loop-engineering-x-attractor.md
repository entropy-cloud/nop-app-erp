# Loop Engineering in Practice：Plan Loop 与 Mission Driver 的设计分析

> nop-app-erp 在 22 天内从空骨架完成 154 模块、352 实体、约 2890 测试的全栈开发，零测试失败。本文专注分析驱动这一过程的两层嵌套控制 Loop——Plan Loop（单次变更的关闭契约）和 Mission Driver Loop（自动路线图编排）——的工程设计、决策逻辑与运行数据。全文不展开 Attractor 理论（该内容已由 AGE 模板下的独立文章覆盖），仅将吸引子作为控制 Loop 的方向前提简要引用。

## 1. 问题界定：什么是 Loop Engineering

当前业界对 Loop Engineering 的讨论正在升温。Andrew Ng 提出了三层 Loop 模型——Agentic Coding 分钟级、Developer Feedback 小时级、External Feedback 天/周级。Boris Cherny 用一句话概括自己的工作方向："我不再写 prompt 了，我改成设计写 prompt 的系统——我在写 Loop。"

这些讨论触及了一个核心方向：AI 辅助开发的终局不是写出更好的 prompt，而是构造出能在无人持续指挥下自运转的**闭环控制系统**。但"构造一个 Loop"究竟意味着什么？它需要解决哪些工程问题？两个相邻的 Loop 如何嵌套而不互相污染？loop 的边界在哪里——它能做什么，不能做什么？

本文以 nop-app-erp 为实证案例，依次回答三个问题：一次变更如何可靠关闭（Plan Loop）、多轮变更如何自动编排（Mission Driver Loop）、这两层控制如何嵌套而不耦合。两层 Loop 共同构成了 AGE 工作流的 Harness 层（控制层）。

## 2. Plan Loop：从 In 到 On 的关键——Plan 作为 AI 自主执行的基本单位

### 2.1 Vibe Coding 的循环困境

在引入 Plan 之前，需要先看清楚它要解决的问题。传统 AI 辅助开发的默认模式——通常被称为 Vibe Coding——是一个无限循环：用户说，AI 写，用户改，AI 再写。这个循环没有退出条件，因为它有四个结构性缺失。

**无基线。** AI 从对话记忆推断代码状态，不知道仓库里实际有什么。**无范围。** 边界模糊，经常做多或做少。**无退出标准。** "差不多了"就算完成——不可验证、不可审计、不可复现。**无独立验证。** AI 自己判断自己是否做对了——同一个上下文同时生成代码和验证代码的证据。如果生成时的理解存在偏差，所有"验证证据"会在同一个方向上系统性偏差。它们不会自相矛盾，但整体可能全错。

这四个缺失的共同指向是：Vibe Coding 把每一次变更当作**对话**——对话有来有回，但没有关闭条件。解决思路是把"对话"变成"契约"。

### 2.2 Plan 是什么：关闭契约，不是任务清单

Plan Loop 的核心是 Plan。Plan 不是任务清单——**它是 AI 自主执行的基本单位，是一份关闭契约。**

它包含三个核心要素和一个辅助项：

**当前基线（Current Baseline）**——从仓库读，不靠记忆。这是 Plan 的起点锚点。基线让审查者可以独立判断 Plan 承诺的变更量是否与仓库现状匹配——不需要知道执行者之前讨论过什么。

**目标集（Goals + Non-Goals）**——明确做什么，更重要的，不做什么。这是 Plan 的范围约束。不做什么防止执行中 Scope Creep，也防止审查者用 Plan 未承诺的标准来评估交付。

**退出标准（Exit Criteria + Closure Gates）**——可观察的完成条件。Exit Criteria 落在每 Phase 级别的可验证检查项中，Closure Gates 是 Plan 末尾的关门检查清单。每一项后面都有一个小方框等待被打勾。不是"功能做完"这种无法判定的描述。

加上一个辅助项：**审查记录（Draft Review Record）**——独立审查迭代的过程记录，包括发现的缺陷、修正轮次、最终裁决。它使 Plan 的演化过程可追溯。

### 2.3 生命周期：生成与验收分离

Plan 不是写了就执行的清单。它必须走完一个受控的生命周期。PPT 中的原话是：**"起草 → 独立草案审查（必须用全新会话，fresh session）→ active（可执行）→ 执行阶段（含验证）→ 独立结束审计（也是全新会话）→ completed。"**

两个审查节点——草案审查和结束审计——是 Plan Loop 的工程核心。它们共享一条关键规则：**生成与验收分离。** 审计者完全不看之前的聊天记录，从零开始审仓库。它唯一的证据来源是仓库中的文件和 Plan 本身。

这条规则是整个 Plan Loop 的前提性机制。它回答的不是"如何让审查更严格"——这是护栏的思路。它回答的是：在 AI 协作中，当生成者（执行者）和验收者（审计者）可以是同一个 AI 模型时，如何强制恢复它们之间的认知独立性。让审计者不进入执行者的思维框架、不被执行者的偏差捕获——**这是从 Human In The Loop（人在环中）升级到 Human On The Loop（人在环上）的前提。**

### 2.4 Plan 1100-1：四个在编码前拦截的 P0 缺陷

Plan 1100-1（销售定价引擎）批次是 Plan Loop 价值的浓缩案例。该批次包含四个 Plan，独立草案审查在编码开始前拦截了四个 P0 级别的缺陷。

1100-1 自身被审查发现缺少客户维度 Decision——定价引擎需要考虑客户分组，但 Plan 的设计没有覆盖这个维度，被打回 `needs_revision`。修正后二次审查通过。1100-3 被发现 businessType 码值冲突——Plan 为定价引擎新定义的业务类型码值与已有模块的码值范围重叠，如果进入执行阶段会导致运行时码值混乱。1100-4 的问题最严重——预算逻辑（BUDGET）被错误地设计为会影响实际财务数据（GlBalance），这意味着一个预算试算模块会污染真实的总账余额表。

四个缺陷的共同特征是：每个 Plan 单独看都自洽——格式正确、逻辑完整、类型一致、测试设计合理。四个加起来，方向在系统性漂移。Plan Loop 的审查抓住了它们，但抓住它们的前提是审查者知道"什么是对的"——这就是下一节要讨论的局限。

如果在 Vibe Coding 模式下，这些缺陷大概率要到实施甚至测试阶段才暴露。那时代码已写好了，测试已和其他逻辑耦合，修复不仅需要改实现，还需要重构已经依赖错误假设的周边代码和测试。Plan Loop 的审查成本约占总时间的 20%，但避免了运行期 5-10 倍的修复成本——一个 10 倍的工程杠杆。

### 2.5 Plan Loop 的局限：不解决调度

Plan Loop 让单次变更有了可靠的关闭契约。但它有一个原生限制：它不解决"编排"问题。

Plan 定义了"一次变更如何可靠完成"，但没有定义"下一个该做什么"。在 22 天、187 份 Plan [^1] 的尺度下，如果每一步都需要一个人来决定"下一步干什么"，Plan Loop 只是在更细粒度上复制了 Vibe Coding 的问题——每次循环的质量更高了，循环仍然是人驱动的。人的注意力成为了系统的吞吐上限。

更深一层看，Plan Loop 从 Vibe Coding 改进的地方是**执行质量**——把"一次变更"的完成标准从无形的主观判断变成了可审计的契约。但它没有触及的是**编排效率**——187 次变更的排序、状态追踪、推进节奏仍然需要人来管理。这就是为什么需要下一层 Loop。

## 3. Mission Driver Loop：把编排从人的注意力变成自动循环

### 3.1 编排问题的分解

一个编排系统需要同时解决四个子问题，而它们的工程特性截然不同：

**排序（Ordering）**：187 次变更的执行顺序。这不是一个可以静态预计算的调度表——每次变更完成后可能揭示新的依赖关系，使后面的顺序需要调整。排序需要的是一个可动态查询的路线图结构，而非一次性确定的甘特图。

**状态（State）**：每张 Plan 当前处于什么阶段——draft、active、执行中、completed？状态需要持久化在 Plan 文件自身中，使任何读取仓库的人（或下一轮循环）都能独立判断。

**推进（Advancement）**：谁来选择下一个工作项？谁来起草新的 Plan 并标记为 active？这个动作需要能自动执行，但执行结果的接受又需要独立审查来把关——推进和审计不能是同一个主体。

**审计（Audit）**：空闲时谁来触发深度审查？谁来扫描已完成的工作中隐藏的偏差？审计需要在系统稳定时（全部 active Plan 完成）自动启动，不能在系统处于半完成状态时介入（会被中间状态的噪音干扰）。

在纯 Plan Loop 模式下，这四个问题都由人来回答。Mission Driver Loop 的设计目标是把它们从人脑转移到机器——但不是把四个问题打包成一个黑盒，而是为每个子问题设计独立的处理步骤，步骤之间不传参数、只共享文件。

### 3.2 五步闭环的工程语义

Mission Driver 的核心是一个五步闭环引擎。以下描述的不是"流程"，而是每步解决的具体问题和它在闭环中的位置。

**第一步 CHECK——建立整个循环的基线前提。** 运行 `mvn clean install -DskipTests` 确认全量构建零红。这不是形式化的启动仪式。如果构建本身就有问题，后续所有 Plan 的执行和审查都会在错误基线上叠加。CHECK 的工程后果是：如果失败，循环中止——因为继续执行没有意义。这是 Mission Driver 的 **fail-stop 机制**：在发现系统性前提不可满足时立刻停止，不进入降级路径。

**第二步 REVIEW_PLANS——拾取所有待执行的 Plan 并完成审查。** 扫描仓库中所有 Status=draft 的 Plan 文件，对每个 Plan 启动独立子代理审查（新会话、冷重放），通过的标记为 active。这一步的语义是"拾取"——它不区分 Plan 的来源（上一轮循环自动起草的，还是人手工创建的），只关心状态。审查失败（`needs_revision`）的 Plan 留在 draft 状态，等起草者（人或下一轮 DRAFT_PLANS）修正后重新进入审查。

REVIEW_PLANS 放在 EXEC_PLANS 之前而不是之后，解决的是**避免重复起草**的问题。如果先执行再审查，系统中会同时存在多张针对同一目标的不同 Plan——因为起草者不知道另一张 Plan 已经在执行同一件事。

**第三步 EXEC_PLANS——对每个 active Plan 执行完整 Plan Loop。** 使用 forEach 逐个执行，顺序是 Phase N → BUILD_VERIFY（局部构建验证）→ CLOSURE_AUDIT（结束审计）。每个 Plan 执行完成后重新扫描 Plan 状态——不停留在循环开始时的状态快照上，即时发现执行过程中产生的新信息。BUILD_VERIFY 放在 EXEC_PLANS 内而非独立步骤，是因为构建验证需要 Plan 的上下文——执行了哪些文件、预期哪些模块受影响——这些信息在 EXEC 步骤内，如果作为独立步骤则失去上下文。

**第四步 DRAFT_PLANS——读取路线图起草新 Plan。** 从 `missions/erp.json` 读取路线图（包含三个子路线图的全部工作项和依赖关系），选择下一个可执行的工作项，起草新 Plan 并进行自审计，标记为 active。这一步的核心是 Plan 的**自动生成**——它把路线图中声明式的"下一步做什么"翻译为结构化的 5 要素 Plan。自审计（self-audit）确保起草的 Plan 在格式和范围上满足 Plan Loop 的门控要求，但在语义正确性上不做深度校核——那是 REVIEW_PLANS 的职责。

**第五步 DEEP_AUDIT——在系统最稳定时启动最深度的审查。** 触发条件不是"发现错误"，而是路线图上的所有工作项状态均为 done。这是一个经过设计的语义反转：**全部完成不是停止信号，是升级审计的信号。** 当系统还在高速推进时，深度审计会被中间的半成品状态干扰——它需要系统处于稳定态才能判断整体是否有结构漂移。全部 done 意味着所有已知工作项已经完成、构建全绿、测试全绿——系统处于最稳定的状态，此时审计的结论最可靠。

DEEP_AUDIT 发现的问题会自动反馈到下一轮循环：问题被记录为新 Plan（或追加到现有 Plan 的 Deferred But Adjudicated 列表中），进入 REVIEW_PLANS → EXEC_PLANS 的标准流程。

### 3.3 三个设计决策：复用性、可靠性、可观测性

Mission Driver 的工程实现有三个基础性设计决策，分别解决不同层面的问题。

**注入机制——解决跨项目复用。** Prompt 模板包含 `{{planGuide}}`、`{{missionConfig}}`、`{{roadmapIndex}}` 等占位符，引擎在启动每个步骤时从 `mission.json` 配置文件中读取实际值注入。这类似于考试：prompt 是试卷模板，配置文件是密封线里的个人信息。AI 只负责答卷，不管这个卷子是给哪个项目的。换项目时只需修改 JSON 配置文件，不需要修改 prompt 中的逻辑。

注入机制的反面是硬编码——把项目特定的路径、命令、约束写死在 prompt 里。硬编码的问题是：每换一个项目就要重写 prompt。而重写 prompt 需要理解 prompt 的内部逻辑，这本身就构成了一种隐式的耦合——prompt 作者必须同时理解项目结构和 prompt 结构。注入机制把这两个关注点分离了。

**隔离的顶级步骤——解决断点恢复和部分重试。** 五步之间不传递参数。CHECK 不知道 REVIEW_PLANS 发现了几张 Plan，REVIEW_PLANS 不知道 EXEC_PLANS 会执行哪些模块。所有状态——Plan 的 status 字段、审计的 PASS/FAIL 结论、构建的 stdout 输出——全部通过文件持久化到磁盘。每个步骤从磁盘读取自己的输入，执行后把输出写回磁盘。

这种设计的工程效果是：任意步骤可以单独执行（比如只跑 DEEP_AUDIT 而不走前面的完整链——这在开发阶段很常用）；断点可以从磁盘状态自动恢复（扫盘就知道"当前在哪"——哪些 Plan 还是 draft、哪些 active 但本轮尚未执行）；不需要内存中的状态管理——没有进程需要保持活跃，没有幂等性问题来自内存中残留的状态。隔离的代价是步骤间不能传递"为什么某个 Plan 审查未通过"这种上下文信息——这迫使每个步骤从文件证据中自行推导。

**零 IPC，全文件——解决可观测性。** 没有进程间通信。没有 web server。没有消息队列。没有数据库（除项目本身的 H2 测试库）。Mission Driver 运行时不需要任何持久运行的进程——它只是按流程顺序启动子代理会话，每个会话读写文件后结束。

这不是架构洁癖。它解决的核心问题是**可观测性**。在一个基于 IPC 的编排系统中，要判断"系统当前处于什么状态"通常需要查询某个运行中进程的内部状态——这个状态可能因为进程崩溃而丢失，可能因为并发更新而不一致，可能因为会话过期而不可读。全文件模式的可观测性是最简单的：你只需要读文件。任何一个步骤的输出都可以被独立审计——不是通过"检查日志"来实现（日志也是一种文件），而是所有状态天然在文件中，执行和审计共享同一个真相层。

### 3.4 对人的介入：异步协作模式

Mission Driver 的目标不是"替代人"，而是"让人可以在任意时间点介入，但不需要在每一步都介入"。

DRAFT_PLANS 持续起草新 Plan，将状态设为 draft。REVIEW_PLANS 自动拾取任何 Status=draft 的 Plan——不区分来源（机器起草的还是人手工创建的）。这意味着人可以在任何时候自己起草一张 Plan，扔进仓库，等下一轮循环时 REVIEW_PLANS 会自动审查并标记为 active。同样，人可以在任何时候修改一张 Plan 的状态（比如在 active 被错误标记时改回 draft），或者调整路线图的优先级（修改 `missions/erp.json` 中的工作项顺序）——下一轮 DRAFT_PLANS 会读取最新路线图。

这本质上是一种**异步协作模式**：人和 Mission Driver 在不同的时间尺度上工作（人可能每天集中处理一次，Mission Driver 每 4-12 小时自动完成一个完整循环），两者通过文件系统而不是即时通信来交换状态。这个模式的激进处不在于自动化程度——它在于**把人的介入从"必须"变成了"可选"**。在 Vibe Coding 中，环中的每一步都依赖人的触发。在 Plan Loop 中，每个 Plan 的触发仍然依赖人。在 Mission Driver 中，人可以选择参与——但系统在没有人的干预下，也能从路线图的第一项走到最后一项。

AGE 框架用一个术语来命名这个转变：**从 Human In The Loop 到 Human On The Loop**。In The Loop 意味着系统在等待人的输入才能推进到下一步。On The Loop 意味着系统可以自驱动，人在外围定义方向、调整优先级、在关键节点审查——而不是在每个步骤中都必须出现。

### 3.5 运行证据：07-10 的一天

07-10 是核心路线图收尾日。当天 Mission Driver 在一个 `draft-from-roadmap` 会话中完成了对 4 个 successor Plan 的并行独立审查（4 个独立子代理新会话同时审核 1100-1 ~ 1100-4）。审查发现其中 3 个 Plan 存在需要修正的问题：1100-4 发现了 6 个 blocker（包括 BUDGET 凭证污染实际财务），1100-3 存在 businessType 码值冲突，1100-1 缺少客户维度 Decision——全部在编码前拦截。

但这只是当天工作的一部分。同一天还有至少 10 个其他 Plan 在并行执行：资产折旧、制造完工入库与领料过账、维修备件过账、HR 休假/考勤/招聘/合同、凭证行精确数值断言、CRP 负荷引擎、SPC 统计过程控制、CRM 销售序列和配额、客服工单 SLA/CSAT、合同版本/返利。全天 14+ 个 Plan 执行完成。

Mission Driver 的循环不是"4 个 Plan → 一条直线"，而是多线并行下的状态管理：REVIEW_PLANS 拾取新的 draft Plan，EXEC_PLANS 在处理 active Plan 的同时，DRAFT_PLANS 已经在起草下一批 successor Plan。当所有核心路线图项 done 后，DEEP_AUDIT 自动触发——不是庆祝模块完成，而是趁系统稳定开始审查整体是否有结构漂移。

从更大的时间尺度看，整个路线图从 06-22 到 07-13 横跨 22 天。Mission Driver 在这期间完成了数十次完整循环，遍历了三个子路线图——`crud-roadmap.md`（CRUD 全覆盖）、`core-business-roadmap.md`（核心业务逻辑 M1/M4/M5）、`extended-roadmap.md`（扩展域 M2/M3）——的全部工作项。

### 3.6 Mission Driver Loop 的失败模式

任何工程系统都需要诚实地对待它的失败模式。Mission Driver 不是没有失败——它的设计决定了失败的**表现形式**和**可恢复性**。

**Review 误判。** REVIEW_PLANS 的独立审查可能因为子代理对平台机制的不熟悉而误判一个正确的 Plan 为 `needs_revision`。这在 07-12 发生过一次——竞争杠杆审计中一个子代理误判了 Delta 文档数据错误，主代理用构建事实复核后纠正。误判的恢复路径是：起草者（人或 DRAFT_PLANS）重新解读审查结论，修正 Plan，重新进入 REVIEW。这是一个"噪声"而非"失败"——系统在噪声中正常运转，但每次误判增加了循环时间。

**Step 孤岛化。** 隔离的顶级步骤不传参数，这意味着如果 REVIEW_PLANS 的审查结论中包含对 EXEC_PLANS 执行策略有价值的信息（比如"这个 Plan 的 Phase 3 需要特别注意——它的外部 API 依赖可能不兼容"），这个信息无法传递到 EXEC_PLANS。EXEC_PLANS 只能从 Plan 文件和仓库中重新推导。这不是 bug——它是"隔离换取可靠性"的必然代价。补救机制是：审查者把这种信息写入 Plan 的 Draft Review Record 中，EXEC_PLANS 从文件中读取。

**路线图漂移。** 路线图本身是一个静态文件——它定义了"做什么"的顺序。但如果执行过程中发现了路线图未预料到的工作（比如综合审计发现了一个新的跨域红冲缺口），这个新工作项的优先级需要人工评估。DRAFT_PLANS 的职责是按路线图顺序起草新 Plan——但路线图中不存在的项，它无法自动发现。这意味着 Mission Driver 在"已知范围内"是自动的，但"发现未知范围"仍然需要外部输入（DEEP_AUDIT 是发现未知的主要机制）。

## 4. 嵌套是 In → On 的结构性前提

两层 Loop 不是"两个独立系统同时工作"——是嵌套。Mission Driver 的 EXEC_PLANS 内部触发完整的 Plan Loop 实例。但这种嵌套关系的工程意义不是"一个调用另一个"，而在于它回答了一个更根本的问题：**为什么单层 Loop 不足以实现完全的 Human On The Loop？**

这个问题的难点在于一个表面上的悖论。Plan Loop 已经让 Human 在它的定义域内实现了 On——每个变更自主关闭，不需要人判断"是否完成"。从 Plan Loop 自身的角度来看，它已经完成了它该做的事：它的设计目标是"一次变更的可靠关闭"，这个目标已经达到了。那为什么还需要第二层 Loop？"调度"不是 Plan Loop 定义域之外的东西吗？把调度自动化难道不是"加一个新功能"而不是"补全一个不完整的 Loop"？

要回答这个质疑，需要区分两个不同的问题。一个是"Plan Loop 作为机制是否完成了它的设计目标"——答案无疑是 yes。另一个是"Human 是否从整个系统中退出了"——答案不是 yes，因为 Human 仍然在决定"下一个做什么"。**In → On 的判定标准不是单个 Loop 是否完成了自己的定义域，而是 Human 在整个项目推进中是否还需要作为环内驱动者存在。**

在此判定标准下，Plan Loop 实现的是**局部的 On**——在单次变更的尺度上 On。Mission Driver Loop 实现的是**更高一层逻辑类型的 On**——在变更间关系的尺度上 On。两层之后，Human 不再在任何操作层环内，而只在方向层环上。

### 4.1 为什么不能合并：逻辑类型而非尺度的差别

一个自然的追问是：不能把这两层合并成一个更大的单层 Loop 吗？

在工程上可以——Mission Driver 的五步加上 Plan Loop 的审查-审计链，全写进一个循环体。功能性上等价。但合并会失去一个关键性质：**执行层的审查者和调度层的决策者不能是同一个认知框架。**

执行层审查者判断的是："这段代码变更是否正确？"它的评估框架由 Plan 的范围和退出标准定义——它在一个局部坐标内工作。调度层决策者判断的是："在所有已完成和待完成的 Plan 中，下一步应该做什么？"它的评估框架由路线图的依赖关系和全局收敛状态定义——它在一个全局坐标内工作。

两种判断需要的**证据类型不同**。执行层证据 = 代码 diff、局部测试、Build 输出——改变更的内部产物。调度层证据 = Plan 状态、路线图完成率、审计发现的跨模块偏差——是 Plan **之间的**关系，不是任何单个 Plan 内的产物。

这两类证据**不可复合**。如果你让同一个 agent 在同一个循环中既判断 Plan A 的局部正确性、又判断 Plan A/B/C 中该先执行谁，那么局部判断中建立的认知模式会渗透到全局判决中——因为它是同一个 agent，它不可能在评估完 Plan A 的代码语义后立刻"清零"自己，以全新的全局视角来评估 A/B/C 的优先级。局部评估的结论会成为全局评估的先验偏置。

反过来也一样。如果一个 agent 先做了全局调度决策（"根据路线图，下一个是 Plan B"），然后立刻进入 Plan B 的内部审查，它的审查会倾向于确认全局决策的正确性——"因为 B 应该被排在下一个，所以 B 的设计应该没问题"。

这是**框架污染**。不是疏忽，是结构性矛盾——局部和全局的评估不能在同一个认知序列中先后发生而不互相污染，因为后者必须把前者的输出作为客观事实来对待，而同一个认知主体做不到这种自我客体化。

所以结论不是"调度需要不同的技能"或"调度的频率不一样"——这是表面现象。深层原因是：**局部闭合和全局闭合需要的证据分属不同逻辑类型。任何单层 Loop 只能在一个逻辑类型内工作，因此只能在那个类型上实现 On。要实现跨逻辑类型的 On，结构上需要第二层 Loop——它不进入内层的评估框架，只读取内层的输出结果。**

### 4.2 置换为逻辑类型语言的完整推导

**前提 1**：Plan Loop 的审查者判断的是"这个变更是否正确"——它在一个局部坐标（Plan 定义的 scope 和 exit criteria）内工作。完成的标志是可观察的关闭条件（test、build、audit）。这是**局部闭合**.

**前提 2**：项目级推进需要判断"在所有 Plan 中下一步应该做什么"——这需要全局坐标（路线图依赖、整体收敛状态）。判断的依据是 Plan 之间的**关系**，不是 Plan 内部的内容。这是**全局推进**。

**前提 3**：局部闭合的证据（代码 diff、测试输出）和全局推进的证据（Plan 状态、路线图拓扑、跨模块审计发现）分属**不同逻辑类型**——前者是 Plan 内部产物，后者是 Plan 间关系。

**前提 4**：同一个评估主体在同一个循环周期内先后做出这两类判断时，第一类判断的框架会污染第二类——因为该主体的认知状态继承了对内层证据的参与经验，无法将内层输出当作客观黑箱来处理。这是**框架污染**。

**结论**：因此，要实现完全的 Human On The Loop——即 Human 退出所有操作层环——结构上需要两个 Loop，内层负责局部闭合，外层负责全局推进。外层**不参与**内层的证据评估，只读取内层的输出状态。两层之间的接口是 Plan 文件的状态字段——这是确保外层用"不同逻辑类型"处理内层结果的工程边界。

### 4.3 终止条件：为什么 Attractor 不是第三层 Loop

一个合理的追问是：如果每个逻辑类型都需要一层 Loop，为什么到两层就停了？调度层之上不需要新的 Loop 吗？

Attractor 不是一个 Loop。它不执行任何操作——它不循环，不拾取，不审查，不起草。它是一个结构不变量（ORM 唯一真相源、DAG+ R/S/P 跨域契约、三轴正交状态分离）。它回答的是"方向是什么"，不是"下一步如何推进"。

所以递归在两层停止，不是因为"两层够了"——而是因为再往上不再需要操作层。Attractor 是方向层的静态结构，不需要动态 Loop 来维持。它被人定义，被文件承载，被两层 Loop 作为方向前提引用，但它自身不是环。**Loop 解决的是"如何执行"——当所有需要执行的操作都有了对应的 Loop 之后，不需要更多 Loop。**

### 4.4 嵌套的工程实现

这个嵌套关系在 Mission Driver 源码中有直接的对应。外层流程（`flows/mission-driver.json`）将 EXEC_PLANS 声明为 `type: "subflow"`，指向内层流程 `plan-execution.json`。引擎在执行时，不是"在同一个循环里多跑几步"——而是调用 `_runChildSubflow`（engine.js:1055），创建一个**全新的 FlowEngine 实例**，带有独立的状态空间（`flowVars`、`visitCounts`、`retryCounts`）。

内层 plan-execution 有自己的状态机：EXECUTE → CLOSURE_SCRIPT_CHECK → CLOSURE_AUDIT → BUILD_VERIFY。每一步有独立的 marker 驱动转换，有自己的 `maxCycleVisits` 和 `pingPongWindow`——完全隔离于外层 mission-driver 的五步循环（CHECK → REVIEW_PLANS → EXEC_PLANS → DRAFT_PLANS → DEEP_AUDIT）。

两层之间的接口只有函数调用参数：外层的 `forEach: "activePlans()"` 扫出 Plan 文件列表，每个文件作为 `PLAN_FILE` 传递给子 FlowEngine；子返回 `{status, flowVars}`，外层的 `_executeSubflowStep` 聚合为 `all_complete` / `some_failed` 然后查自己的 transitions 表决定下一步。外层不进入内层的 EXECUTE → CLOSURE_AUDIT → BUILD_VERIFY 链。内层不感知外层是否有 DRAFT_PLANS 在等待。

## 5. 两层 Loop 不能做的事

控制 Loop 有它的边界。明确这个边界，与明确它做什么同等重要。

### 5.1 两层 Loop 不定义方向

Plan Loop 可以在编码前拦截 4 个 P0 缺陷——但它依赖于审查者知道"什么是对的"。Mission Driver 可以自动遍历路线图——但它依赖于路线图本身是正确的。

07-12 的竞争杠杆审计是一个清晰的例证：独立子代理核实项目承诺的 8 个竞争优势，发现 2 个夸大（文档说"多公司 ERP"实际只有地基、说"域独立部署"言过其实）和 2 个缺口（制造委外是空壳、Delta 文档数据错误）。从 Plan Loop 和 Mission Driver 的角度，8 条中每条对应的 Plan 都通过了审查和审计——控制层没有失败。失败的是方向定义层。

方向定义属于 Attractor（吸引子）的范畴，不在本文范围内（详见 AGE 模板下的独立文章）。但为了给两层 Loop 一个完整的方向上下文，nop-app-erp 中吸引子的三个具体维度有必要简述：ORM XML 唯一真相源（所有持久化代码从 XML 生成，手写绕过等于编译失败），DAG + R/S/P 跨域契约（模块依赖为无环有向图，跨域引用分只读桩/同步接口/弱关联字符串三类，共 369 条白名单），三轴正交状态分离（docStatus × approveStatus × posted 三维独立互不污染）。这三个结构不变量贯穿了 22 天的全部设计决策，为 Plan Loop 的审查者和 Mission Driver 的路线图提供了统一的判断标准。

### 5.2 两层 Loop 只构成一个更大系统中的控制层

Plan Loop 和 Mission Driver Loop 构成了 Harness 层中的两层——但 Harness 只是项目自驱动体系的一部分。基材中的分析识别了另外五种支撑机制：路线图驱动（三个子路线图定义实施顺序）、审计-整改驱动（每隔 2-3 天的深度审计发现偏差后触发整改）、测试回归防护（约 2890 单元测试 + 260+ E2E 的分层防护）、竞品对标（24 个开源项目的源码级调研作为功能完整性参照系）、文档一致性驱动（ORM XML 是唯一真相源，散文不重复字段定义）。

这些支撑机制不是"额外的好习惯"——它们和两层 Loop 之间形成了一个闭合的信息流：路线图驱动起草新 Plan → Plan Loop 的审计发现偏差 → 偏差通过 Mission Driver 触发整改 Plan → 整改后测试验证 → 测试发现的回归和竞品对标揭示的功能缺口驱动新一轮 DRAFT_PLANS → 文档一致性防止认知漂移。两层 Loop 是这个循环的核心引擎——但没有五种支撑机制的输入，引擎就没有"要做什么"的燃料。

## 6. 结论

从 nop-app-erp 的实证中，可以提炼出两层 Loop 的工程规律。

Plan Loop 解决的是**执行质量**——把一次变更从无形的主观判断变成可审计的关闭契约。它的核心机制不是"更多的检查"，而是生成与验收的认知分离——用 fresh session 的独立审计强制恢复 AI 协作中失去的认知独立性。代价是每个 Plan 约 20% 的时间消耗在审查上，回报是编码前拦截 P0 缺陷所节省的 5-10 倍修复成本。

Mission Driver Loop 解决的是**编排效率**——把 187 次变更的调度从人的注意力驱动变成路线图和文件状态的自动推导。它的三个基础设计决策——注入机制（跨项目复用）、隔离步骤（断点恢复）、零 IPC 全文件（可观测性）——共同定义了一种"文件系统即真相层"的编排架构。它的限制也同样明确：不能自动发现路线图未定义的新工作项，不能在不损失隔离性的前提下传递步骤间的上下文信息。

两层 Loop 之间的嵌套关系产生了一个比各自独立功能更重要的性质：它们可以在互相不感知对方内部结构的前提下协同工作。接口只有文件系统——Plan 的状态字段和路线图的 JSON 结构。这使两层可以独立演化、独立故障、独立恢复。

但 Loop Engineering 能解决的只是"如何执行得更好"和"如何编排得更高效"。在 22 天里，187 张 Plan 得以逐个完成、数十次循环得以自动推进——前提是方向层已经定义了"正确的系统长什么样"。Loop Engineering 让 AI 跑得更快更稳。但往哪里跑——那个问题的答案不在 Loop 之内。
