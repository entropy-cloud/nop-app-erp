## Current Baseline

实时仓库逐项核实（`grep`/`read`，非采信旧记忆）：

- **全域零 `<index>` 声明**：`grep -rl "<index" module-*/model/*.orm.xml` = 0 命中。18 域源模型中没有任何 `<indexes>` 段。仅存在 156 个 `<unique-key name=>`（154 个由 plan `2026-07-05-1000-1` 今日新增 + 2 个既有保留 `uk_config_org`/`UK_EDI_DOC_FORMAT_BILL`，与 1000-1 Closure 记录一致）。
- **ORM `<index>` 语法已确认**（nop-entropy `entity.xdef:212-226`）：`<indexes>` 段内 `<index name="!var-name" unique="boolean" indexType="string">` + 子元素 `<column name="!var-name" desc="boolean"/>`（**注意**：索引用子元素 `<column>`，与 `<unique-key>` 的 `columns="csv-list"` 属性写法不同）。
- **高频过滤维度已物理列化**（`domain-design-guidelines.md:495` Decision 明列 `orgId`/`posted`/`businessDate`/`currencyId` 为物理列；同范式下 `docStatus`/`approveStatus`/`warehouseId`/`partnerId`/`materialId` 等亦为各实体物理列，实测核实）：标准维度均为物理列（非 EAV），具备索引能力。
- **列表页过滤路径**：18 域 view.xml 列表页普遍按 `orgId`（多租户隔离，几乎每个查询都带）+ 逻辑删除过滤（物理列 `delVersion`=0，平台管道默认追加）+ 业务状态列（`docStatus`/`approveStatus`/`posted`/`status`）+ 外键导航（`orderId`/`materialId`/`partnerId`/`warehouseId`）筛选。当前无索引支撑，全表扫描。
- **bootstrap 阶段无生产数据**（`project-context.md`）：H2 开发/测试库 + codegen 生成 DDL。加索引无数据迁移、无回填脚本需求；DDL 由 `nop-cli codegen` 生成 `_create_*.sql` 时自动产生 `CREATE INDEX`。
- **命名规范前驱已建立**（plan 1000-1 Phase 1）：唯一键用 `UK_{TABLE}_{COLS}` 全大写。本计划索引沿用 `IDX_{TABLE}_{COLS}` 范式保持一致。

### 剩余差距

1. 无任何非唯一过滤索引——列表查询、外键导航、状态过滤全表扫描。
2. 无索引命名规范与「何时加索引」的决策规则（哪些列组合值得索引、多租户 orgId 是否前缀列、逻辑删除 delFlag 是否纳入索引）。
3. 无文档记录全域索引落地状态（`idempotency-pattern.md` 仅记唯一键）。

## Goals

- **18 域源模型补齐高频过滤 `<index>`**：为列表页高频过滤路径、外键导航列、状态筛选列添加非唯一索引，覆盖 `orgId`+`delFlag` 多租户过滤前缀、外键列（`*Id`）、状态列（`docStatus`/`approveStatus`/`posted`/`status`）、业务日期（`businessDate`）。
- **建立索引命名规范与决策规则**：`IDX_{TABLE_WITHOUT_ERP_PREFIX}_{COLS}` 命名；何时单列 vs 复合；多租户 `orgId` 作复合前缀的判定；逻辑删除 `delFlag` 纳入复合索引的判定。
- **幂等/异步入口组合索引**：对 1000-1 Deferred「异步入口的组合幂等键」（webhook eventId 等）在对应入口实体上补建（若存在）。
- **文档对齐**：`idempotency-pattern.md` 规则 2 落地状态补「过滤索引」段；owner doc 无需改（索引是性能优化，不改变业务语义）。
- **全程不手改生成物**；ORM 变更经 `mvn clean install` 增量重新生成 DDL。

## Non-Goals

- **不**新增业务功能——仅添加 DB 索引（性能优化），不改业务逻辑、不改 API 契约、不改视图。
- **不**修改任何 Java 代码（索引由 ORM 模型声明，codegen 生成 DDL，运行时查询规划器自动使用）。
- **不**做查询性能基准测试/benchmark——bootstrap 阶段无生产数据规模；索引设计基于查询模式静态分析（列表页过滤路径）。性能验证归「首次性能测试」后继（触发条件：生产数据规模 + 慢查询出现时）。
- **不**覆盖全部列——仅覆盖高频过滤/导航/外键列；低频列、大文本列（`description`/`remark`/`resolution`）不加索引。
- **不**改唯一键（1000-1 已全域落地；本计划只加非唯一 `<index>`，避免与 `<unique-key>` 冲突）。
- **不**做数据库特定的部分索引/函数索引（如 PostgreSQL 的条件索引）——保持跨数据库（H2/MySQL/PG）可移植，只用标准 `<index>` 语法。
