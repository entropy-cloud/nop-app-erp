## 按严重性排序的发现

### 🔴 发现 1（高）：验证环境不稳定——ErpAllWebPagesCollectTest 的 PAGE_ERROR_COUNT 在同日内 0 ↔ 213 跳变

**来源**: `docs/logs/2026/07-20.md:195` 记录当前报 213 errors（JDK 26 / antlr 版本不兼容）。但同文件较早条目（第 6、16、35、105、127、164、178 行）报告 `PAGE_ERROR_COUNT=0`。

**问题**: 同一天内关键指标从 0 跳变到 213，揭示了：
- 验证环境不稳定（不同运行间 JDK 版本不同）
- 关键安全网（页面模式验证）在开发环境中实际失效
- 项目已学会检查前后 delta 而非绝对值，掩盖回归

项目管理此风险恰当（"前后错误计数完全一致"），但此桥接削弱了验证管线且无明确修复计划。

**建议**: 将此修复列为阻塞项，或临时 `@Disabled` 并附带明确的重新启用标准。当前状态——已知失效测试继续报告 PASS 但实际计数 213——是危险的半健康状态。

---

### 🔴 发现 2（高）：代码生成产物编辑——跨计划、跨会话的重复失败模式

**来源**: `docs/logs/2026/07-20.md:118-128`（notify-inbox 三回合修复）、`docs/logs/2026/07-18.md:195-200`（business-type.dict.yaml `__XGEN_FORCE_OVERRIDE__` 三回合修复）

**问题**: 两个独立计划（相隔 2 天）均因编辑代码生成产物而在多轮独立审计失败后才被发现。两个案例共同模式：
- 实施者编辑了代码生成产物
- `mvn clean install` 覆盖了这些编辑
- 审计者检查生成文件（`git diff` 空）而非保留层
- 每案例需三轮审计才找到根因

这是**跨多计划跨多会话才可见的重复系统性失败模式**。AGENTS.md 已陈述"切勿手动编辑生成的文件"，但两案均违反。07-20 元审计已在平台合规技能加维度 13，但尚未提升为持久经验教训。

**建议**: 向 `docs/lessons/` 新增教训 06："代码生成产物编辑：`_` 前缀和 `__XGEN_FORCE_OVERRIDE__` 文件被 `mvn clean install` 覆盖——始终编辑保留层。"

### 🟡 发现 3（中）：XWF 审批浏览器层可行性裁决——下游影响未传达

**来源**: `docs/logs/2026/07-09.md:96-103`（2330-1 xwf 可行性探索）、`docs/backlog/README.md:41`

**问题**: 2330-1 对 4 实体 xwf 审批轴给出"NOT FEASIBLE"裁决。此裁决正确且诚实记录，但 `roles-and-permissions.md` 和受影响 owner docs 均未更新以注明浏览器层不支持的审批路径。下游用户/开发者面对受限审批路径的实体时，需跨多文档推理才可理解限制来源。

**建议**: 在 `roles-and-permissions.md` 和受影响域 owner docs 的状态机章节中注明 xwf 浏览器层受限。

### 🟡 发现 4（中）：Owner-doc 漂移——记录良好但无系统性扫描

**来源**: `docs/logs/2026/07-18.md:41`（expense-claim.md l.196 漂移）、`docs/logs/2026/07-18.md:168`（business-type.dict.yaml MAINTENANCE_ISSUE 声称已添加但从未落地）

**问题**: 项目高度依赖 owner docs 作为稳定吸引子。三次发现声称已落地但实际上并未落地的内容。Owner-doc 漂移在事后被捕获，但**无系统性扫描**来检测漂移——仅通过审计中的对抗性抽样发现。18+1 域、数十 owner docs、269 计划，可确信还有更深漂移未被发现。

**建议**: 建立周期性 owner-doc → 代码一致性扫描机制（脚本或 AI 审计），而非仅靠对抗性抽样。
